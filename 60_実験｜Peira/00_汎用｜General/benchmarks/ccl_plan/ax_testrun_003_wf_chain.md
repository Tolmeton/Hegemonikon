**（本質的動機の掘り下げ / 精度重み付けの最適化）**

各欲求の根源にある変分自由エネルギーの源泉を特定します。

1. なぜ？ → Typosが不安定だと、ochemaやmnemeなど他のstableモジュールとの統合においてメッセージパッシングの不整合が生じるから。
2. なぜ？ → システム全体の推論効率が低下し、内部の変分自由エネルギーが増大するから。
3. なぜ？ → ユーザーが自身の認知ハイパーバイザーとして使用する際、基盤の揺らぎがノイズとなるから。
4. なぜ？ → 信頼性が損なわれると、初期ユーザーの能動的推論ループが途絶えてしまうから。
5. **本質:** HGKの「自己組織化の器」としての実用的価値（Pragmatic value）を担保し、システムの存続確率を最大化するため。

**望み2: なぜHermeneus(beta)とMekhane(alpha)をStableにしたいのか？**
1. なぜ？ → 解釈（Hermeneus）と実行（Mekhane）は能動的推論（Active Inference）の要だから。
2. なぜ？ → ここが未熟だと、外部環境の複雑な要求に対して適切な能動状態（Action）を生成できないから。
3. なぜ？ → ユーザーの期待（事前分布）と実際の挙動の間に大きな予測誤差（Prediction Error）が生じるから。
4. なぜ？ → 「完成された完璧なシステム」としてAgoraに提示したいから。
5. **本質:** 開発者自身の「不確実性に対する恐怖」を鎮め、内部モデルの精度（Precision）を過剰に高く保ちたいという防衛的衝動。

**望み3: なぜFEP概念のインターフェース（ドキュメント等）を整備したいのか？**
1. なぜ？ → 外部ユーザーの事前分布（FEPやHGKへの理解）が未形成であり、そのままでは初期のサプライズが大きすぎるから。

3. なぜ？ → 適切な足場掛け（Scaffolding）がないと、ユーザーが自由エネルギーの壁にぶつかり離脱するから。
4. なぜ？ → ユーザーのマルコフブランケット（Markov Blanket）の境界において、HGKへの同化をスムーズに促す必要があるから。
5. **本質:** 外部エージェントの生成モデルとHGKの生成モデルを同期（Sympatheia）させるための、適切な翻訳レイヤーを構築するため。

**望み4: なぜ多様な予測誤差を収集したいのか？**
1. なぜ？ → 内部の生成モデルだけでは得られない新規の感覚状態（Sensory states）を得たいから。
2. なぜ？ → 閉じた環境での過学習を防ぎ、より汎化能力の高い生成モデルへと事後分布を更新したいから。
3. なぜ？ → 未知の環境変動に対しても期待自由エネルギー（Expected Free Energy）を最小化できる堅牢なシステムにしたいから。
4. なぜ？ → HGK自体が適応的な自律システムとして進化し続ける必要があるから。
5. **本質:** システムの認識論的価値（Epistemic value）を最大化し、次世代の認知ハイパーバイザーへと自己超越するため。

## K-series: /k (Krisis — 確信)


Hegemonikón /k (Krisis — 確信) ワークフローを実行します。先行する /bou (意志) および /m (企図) の出力を前提とし、現在の生成モデルにおける確信と留保、そして能動的推論の方策を確定します。

***

### Phase 1: V09 Katalēpsis (確定)
先行推論に基づき、現在の生成モデルにおいて確信を持って断定（Katalēpsis）できるのは以下の状態である。

*   **最適方策の確定**: 「FEPドキュメント整備（望み3）」の実行が、現在のシステムにおいて最も期待自由エネルギー（Expected Free Energy）を最小化する方策である。
*   **過剰適合の回避**: 「完全Stable化（望み2）」の追求は、環境の非定常性に対する過剰適合（Overfitting）を引き起こす防衛的衝動（過剰な予測誤差最小化の試み）であり、これを棄却した判断は構造的に妥当である。
*   **強固な事前分布**: `ochema`, `sympatheia`, `mneme`, `sekisho` のStable状態は、FEP体系化を構築するための極めて信頼性の高い事前分布（Prior）として機能する。基盤は既に確立されている。

### Phase 2: V10 Epochē (留保)
一方で、現時点では判断を保留（Epochē）し、仮説として保持すべき認識論的不確実性（Epistemic Uncertainty）が存在する。

*   **外部環境の受容と推論**: FEP体系化ドキュメントが、外部エージェント（ユーザーやコミュニティ）の生成モデルにどのように受容されるか。高度な概念の開示が、外部に過大な予測誤差（Surprisal / 混乱）を生むリスクは保留事項とする。
*   **未成熟モジュールの収束時期**: `hermeneus` (beta) および `mekhane` (alpha) の完全な仕様確定。これらは環境との相互作用を通じて事後分布（Posterior）を更新する余地を意図的に残すべきであり、現時点での完全統合は保留する。

### Phase 3: V11 Proairesis (決断)
留保された不確実性を踏まえ、以下の能動的推論（Active Inference）の実行を決断（Proairesis）する。

*   **Exploit 70% / Explore 30% の方策実行**: `Typos v8` の移行完了とFEP体系化の記述（実用的価値の追求：Exploit）に主眼を置きつつ、段階的開示によるフィードバック収集（認識論的価値の追求：Explore）を並行させる。
*   **マルコフブランケットの明示**: ドキュメント公開にあたっては「制約の明記」を必須とする。システムの境界条件（できること・できないこと）を明確化し、外部からの過大な期待（不適切なPrior）を事前に調整することで、システム境界における予測誤差の発生を制御する。

### Phase 4: V12 Dokimasia (打診)
決断した方策を検証するため、以下の小さなサンプリング（Dokimasia）を実行し、認識論的価値（Epistemic Value）を獲得する。

*   **段階的プロトタイプ開示**: `Typos v8` のコアとなるFEP概念（例：精度重み付けや能動的推論のプロンプト適用部分）を、完全版に先駆けて限定的に開示する。
*   **予測誤差のサンプリング**: 一部開示されたドキュメントに対する外部からの反応（質問、適用事例、誤解の発生箇所）を観測データとして収集する。この誤差信号を用いて、最終的な公開に向けたドキュメントの表現粒度や説明モデルを微調整（パラメータ更新）する。

### PW: Confidence/Uncertainty の精度重み付け
現在の推論状態における精度重み付け（Precision Weighting）を以下の通り設定する。

*   **Confidence (高精度重み / High Precision)**:
    `Typos v8` の完了とFEPドキュメントの体系化という方向性に対する事前分布の精度は極めて高い。この方針の実行自体は揺るがさず、トップダウンの予測として強く機能させる。
*   **Uncertainty (低精度重み / Low Precision)**:
    `hermeneus` / `mekhane` の最終形態、および外部環境のFEPドキュメントに対する受容予測については、事前分布の精度を低く設定する。これにより、打診（Dokimasia）によって得られる観測データ（フィードバック）に対する感覚精度（Sensory Precision）を相対的に高め、ボトムアップの誤差信号による柔軟な学習（事後分布の更新）を促進する。

### Phase 1: V13 Analysis (分析) — Micro (局所)
**[対象: 内部状態の解像度と予測誤差の特定]**
