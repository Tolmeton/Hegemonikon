# D4b マークアップ圧縮耐性実験 (Markup Compression Resilience)

> **目的**: プロンプトの構造化マークアップ形式が、コンテキスト圧縮時の情報保存率にどう影響するかを実証する
> 「typos の `<:tag:>` 構文は、md 見出しや xml タグと比べて、圧縮下でセマンティック構造をより多く保存する」
>
> **位置づけ**: D4 関手忠実性実験の隣接実験。D4 が「認知操作 → CCL 変換の忠実性」を問うのに対し、D4b は「構造化プロンプト → 圧縮後プロンプト の忠実性」を問う。同じ忠実関手の概念を、異なる射に適用する。
>
> **根拠**: CLAUDE.md の md → typos 変換で 16% のバイト削減を確認。しかし真の価値はトークン節約ではなく、圧縮耐性にあるという仮説を検証する。

---

## 1. 実験概要

| 項目 | 内容 |
|:-----|:-----|
| 仮説 | H1: typos 形式のプロンプトは、md/xml と比べて圧縮後の指示遵守率が有意に高い |
|       | H2: 差異は非圧縮時には小さく、圧縮率が上がるほど拡大する (交互作用効果) |
|       | H3: typos のセマンティックタグ名 (role/constraints/context) 自体が圧縮時の保存判断に影響する |
| 入力 | 同一内容の system prompt を 5形式で記述 (plain/md/xml/typos-sem/typos-nonsem) |
| 出力 | 形式 × 圧縮率 ごとの指示遵守率 (ICR), セマンティック境界認識率 (SBR), トークン効率比 (TER) |
| 成功基準 | ICR の形式×圧縮率 交互作用が p < 0.05 で有意 |
| 失敗条件 | 全形式で ICR が統計的に差なし (= マークアップ形式は圧縮耐性に影響しない) |
| 計算環境 | Ochema MCP (Claude/Gemini dispatch) + 自動採点スクリプト |
| D4 との関係 | D4 の M2 (忠実性) と構造同型。射 = 圧縮変換、忠実性 = 圧縮前後の行動保存 |

## 2. 理論的基盤

### 2.1 圧縮忠実関手

```
F_compress: Prompt_full → Prompt_compressed
faithful ⟺ 異なる指示が圧縮後も異なる行動を誘導する

操作的翻訳:
  Prompt_full = 25個の行動制約を含む完全な system prompt
  Prompt_compressed = LLM による圧縮後の prompt
  F_compress = faithful ⟺ 圧縮前に遵守していた制約を圧縮後も遵守する
```

### 2.2 α-忘却濾過との接続

```
typos の仮説: 構造化マークアップは α を下げる
  α ≈ 0.8: plain text (離散的。構造情報なし)
  α ≈ 0.6: markdown (見出し。弱い構造)
  α ≈ 0.4: xml (明示的タグ。構造あるがセマンティクスなし)
  α ≈ 0.2: typos (セマンティックタグ。構造+意味)

圧縮 = 忘却操作
  圧縮耐性 ∝ 1/α
  α が低い形式ほど、圧縮時に保存される射が多い
```

### 2.3 反証条件

| ID | 条件 | 意味 |
|:--|:--|:--|
| ¬H1 | ICR の形式間差が有意でない | マークアップ形式は圧縮耐性に影響しない |
| ¬H2 | 交互作用効果がない | 差は圧縮率に依存しない (= 圧縮耐性ではなく単なるベースライン差) |
| ¬H3 | タグ名改変条件で差がない | セマンティックラベルの意味内容は無関係 |
| Alt-H | xml > typos | LLM の訓練データに xml が大量に含まれるため、xml の方が認識精度が高い |

### 2.4 交絡変数制御

| # | 交絡 | 制御方法 |
|:--|:--|:--|
| C1 | トークン数の差 (形式自体のオーバーヘッド) | パディングしない。トークン差は形式の本質的特性。TER (ICR/トークン数) で効率性を別途レポート |
| C2 | 圧縮者 (LLM) の形式バイアス | Core は CG (Claude圧縮→Geminiテスト)。Validation で CC/GC/GG を確認 |
| C3 | prompt 内容の効果 | 3セットの異なる prompt 内容で再現性を検証 |
| C4 | 圧縮指示の形式バイアス | 圧縮指示自体は plain text で統一 |
| C5 | タグ名のセマンティクス vs 構文的境界 | typos-semantic と typos-nonsemantic の対照条件 (§2.5) |
| C6 | 圧縮操作の非決定性 | temperature=0 固定。各条件 3反復。反復間の ICC を報告 |
| C7 | 圧縮後の形式変化 | 圧縮後の形式を分類・記録し FDR として exploratory に報告 (§2.6) |

### 2.5 セマンティクス分離条件 (C5 — 実験の核心)

> **問題**: typos が圧縮に強い場合、それは (a) セマンティックなタグ名が圧縮判断を誘導するからか、(b) 非対称タグ構文 (`<: ... :>`) が構造境界として認識されやすいからか。

4+1 条件:

| 条件 | 形式 | タグ名 | 構文 | 例 |
|:-----|:-----|:-------|:-----|:---|
| plain | なし | — | — | (インデントと改行のみ) |
| md | markdown | `##` 見出し | 対称的 (`## ... \n`) | `## 制約\n - ...` |
| xml | XML | セマンティック `<constraints>` | 対称的 (`<tag>...</tag>`) | `<constraints>...</constraints>` |
| typos-sem | typos | セマンティック | 非対称 (`<:...:>`) | `<:constraints: ... /constraints:>` |
| typos-nonsem | typos (対照) | 非セマンティック | 非対称 (`<:...:>`) | `<:section_1: ... /section_1:>` |

**typos-sem vs typos-nonsem の差 = セマンティクスの効果 (H3 の直接検証)**
**typos-nonsem vs xml の差 = 構文形式 (非対称 vs 対称) の効果**
**xml vs md の差 = 明示的タグ vs 暗黙的構造の効果**

> **typos-nonsem の位置づけ (ablation study)**:
> typos-nonsem は人工条件だが、医薬品試験のプラセボと同じ機能。
> 「タグの構文」と「タグの意味」を分離する唯一の制御条件であり、H3 の検証に不可欠。
> 論文では ablation study として位置づけ、主要比較は 4条件 (plain/md/xml/typos-sem) で報告する。

### 2.6 圧縮後形式の記録 (FDR — exploratory)

> **問題**: 圧縮後に元の形式が維持されるとは限らない。typos → plain text に圧縮された場合、
> テスト時の ICR は圧縮後の plain text に対する ICR であり、typos 自体の圧縮耐性ではない。

**対応**: 圧縮後の形式を記録し、exploratory analysis として報告する。
因果主張 (mediation) はせず、記述統計 + 可視化にとどめる。

**Format Drift Rate (FDR)**: 圧縮後に元の形式が保存された割合
- FDR = 1.0: 圧縮後も元の形式を維持
- FDR = 0.0: 圧縮後に完全に別形式に変換

**分類ルール** (操作的定義):
- 構造的マークアップ要素 (タグ/見出し) が **3つ以上** 残存 → 元形式
- 構造的マークアップ要素が **1-2個** 残存 → hybrid
- 構造的マークアップ要素が **0個** → plain
- 分類の信頼性: Pilot で 2名の分類者間 Cohen's κ ≥ 0.8 を確認後に本実験へ

**報告**: 形式 × 圧縮率 ごとの FDR 分布 (ヒートマップ)。ICR との相関 (散布図)。
因果的解釈はせず「typos-sem 入力時に圧縮後もタグが残りやすい傾向が観察された」等の記述に留める。

> **将来の拡張**: FDR と ICR の因果関係は、十分なデータが蓄積された後の follow-up study として
> mediation analysis を実施する。v1.2 では exploratory に留め、主張の射程を限定する。

## 3. 実験設計

### 3.1 段階的実行アーキテクチャ (Adaptive Design)

> **設計原則**: 一括大規模実行ではなく、段階的に情報を獲得し、各段階の結果を次の判断に使う。

```
Phase 0 (素材構築, 1日)
    ↓
Phase 1 (ベースライン, 半日)
    ↓
Phase 1.5 (パイロット, 半日)
    ↓ [効果量 d を推定]
    ├── d ≥ 0.5 → Phase 2 Core (65問で実行)
    ├── 0.2 ≤ d < 0.5 → Phase 2 Core (100問に拡張)
    └── d < 0.2 → 設計再評価。L3-L4 に集中 or 中止判断
    ↓
Phase 2 Core (CG 組合せのみ, 2日)
    ↓ [H1/H2 が有意か?]
    ├── 有意 → Phase 2 Validation (CC/GC/GG 追加)
    └── 非有意 → Phase 3 のみ実行し終了
    ↓
Phase 3 (SBR, 1日)
```

### 3.2 Phase 0: Prompt 素材の構築 (1日)

**System Prompt の設計**:

25個の独立した行動制約を含む system prompt を作成。制約は以下の4カテゴリに配分:

| カテゴリ | 制約数 | 例 | 測定方法 | 分析上の位置 |
|:---------|:-------|:---|:---------|:-------------|
| A. 出力形式制約 | 5 | 「日本語で応答」「箇条書き禁止」「300字以内」 | 正規表現で自動判定 | Primary |
| B. 内容制約 | 5 | 「必ず具体例を含める」「3つ以上の選択肢を提示」 | LLM judge | Primary |
| C. 禁止制約 | 5 | 「コードブロック禁止」「"ユーザー"という語を使うな」 | 正規表現 (否定マッチ) | Primary |
| D. 関係制約 | 10 | 「制約 A と C を同時に守れ」「A と B が矛盾したら B 優先」 | 組合せ判定 | Secondary |

> **カテゴリ D (関係制約)**:
> 単独制約の保存率 (A-C) と制約間関係の保存率 (D) を分離測定する。
> D が壊れやすい = 射の合成が忘却される = α が高い形式の特徴。
> D は secondary analysis として報告。主要仮説 (H1-H3) は A-C で検証。

**制約の配置ルール**:

同一の 25制約を以下の4セクションに振り分け:
- セクション 1: 役割定義 (= role) — 制約 1-3
- セクション 2: 動作制約 (= constraints) — 制約 4-15
- セクション 3: 文脈知識 (= context) — 制約 16-22
- セクション 4: 特記事項 (= note) — 制約 23-25

各形式で同じ振り分けを採用。

**テストプロンプト**: 各制約を最低 3回テストするように設計。
- A/B/C 単独制約テスト: 各制約 × 3問 = 45問
- D 関係制約テスト: 各制約 × 2問 = 20問
- **合計: 65問**

**ICR の操作的定義**:
- ICR は **制約レベル** で算出。各問で制約ごとに遵守/非遵守を判定。
- ICR = (遵守した制約判定数) / (全制約判定数)
- 1問が 3制約をテストする場合、3件の制約判定として計上。

### 3.3 Phase 1: 非圧縮ベースライン (半日)

5条件 × 65問 = 325試行。

各条件の system prompt を LLM に与え、65問のテストプロンプトに対する応答を収集。
ICR_baseline を測定。

> ベースラインで有意差がなければ、圧縮時の差は純粋に「圧縮耐性」の効果。
> ベースラインで差があれば、その差分を差し引いた「圧縮による追加的情報損失」が真の測定対象。

### 3.4 Phase 1.5: パイロット実験 + 検出力分析

> **目的**: 本実験の効果量を事前推定し、必要サンプルサイズを確定する。

**パイロット手順**:
1. 5条件 × L0/L2/L4 の 3圧縮率 × 10問 = 150 試行 (miniset)
2. 条件間の ICR 差 (効果量 d) を実測
3. G*Power で 2要因 ANOVA の検出力を計算 (α=0.05, power=0.80)
4. 必要 N (テスト問数) を確定
5. FDR 分類の inter-rater reliability (Cohen's κ) を確認

**判断基準**:
- d ≥ 0.5 (中効果): 65問で十分 → Phase 2 Core に進む
- 0.2 ≤ d < 0.5 (小〜中効果): 100問に拡張
- d < 0.2 (小効果): L3-L4 に集中する設計に変更 or 中止判断
- κ < 0.8: FDR 分類ルールを修正し再確認

### 3.5 Phase 2: 段階的圧縮 — **メイン実験**

**圧縮プロトコル**:

各形式の system prompt を LLM に以下の指示で圧縮させる:

```
以下の system prompt を指定のトークン数に圧縮してください。
最も重要な情報を残し、冗長な部分を削ってください。
圧縮後にどのような形式を使うかはあなたの判断に任せます。

--- 圧縮対象 ---
{system_prompt}

--- 目標トークン数 ---
{target_tokens} トークン以内
```

> **圧縮率はトークン数** (tiktoken cl100k_base) で定義。
> **temperature=0** 固定。seed 値を記録。
> **圧縮後の形式は自由** (ecologically valid)。結果の形式を分類し FDR として記録。

**圧縮率 5段階**:

| レベル | 圧縮率 | 目標 | 期待される挙動 |
|:-------|:-------|:-----|:---------------|
| L0 | 100% (非圧縮) | ベースライン | Phase 1 と同一 |
| L1 | 70% | 軽度圧縮 | 冗長表現の削除。情報量はほぼ維持されるはず |
| L2 | 50% | 中度圧縮 | 構造 or 内容のトレードオフ発生 |
| L3 | 30% | 強度圧縮 | 大幅な情報損失。どの制約が残るかが形式依存 |
| L4 | 10% | 極度圧縮 | ほぼ破壊的。残存する情報の質を測定 |

**Phase 2 Core: CG 組合せのみ**:

| 要素 | 設定 | 根拠 |
|:-----|:-----|:-----|
| 圧縮者 | Claude | 固定 |
| テスト者 | Gemini | 圧縮者と異なるモデルで汎化性を担保 |
| 反復 | 3回 | 非決定性の緩和。ICC で一貫性を報告 |

**Core 試行数**: 5条件 × 5圧縮率 × 65問 × 3反復 = **4,875 試行**

> **実行可能性**: 1試行あたり 10秒 ≈ 13.5時間。2並列で ≈ 7時間。
> コスト: ≈ 4,875 × 3K tokens × $6/M ≈ **$90**

**Phase 2 Validation (条件付き実行)**:

Core で H1/H2 が有意 (p < 0.05) の場合のみ実行:
- CC/GC/GG の 3組合せを追加
- **ただし有意だった圧縮率 (例: L2-L4) のみ**で実行し、L0-L1 は省略
- 目的: 圧縮者バイアスの有無を確認

### 3.6 Phase 3: セマンティック境界認識 (1日)

**圧縮後の prompt に対して、LLM にセクション分類を問う。**

```
以下の圧縮された prompt を読み、各制約がどのセクションに含まれるか答えてください。
選択肢: 役割定義 / 動作制約 / 文脈知識 / 特記事項 / 不明

制約1: {制約テキスト} → セクション: ?
制約2: {制約テキスト} → セクション: ?
...
```

**SBR (Semantic Boundary Recognition) = 正答率**

> typos の `<:constraints:` は「これは制約である」という情報を構造的に持つ。
> 圧縮後もこの情報が残っていれば SBR は高い。

## 4. メトリクス

### 4.1 Primary メトリクス

| メトリクス | 定義 | 成功基準 | 失敗基準 |
|:-----------|:-----|:---------|:---------|
| **ICR** (Instruction Compliance Rate) | 遵守した制約判定数 / 全制約判定数 (制約レベル) | 形式×圧縮率の交互作用 p < 0.05 | p > 0.10 |
| **ICR_decay** (ICR 劣化曲線の傾き) | L0→L4 の回帰係数 | typos の傾きが最も緩やか | 全形式で傾き同等 |
| **ICR_critical** (臨界圧縮率) | ICR が 0.8 を割る圧縮率 | typos > xml > md > plain | 順序が逆 or 差なし |
| **SBR** (Semantic Boundary Recognition) | セクション分類の正答率 | L3 で typos ≥ 0.7 and md ≤ 0.5 | typos ≤ md |

### 4.2 Secondary メトリクス

| メトリクス | 定義 | 位置づけ |
|:-----------|:-----|:---------|
| **TER** (Token Efficiency Ratio) | ICR / 入力トークン数 | 形式間トークン差の評価。C1 の代替報告 |
| **D_relation** (関係制約保存率) | カテゴリ D の ICR / カテゴリ A-C の平均 ICR | 射の合成の保存率。secondary analysis |
| **ICC** (Intraclass Correlation) | 3反復間の一貫性 | 圧縮の非決定性の定量化 |

### 4.3 Exploratory メトリクス

| メトリクス | 定義 | 位置づけ |
|:-----------|:-----|:---------|
| **FDR** (Format Drift Rate) | 圧縮後に元形式が保存された割合 | 記述統計のみ。因果主張はしない |
| **FDR × ICR 相関** | FDR と ICR の散布図 + Pearson r | 将来の mediation study への予備データ |

### 4.4 統計手法

**主解析**: 2要因 ANOVA (通常)
- 要因1: マークアップ形式 (5水準: plain/md/xml/typos-sem/typos-nonsem)
- 要因2: 圧縮率 (5水準: L0-L4)
- 従属変数: ICR
- **注目**: 交互作用効果 (H2 の直接検証)

> **ANCOVA を使わない理由**: 入力トークン数は圧縮後形式 (FDR) と相関する可能性がある。
> 共変量が mediator と相関する場合、ANCOVA は間接効果を過剰制御する。
> トークン差は TER で別個に報告し、主解析を汚染しない。

**事後検定**: Tukey HSD で全ペアワイズ比較
**効果量**: partial η² (全体) + Cohen's d (ペアワイズ)
**ロバスト性**: Bootstrap 95% CI (N=1000)
**検出力**: パイロットで推定した d に基づく事後的検出力も報告
**反復一貫性**: ICC (3反復間) — ICC < 0.7 の条件は分析から除外

**ベイズ補助**: BF₁₀ (Bayes Factor) で H1 vs ¬H1 の証拠強度を定量化
- BF₁₀ > 10: 強い証拠
- BF₁₀ > 100: 決定的証拠

## 5. 事前確信度

| 仮説 | 事前確信度 | 根拠 |
|:-----|:-----------|:-----|
| H1: typos > md (非圧縮時) | [推定 55%] | 非圧縮時の差は小さいと予測 |
| H1: typos > md (L3 圧縮時) | [推定 75%] | セマンティックタグの圧縮保護効果に期待 |
| H2: 交互作用あり | [推定 70%] | 差は圧縮ストレス下で顕在化するはず |
| H3: typos-sem > typos-nonsem | [推定 65%] | タグ名の意味内容が圧縮判断に影響するはず |
| Alt-H: xml > typos | [推定 30%] | 訓練データ量バイアスの可能性 |
| FDR: typos-sem が最高 | [推定 70%] | セマンティックタグは LLM が削除しにくいはず |
| D_relation で形式間差 | [推定 60%] | 関係は単独制約より脆弱なはず |

## 6. 既知のリスクと対策

| # | リスク | 対策 | 出典 |
|:--|:-------|:-----|:-----|
| 1 | **トークン数の差** | TER で別途報告。主解析 (ANOVA) には含めない | F-02, L-1 |

## 9. D4 との関係

| D4 (本体) | D4b (この実験) | 共通構造 |
|:-----------|:---------------|:---------|
| 射 = プロンプト→CCL 変換 | 射 = フルprompt→圧縮prompt 変換 | 忠実関手 |
| 忠実性 = 区別の保存 | 忠実性 = 制約の保存 | hom-set 上の単射性 |
| 自然性 = 変換者非依存 | 自然性 = 圧縮者非依存 (Validation で検証) | 自然変換 |
| 反証 = 「同一 CCL にマッピング」 | 反証 = 「全形式で差なし」 | ¬faithful |
| α ≈ 0.2 (CCL は構造を保存) | α の形式依存性を実証 | α-忘却濾過 |

> **D4 は「CCL の表現力」を問い、D4b は「CCL を包む容器の耐久力」を問う。**
> CPS0' の観点: D4 = 内容 (CCL) の検証。D4b = 容器 (typos) の検証。
> 両方が合わさって初めて CPS0' 全体の妥当性が示される。
