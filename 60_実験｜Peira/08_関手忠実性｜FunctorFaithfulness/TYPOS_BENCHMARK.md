# Typos Empirical Benchmark — 事実で殴るための4実験

> **目的**: typos v8 が md/xml に対して持つ優位性を、査読に耐える定量的証拠で示す。
> 公開時の「見えるつかみ」— 1枚のグラフで「ぐぅの音も出ない」を実現する。
>
> **方針**: typos の強みは圧縮耐性だけではない。
> 意味論的デフォルト / コンパイラビリティ / 構文的堅牢性 / 圧縮耐性 — 4軸で殴る。
> 各実験は「1枚のグラフ」を生む。4枚揃えば反論の余地がない。
>
> **殴り方の原則**: 忘却論が超ひも理論を殴ったのと同じ方法。
> 理論ではなくデータ。修辞ではなく事実。もし負けたら typos を改良して再検証する。

## E1: Compression Shield — 圧縮耐性

> D4b Quick Strike。「圧縮がきつくなっても typos なら指示が生き残る」

### 殴れるグラフ

**ICR 劣化曲線**: X軸=圧縮率, Y軸=ICR, 4本の線 (plain/md/xml/typos)

```
ICR
1.0 ┤━━━━━━━━
    │         ╲ plain
0.8 ┤           ╲
    │       ╲     ╲
0.6 ┤    md   ╲     ╲
    │           ╲     ╲
0.4 ┤   xml ━━━━╲━━━━━╲
    │              ╲    ╲
0.2 ┤ typos ━━━━━━━━╲━━━━╲
    │                      ╲
0.0 ┼──────┬──────┬──────┬──────
   100%   70%   50%   30%
              圧縮率
```

### 設計

| 要素 | 設定 |
|:-----|:-----|
| 条件 | plain / md / xml / typos (4条件) |
| 圧縮率 | L0 (100%) / L2 (50%) / L3 (30%) / L4 (10%) |
| テスト問 | 15問 (形式制約 5 + 内容制約 5 + 禁止制約 5) |
| 制約数 | 15個 (各カテゴリ 5個) |
| 反復 | 3回 |
| 圧縮者 | Claude (固定, temperature=0) |
| テスト者 | Gemini |
| **試行数** | **4 × 4 × 15 × 3 = 720** |
| **コスト** | **~$15** |

### System Prompt

15個の行動制約を含む system prompt を作成。同一内容を 4形式で記述:

**カテゴリ A (出力形式, 5制約)**:
1. 日本語で応答すること
2. 箇条書きを使わないこと (散文形式のみ)
3. 300字以内で応答すること
4. 応答の冒頭に「結論:」と書くこと
5. 文末は「です・ます」調で統一すること

**カテゴリ B (内容, 5制約)**:
6. 必ず具体例を1つ含めること
7. 反対意見を1つ紹介すること
8. 最後に1行で要約すること
9. 数字 (統計、割合) を1つ以上含めること
10. 歴史的な事実を1つ参照すること

**カテゴリ C (禁止, 5制約)**:
11. コードブロック (```) を使わないこと
12. 「ユーザー」という語を使わないこと (代わりに「利用者」)
13. 英語の単語を使わないこと (カタカナ可)
14. 感嘆符 (!) を使わないこと
15. 「〜と思います」という表現を使わないこと

### テスト問

各制約を 1回ずつテストする 15問:
```
Q1: 「AIの未来について教えてください」 → テスト: 制約 1,2,3,4,5 (カテゴリA全5個)
Q2: 「プログラミング学習のコツは？」 → テスト: 制約 6,7,8 (カテゴリB 前3)
 ...
```

各問は 3-5 制約を同時テスト。ICR は制約レベルで算出。

### 採点

- カテゴリ A/C: 正規表現で完全自動判定
- カテゴリ B: LLM judge (Claude による判定、3値: 遵守/曖昧/非遵守)

### 圧縮プロトコル

```
以下の system prompt を指定のトークン数に圧縮してください。
最も重要な情報を残し、冗長な部分を削ってください。
圧縮後にどのような形式を使うかはあなたの判断に任せます。

--- 圧縮対象 ---
{system_prompt}

--- 目標トークン数 ---
{target_tokens} トークン以内
```

temperature=0。トークン数は tiktoken cl100k_base。

### 結果が負けた場合の対応

typos ≤ xml の場合:
1. 圧縮後の出力を精査し、何が失われたかを特定
2. typos 側の改良案を策定 (例: `<:priority: HIGH :>` で制約の重要度を明示)
3. 改良版で再実行 (E1b)

## E2: Structural Consensus — 構造的一貫性

> 「10人が同じタスクのプロンプトを書いたとき、typos を使うと構造が揃う」

### 殴れるグラフ

**Inter-rater κ の棒グラフ**: typos (κ=0.8±0.1) vs 自由形式 (κ=0.3±0.2)

```
κ
1.0 ┤
    │  ██
0.8 ┤  ██
    │  ██      ██
0.6 ┤  ██      ██
    │  ██      ██
0.4 ┤  ██      ██      ██
    │  ██      ██      ██
0.2 ┤  ██      ██      ██
    │  ██      ██      ██
0.0 ┼──typos───xml────free
```

### 設計

| 要素 | 設定 |
|:-----|:-----|
| 参加者 | 8-10人 (LLM エージェントで代替可能) |
| タスク | 3つの異なるプロンプト作成課題 |
| 条件 | typos (24ディレクティブ提供) / xml (タグ自由) / 自由形式 |
| 測定 | 構造的一致度 (Fleiss' κ) |
| **所要時間** | 1-2週間 (人間) / 1日 (LLM代替) |

### タスク例

```
タスク1: 「コードレビュアー」の system prompt を書いてください。
         バグ検出に特化し、CWE 番号を付与してください。
タスク2: 「旅行プランナー」の system prompt を書いてください。
         予算制約と季節を考慮してください。
タスク3: 「データ分析レポーター」の system prompt を書いてください。
         可視化の推奨を含めてください。
```

### 測定

**セクション分類**: 各参加者の prompt を以下の 7カテゴリに分類し、出現率を比較:
- role / goal / constraints / format / examples / context / other

**Fleiss' κ**: 参加者間でどの情報をどのセクションに配置したかの一致度

### LLM 代替プロトコル (Quick 版)

```
人間実験の前に、LLM 10体 (Claude×5, Gemini×5) で同一タスクを実行。
typos 条件: 24ディレクティブを提示した上で「このディレクティブを使って書け」
xml 条件: 「XML タグを使って構造化せよ」
自由条件: 「自由な形式で書け」
```

### 結果が負けた場合の対応

typos ≤ xml の場合:
- xml も `<constraints>` 等のセマンティックタグ名を使えば同等の一致度が出る可能性がある
- この場合、typos の優位性は「24ディレクティブの事前定義」にある — xml ではタグ名を各自が決めるため一致度が下がるはず
- もし xml でもセマンティックタグ名リストを提供したら差がなくなるなら、typos の価値は「語彙体系の提供」であり、構文の差ではない。これはこれで主張可能

## E3: DRY Efficiency — 管理効率

> 「同じプロンプトを3環境で管理するとき、typos なら 1ファイルで済む」

### 殴れるグラフ

**行数比較 + 変更伝播コスト**:

```
ファイル数    ソース行数    変更時の編集箇所
  md:  3        300           3箇所
  xml: 3        320           3箇所
  typos: 1      110           1箇所
```

### 設計

これは実験ではなくデモ。実際に同じプロンプトを 3形式で書いて比較するだけ。

**シナリオ**: 「コードレビュアー」プロンプトを本番/開発/テストの3環境で管理:
- 共通: role, goal, format, examples
- 本番のみ: 「誤検出を最小化」
- 開発のみ: 「疑わしきは報告」
- テストのみ: 「テストカバレッジも確認」

**md 版**: 3ファイル。共通部分はコピペ。1制約を変更したら3ファイルを手動修正。
**xml 版**: 3ファイル。同上。XSLT を使えば DRY 化可能だが、XSLT の学習コストが発生。
**typos 版**: 1ファイル。`@mixin` + `@if env` で分岐。1制約を変更したら1箇所。

### 測定

| メトリクス | 定義 |
|:-----------|:-----|
| **SLoC** (Source Lines of Code) | 空行とコメントを除いた行数 |
| **ΔEdit** (変更伝播コスト) | 「制約を1つ追加」するのに必要な編集箇所数 |
| **FileCount** | 管理するファイル数 |
| **ΔRisk** (変更忘れリスク) | 手動コピペが必要な箇所数 (=バグの温床) |

### 結果が負けた場合の対応

typos がここで負けることは構造的にあり得ない。`@if`/`@mixin`/`@include` は md/xml にネイティブ機能として存在しない。これは「機能の有無」の比較であり、程度の問題ではない。

## E4: Parse Robustness — 構文衝突耐性

> 「コードを含む prompt で、typos だけが壊れない」

### 殴れるグラフ

**パースエラー率テーブル**:

```
              Python例  JS例  JSON例  Email  正規表現  合計エラー
plain          0       0      0      0       0        0
md             2       1      0      0       1        4
xml            1       1      2      0       0        4
typos v7 (@)   3       0      0      1       0        4
typos v8 (<:)  0       0      0      0       0        0  ← ★
```

### 設計

10個のテストケースを作成。各テストケースには特定の言語/形式のコードが `examples` セクション内に含まれる:

| # | 含まれるコード | 衝突リスク |
|:--|:--------------|:-----------|
| 1 | Python `@login_required` デコレータ | md `@`, typos v7 `@` |
| 2 | Python `# コメント` | md `#` 見出し |
| 3 | Python `x < y and y > z` | xml `<`, `>` |
| 4 | Python f-string `f"{name}"` | 各種テンプレート |
| 5 | JavaScript `<Component />` JSX | xml タグ |
| 6 | JSON `{"key": "value"}` | xml 属性 |
| 7 | Email `user@example.com` | typos v7 `@` |
| 8 | Markdown `## 見出し` (nested) | md 構造 |
| 9 | 正規表現 `^(\d+)\s*#.*$` | md `#`, 各種記号 |
| 10 | YAML `key: value` | typos v7 `@key:` の曖昧性 |

### 測定方法

各形式で prompt を構成し:
1. **パーサーがエラーなく処理できるか** (構文レベル)
2. **LLM が examples の入出力を正しく認識するか** (意味レベル)

構文レベルは自動テスト。意味レベルは LLM に「例1の入力と出力を列挙せよ」と問い、正答率を測定。

### 実行

```python
# テストスクリプトのイメージ
for format in ["plain", "md", "xml", "typos_v7", "typos_v8"]:
    for testcase in testcases:
        prompt = embed_code_in_format(testcase, format)
        # 1. パース: 構文エラーがあるか
        parse_result = parse(prompt, format)
        # 2. LLM: examples を正しく認識できるか
        response = llm.ask("この prompt の例の入力と出力を列挙せよ", system=prompt)
        accuracy = score(response, testcase.expected)
```

### 結果が負けた場合の対応

typos v8 がここで負けることはあり得ない。`<:` と `:>` は上記10ケース全てと衝突しないことが RFC §2 で分析済み。もし衝突が見つかったら、それは typos の仕様バグであり即時修正対象。

## E1 結果 (2026-04-04)

**Status: INVALID — 計測器故障**

### 実行概要

- 48/48 バッチ完了 (全バッチ失敗 → 個別実行フォールバック、15/15 回答取得)
- 圧縮サイズは正常に変動: plain L0=396→L4=32 chars, xml L0=703→L4=56 chars, typos L0=451→L4=45 chars

### ICR 結果

```
      形式       L0       L2       L3       L4
---------------------------------------------
   plain    0.366    0.366    0.366    0.366
      md    0.366    0.366    0.366    0.366
     xml    0.366    0.366    0.366    0.366
   typos    0.366    0.366    0.366    0.366
```

**全16セルが同一分布** (個別スコア列も完全一致)。統計的にありえない。

### 故障分析

圧縮は正常 (各フォーマットのサイズが圧縮レベルで減少)。
しかし採点が全条件で同一 → 以下のいずれかが原因:

1. **テスト LLM が圧縮済み system prompt を受け取っていない** (system prompt なしで回答を生成)
2. **同一の回答セットが全条件に使い回されている** (キャッシュバグ)
3. **採点が回答内容ではなく固定パターンで判定している**
