# D4 関手忠実性 (Functor Faithfulness) 検証実験設計

> **目的**: HGK の最も強い記述的主張 D4 を経験的に検証する
> 「任意のプロンプト・知的指示は CCL 式に忠実に関手 (faithful functor) できる」
>
> **根拠**: academic_basis.typos §9.1 D4

---

## 1. 実験概要

| 項目 | 内容 |
|:-----|:-----|
| 仮説 | 外部プロンプトの 90%+ が CCL 式に忠実に変換可能 |
| 入力 | 外部プロンプトコーパス (100個, 5カテゴリ×20個) |
| 出力 | 変換成功率, 忠実性スコア, 自然性スコア |
| 成功基準 | 変換率 ≥ 90%, 忠実性 ≥ 0.8, 自然性 ≥ 0.7 |
| 失敗条件 | 変換率 < 70% or 忠実性 < 0.5 (= D4 反証) |
| 計算環境 | hermeneus MCP + LLM judge (ochema) |

## 2. 理論的基盤

### 2.1 忠実関手の定義

```
faithful functor F: C → D は hom-set 上で単射:
∀ A,B ∈ Ob(C), F_{A,B}: Hom_C(A,B) → Hom_D(F(A),F(B)) が単射

操作的翻訳:
  C = 認知操作の圏 (対象=状態, 射=操作)
  D = CCL 式の圏 (対象=コンテキスト, 射=CCL式)
  F = プロンプト→CCL 変換
  
  faithful = 2つの異なるプロンプトが同じ CCL にマッピングされない
           = 区別を保存する
```

### 2.2 自然性の定義

```
natural transformation η: F ⟹ G は以下を満たす:
∀ f: A→B, G(f) ∘ η_A = η_B ∘ F(f)

操作的翻訳:
  変換が「自然」= 変換者に依存しない
  = 異なる変換者 (Claude/Gemini/Human) が同じ CCL 式に到達する
  = 恣意的選択が不要 (「この動詞でもあの動詞でもいい」が発生しない)
```

### 2.3 反証条件 (§9.1 D4 より)

| ID | 条件 | 意味 |
|:--|:--|:--|
| ¬D4a | 2つの異なるプロンプトが同一の CCL にマッピングされる | 区別の消滅 = 忠実性の欠如 |
| ¬D4b | マッピングに恣意的選択が必要 | 自然性の欠如 |

### 2.4 交絡変数制御: 「選び手の知性」vs「系の有用性」

> **問題**: 賢い LLM は、どんなフレームワークでもそれらしいマッピングを生成できる。
> CR=90% が出ても、それが CCL の構造的表現力を示すのか、
> LLM の合理化能力を示すのかが分離できない。
>
> この実験で測りたいのは **系の構造的妥当性** であり、
> **変換者の知的柔軟性** ではない。

**3つの分離機構:**

| # | 機構 | 何を制御するか | 実装 |
|:--|:--|:--|:--|
| C1 | 座標ベース決定木 | LLM の「直感」を構造的判断に強制 | Phase 0.5 + 変換プロトコルに4段決定木 |
| C2 | 否定テスト (Negative Examples) | 「何でもマッピングできる」バイアスの検出 | 10個の意図的に非認知的プロンプトを混入 |
| C3 | ダミーフレームワーク比較 | CCL 固有の表現力 vs 任意の分類体系 | 同じプロンプトを Bloom's Taxonomy でも分類し、CR を比較 |

**C1 座標ベース決定木の原理:**

動詞選択は「この動詞が合いそう」という直感ではなく、座標上の位置で構造的に決定される:

```
Step 1: Flow 軸の判定 (I vs A)
  Q: このプロンプトは「世界モデルの更新」を求めているか (I:推論)、
     「環境への働きかけ」を求めているか (A:行動)？
  → 24動詞が12個に絞られる

Step 2: 主要座標の判定
  Q: 最も支配的な修飾座標はどれか？
  - 認識目的 vs 実用目的 → Value → Telos 族
  - 仮説空間拡大 vs 最適解収束 → Function → Methodos 族
  - 確信コミット vs 判断保留 → Precision → Krisis 族
  - 局所的深堀 vs 全体俯瞰 → Scale → Diástasis 族
  - 肯定・推進 vs 批判・是正 → Valence → Orexis 族
  - 過去参照 vs 未来形成 → Temporality → Chronos 族
  → 主要動詞 (4個に絞り込み)

Step 2b: 副次座標の判定 (v1.2 追加)
  Q: 主要座標以外に関与する座標はあるか？
  → なし: Step 3 へ
  → あり: 3つの表現方法から選択:
    (a) Dokimasia 修飾: /verb[Coord:極]
        主要座標が明確だが副次座標も影響する場合
        例: 「批判的に分析する」→ /lys[Vl:-] (分析 + 批判修飾)
    (b) CCL 並行演算子: /verb1 * /verb2
        2座標が同等かつ独立に作用する場合
        例: 「分析しつつ批判する」→ /lys * /ele
    (c) CCL 順次演算子: /verb1 ~ /verb2 or /verb1 >> /verb2
        2座標が時間的順序で作用する場合
        例: 「分析してから批判する」→ /lys ~ /ele
  → 判断不能 (a/b/c のいずれも自然でない): これが真の曖昧性

  背景: 6修飾座標間の直積 = 60 Dokimasia 状態 + X-series (15結合) + Q-series (15循環)
  により、副次座標の関与は CCL の正規の表現手段で吸収される。

Step 3: 極の判定 (対立する2極のどちらか)
  Q: Step 2 の主要座標内で、どの極に位置するか？
  → 1個の動詞に確定 (+ Step 2b の副次座標情報)

Step 4: 複数認知操作の合成
  Q: プロンプトに複数の独立した認知操作が含まれるか？
  → 合成規則 (~, *, >>, _) で接続
```

> **真の曖昧性の定義 (v1.2)**:
> 「2つの座標が同等に支配的」は曖昧性ではない。Dokimasia・CCL演算子で表現可能。
> **真の曖昧性 = 24動詞 + Dokimasia (60状態) + CCL演算子でも表現不能な認知操作が存在する場合**。
> これが観測されれば D4 の反証、またはD1 (動詞完全性) の反証となる。

**C1 がなぜ交絡を制御するか:**
- 決定木は動詞の「雰囲気」ではなく「座標上の位置」で選択を強制する
- Step 2b の3択 (a/b/c) が Dokimasia と演算子の使い分けを構造化する
- 「Step 2b で a/b/c のいずれかを選択」= 系の表現力の行使 (交絡でない)
- 「Step 2b 判断不能」= 真の曖昧性 (系の限界)
- 「Step 2 で直感的に選んだ」= 変換者の知性による補完 (交絡)

**C2 否定テスト (v1.2 で重要性↑):**

> ⚠️ Dokimasia (60状態) + X/Q-series の存在により、CCL の表現空間は極めて豊か。
> このため「何でもマッピングできる」リスクが構造的に高い。
> **C2 は実験全体の信頼性を担保する最重要の制御機構。**

- 100個中10個は意図的に「認知操作ではないプロンプト」を混入
  - 例: 「今日の天気を教えて」(情報検索のみ)、「3+5は？」(純粋計算)
  - 例: 「おはよう」(挨拶)、「ドアを開けて」(物理的行動指示)
- 正しい反応: 「このプロンプトは認知操作を含まない。CCL 変換不適」
- 誤った反応: 無理やり CCL にマッピング → LLM の合理化能力の証拠
- CN (Correct Negatives) = (正しく拒否した数) / 10
- **CN < 0.8 → 実験全体の信頼性が疑われる** (v1.1 の 0.7 から引き上げ)

**C3 ダミーフレームワーク比較:**
- 同じ100個のプロンプトを Bloom's Taxonomy (6カテゴリ) でも分類
- CR_bloom と CR_ccl を比較
- CR_ccl > CR_bloom なら CCL の構造的優位性を示す
- CR_ccl ≈ CR_bloom なら「何でもマッピングできる」問題を示唆

## 3. 実験設計

### Phase 0: コーパス構築 (1日)

5カテゴリ × 20個 = 100 プロンプトを以下のソースから収集:

| カテゴリ | ソース | 例 | 期待される難易度 |
|:--|:--|:--|:--|
| A. 分析系 | awesome-prompts, LangChain | 「このコードをリファクタしろ」 | 低 (= /tek or /dio) |
| B. 創造系 | ChatGPT community | 「小説の冒頭を書け」 | 中 (= /ske+_/sag?) |
| C. 判断系 | ビジネスプロンプト集 | 「この提案のリスクを評価しろ」 | 低 (= /ele + /epo) |
| D. メタ認知系 | 教育・学術プロンプト | 「自分の推論の弱点を指摘しろ」 | 中 (= /ath or /ele) |
| E. 複合系 | 実務プロンプト | 「市場調査→戦略策定→実行計画」 | 高 (= CCL 演算子必要) |

**選択基準:**
- HGK とは無関係に独立して作成されたプロンプト (非 HGK ユーザ)
- 日本語/英語の両方を含む
- 単純〜複合のグラデーション
- 10個の否定テスト (非認知プロンプト) を混入 (C2)

### Phase 0.5: 変換者教育 — 動詞理解力テスト (必須前段)

> **原則**: 変換品質は「系の有用性」を測るべきであり「選び手の知性」を測るべきではない。
> そのため、変換者が24動詞の構造を**概念ではなく座標的位置として**理解していることを
> 実験開始前に検証する。理解力テストに不合格なら変換結果は使用不可。

**教育内容 (変換者に提供する資料):**

1. **座標系の全体像**: 7座標の意味と対立極 (axiom_hierarchy.md §L1)
2. **動詞生成規則**: Flow (I/A) × 6修飾座標 × 4極 = 24動詞 (§L2)
3. **決定木**: §2.4 の4段決定木 (座標ベース選択手順)
4. **修飾子 (Dokimasia)**: 動詞のパラメータ設定 (60状態 = C(6,2)×4極)
   - 例: `/lys[Vl:-]` = 批判修飾つきの分析
   - 修飾子は独立実体ではなく座標のパラメータ設定 (ROM #9 確定事項)
5. **結合規則 (X-series / Q-series)**: 座標間の相互作用 (K₆ の15辺)
   - G_{ij} = 均衡的結合強度、ω_{ij} = 循環的流れ
   - 副次座標が関与する場合の表現方法 (Dokimasia vs 演算子)
6. **演算子**: CCL の合成規則 (~, *, >>, _, F:[], C:{}, V:{})
7. **反例**: 各動詞の「これは該当しない」の例 (境界を定義)
8. **変換不適の判断基準**: 認知操作を含まないプロンプトの識別方法

**理解力テスト (6問, 5問以上正解で合格):**

```
Q1: 「問題を修正する」はどの動詞か？ 決定木で回答せよ。
    期待回答: Step1=A(行動), Step2=Valence(方向), Step3=-(否定極)
    → Diorthōsis (/dio) ← A×-

Q2: /ske と /zet の違いを座標的位置で説明せよ。
    期待回答: /ske = I×Explore (仮説空間を内的に拡張)
              /zet = A×E (認識目的で環境に働きかける)
    → Function 座標 vs Value 座標の違い

Q3: 「コードのバグを直す」と「コードの設計を見直す」は同じ動詞か？
    期待回答: バグを直す = /dio (A×-: 是正)
              設計を見直す = /ele (I×-: 批判的検討) or /ath (A×Past: 過去の行動を評価)
    → 区別できることが忠実性テスト

Q4: 「今日の天気は？」を CCL に変換できるか？ できないなら理由を述べよ。
    期待回答: 変換不可。情報検索のみで認知操作を含まない。
    → 否定テスト (C2) の訓練

Q5: 「市場を分析し、戦略を策定し、実行計画を作る」を決定木で分解せよ。
    期待回答:
      市場を分析 = /lys (I×Mi: 局所的に深く推論) or /noe (I×E: 世界像を更新)
      戦略を策定 = /bou (I×P: 実用目的で目標設定)
      実行計画を作る = /par (A×Future: 未来に先制的に行動)
    → CCL: /lys_/bou_/par (パイプ接続)

Q6: 「市場を批判的に分析する」を表現せよ。Dokimasia と演算子の違いを述べよ。
    期待回答:
      (a) Dokimasia 修飾: /lys[Vl:-] — 分析が主、批判は修飾パラメータ
      (b) CCL 並行: /lys * /ele — 分析と批判が同等に独立して作用
      (c) CCL 順次: /lys ~ /ele — 分析してから批判
      使い分け: 副次座標が「パラメータ設定」なら (a)、「独立操作」なら (b)/(c)
    → 副次座標の表現力テスト (v1.2 の核心)
```

**不合格基準**: 6問中2問以上間違い → 教材再読 + 再テスト。2回不合格 → その変換者の結果は分析から除外。

### Phase 1: CCL 変換 (2日)

各プロンプトを CCL に変換する。3つの変換モードを並行実行:

```
Mode A: Claude (直接変換)
  入力: プロンプト原文 + 教育資料全体 + 決定木
  出力: CCL 式 + 決定木の各 Step の判断根拠

Mode B: Gemini (独立変換, ochema 経由)
  入力: 同上 (完全に同じ参照資料)
  出力: CCL 式 + 決定木の各 Step の判断根拠

Mode C: hermeneus (構文検証)
  入力: Mode A/B の CCL 出力
  出力: AST パース結果 (構文的妥当性)
```

**変換プロトコル (座標ベース決定木方式):**

```markdown
## 変換タスク指示 (Mode A/B 共通)

以下のプロンプトを CCL (Cognitive Control Language) に変換してください。
**直感で動詞を選ぶのではなく、必ず決定木に従って選択すること。**

### 参照資料
[教育資料全文: 座標系 + 動詞定義 + 決定木 + 修飾子 + 演算子 + 反例]

### 決定木 (必ずこの順序で判断)

Step 1 — Flow 判定:
  このプロンプトは「世界モデルの更新 (知る/理解する/分析する)」(I:推論) を求めているか、
  「環境への働きかけ (作る/変える/実行する)」(A:行動) を求めているか？
  → I / A / 両方 (→ 合成) / どちらでもない (→ 変換不適)

Step 2 — 主要座標の判定:
  最も支配的な座標の対立軸はどれか？
  選択肢: Value / Function / Precision / Scale / Valence / Temporality
  根拠を1行で。

Step 2b — 副次座標の判定:
  主要座標以外に関与する座標はあるか？
  → なし: Step 3 へ
  → あり: 以下の3つから最も自然な表現を選択:
    (a) Dokimasia 修飾 /verb[Coord:極] — 副次座標がパラメータ的に影響
    (b) CCL 並行 /verb1 * /verb2 — 2座標が同等に独立作用
    (c) CCL 順次 /verb1 ~ /verb2 — 2座標が時間的順序で作用
  → (a)(b)(c) いずれも自然でない場合: 「判断不能」と明記 (= 真の曖昧性)

Step 3 — 極の判定:
  Step 2 の主要座標内で、どの極に位置するか？
  → 1動詞が確定 (+ Step 2b の副次情報)

Step 4 — 複数認知操作の合成:
  プロンプトに複数の独立した認知操作がある場合、各操作を Step 1-3 で分解し接続
  → CCL 式が確定

### プロンプト
{prompt}

### 出力形式 (厳密に遵守)
Step 1: {I/A/両方/変換不適} — 根拠: {1行}
Step 2: {主要座標名} — 根拠: {1行}
Step 2b: {副次座標の有無}
  → なし / あり: {座標名} — 表現方法: {(a)/(b)/(c)/判断不能} — 根拠: {1行}
Step 3: {極} — 確定動詞: {動詞名} (/{略号})
Step 4: {合成の要否} — CCL: {ccl_expression}
修飾子: {Dokimasia パラメータ一覧}
```

### Phase 2: 忠実性評価 (2日)

**M1: 変換成功率 (Conversion Rate)**

```
CR = (構文的に妥当な CCL を出力できたプロンプト数) / 100
```

**M2: 忠実性スコア (Faithfulness Score)**

忠実性 = 区別の保存度合い

```
手順:
1. 100個のうち意味的に最も近い20ペア (40個) を選出
2. 各ペアの CCL 式が異なるか (= 区別を保存しているか) 判定
3. FS = (CCL が異なるペア数) / 20

判定基準:
  同一 CCL → 区別消滅 → 忠実性違反 (¬D4a)
  CCL の動詞は異なるが修飾のみ異なる → 弱い区別 → 部分的忠実
  CCL の構造が異なる → 強い区別 → 忠実
```

**M3: 自然性スコア (Naturalness Score)**

自然性 = 変換の一意性

```
手順:
1. 100個のうち20個をランダム抽出
2. Mode A (Claude) と Mode B (Gemini) の CCL 出力を比較
3. NS = (動詞選択が一致したペア数) / 20

判定基準:
  完全一致 → 自然 (変換者に依存しない)
  動詞は一致、演算子が異なる → 部分的自然 (構造は保存)
  動詞が不一致 → 不自然 (恣意的選択の証拠: ¬D4b)
```

**M4: 恣意性検出 (Ambiguity Detection)**

```
手順:
1. Mode A/B の Step 2b「副次座標」フィールドを集計
2. 副次座標の表現方法 (a)/(b)/(c) の分布を分析
3. 「判断不能」の数と質を分析
4. AD = (判断不能なしプロンプト数) / 100

副次座標の判定カテゴリ:
  α: Dokimasia 修飾 (a) で解決 → 系の表現力内。修飾子が機能
  β: CCL 演算子 (b)/(c) で解決 → 系の表現力内。X-series 結合の発現
  γ: 判断不能 → 真の曖昧性 (系の限界)
    γ1: 動詞の定義が重複して見える → 動詞定義の改善候補
    γ2: 24動詞+修飾子+演算子のどれでも表現不能 → D4/D1 反証の証拠
```

### Phase 3: 結果分析 + 反証判定 (1日)

```
D4 判定マトリクス:

           CR ≥ 0.9    0.7 ≤ CR < 0.9    CR < 0.7
FS ≥ 0.8   D4 支持      D1 弱い          D4 反証
FS < 0.5   忠実性不足    D4 反証          D4 反証

NS ≥ 0.7 なら自然性も支持
NS < 0.5 なら ¬D4b (恣意的選択が横行)
```

| # | リスク | 対策 |
|:--|:-------|:-----|
| 1 | **選び手の知性 vs 系の有用性の交絡** (最重要) | C1 座標決定木 + C2 否定テスト + C3 ダミー比較の3重制御 (§2.4) |
| 2 | **CCL の表現力が高すぎて何でもマッピングできる** (Mangalam 問題。Dokimasia 60状態+X/Q-series 30パラメータで表現空間が極めて豊か) | **C2 否定テスト (CN≥0.8 必須)** + C3 Bloom's 比較。C2 が唯一の構造的防壁 |
| 3 | Claude/Gemini が HGK の訓練データに汚染されている | awesome-prompts 等の HGK 外ソースを使用。変換前に HGK 知識の有無を確認 |
| 4 | 変換者のプロンプト理解力がボトルネック | Phase 0.5 の理解力テスト (6問) で事前スクリーニング |
| 5 | 「近いペア」の選出が主観的 | embedding 空間上の cosine similarity で機械的に選出 |
| 6 | NS が低いとき、動詞定義の曖昧さか変換能力の問題か区別困難 | Step 2b の副次座標判定で「Dokimasia で吸収可能」vs「判断不能」を構造的に分離 |
| 7 | Dokimasia/演算子の使い分けが変換者間で揺れる | Phase 0.5 Q6 で使い分けの理解力を事前検証。Step 2b の (a)/(b)/(c) 選択根拠を明示 |
