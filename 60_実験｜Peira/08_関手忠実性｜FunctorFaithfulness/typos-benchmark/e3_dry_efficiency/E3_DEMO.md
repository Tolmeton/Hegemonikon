## E3: DRY Efficiency — 3環境プロンプト管理のデモ

> シナリオ: 「コードレビュアー」プロンプトを本番/開発/テストの3環境で管理する。
> 共通: role, goal, format, examples
> 本番のみ: 誤検出最小化、後方互換性重視
> 開発のみ: 疑わしきは報告 (false positive 許容)
> テストのみ: テストカバレッジも確認

---

### 1. Markdown 版 — 3ファイル必要

#### prod.md (ファイル1)
```markdown
## Role
シニアコードアナリスト

## Goal
コードレビューで見落としがちな隠れバグを検出し、修正提案を出す

## Constraints
- 実行時に問題を引き起こすバグに集中
- 各指摘には「発生条件」「影響度」「修正案」を含める
- CWE 番号を付与可能な場合は付与
- 誤検出を最小化すること
- 後方互換性を維持すること

## Format
JSON形式で出力

## Examples
(省略 — 共通)
```

### 2. XML 版 — 3ファイル必要 (同様)

#### prod.xml
```xml
<role>シニアコードアナリスト</role>
<goal>コードレビューで見落としがちな隠れバグを検出し、修正提案を出す</goal>
<constraints>
  <item>実行時に問題を引き起こすバグに集中</item>
  <item>各指摘には「発生条件」「影響度」「修正案」を含める</item>
  <item>CWE 番号を付与可能な場合は付与</item>
  <item>誤検出を最小化すること</item>
  <item>後方互換性を維持すること</item>
</constraints>
<format>JSON形式で出力</format>
<examples>
  (省略 — 共通)
</examples>
```

**注**: XML + XSLT で DRY 化は理論的には可能だが:
- XSLT は別言語の学習コスト
- プロンプトエンジニアが XSLT を書く現実は想像しにくい
- XML 単体では DRY 化不可能

**計測**: 3ファイル × 約12行 = 36 SLoC, 3ファイル

### 4. 定量比較

| メトリクス | md | xml | typos v8 | 勝者 |
|:-----------|:---|:----|:---------|:-----|
| **ファイル数** | 3 | 3 | **1** | typos |
| **SLoC** (examples除く) | 42 | 36 | **30** | typos |
| **共通部分の重複率** | 71% (3回コピペ) | 75% (同上) | **0%** | typos |
| **ΔEdit: 共通制約を1つ追加** | 3箇所 | 3箇所 | **1箇所** | typos |
| **ΔEdit: 環境固有制約を追加** | 1箇所 (該当ファイル) | 1箇所 | **1箇所** | 同等 |
| **変更忘れリスク** | HIGH (3ファイル同期) | HIGH | **ZERO** | typos |
| **条件分岐の表現** | 不可能 | XSLT 別言語 | **ネイティブ** | typos |
| **include/mixin** | 不可能 | 不可能 | **ネイティブ** | typos |

### 5. 「変更忘れ」のコスト

現実のシナリオ:
```
「CWE 番号の付与基準を変更する」
  md:   3ファイルの同じ行を手動編集 → 1ファイル更新忘れ → 本番で旧基準が適用
  xml:  同上
  typos: 1箇所を編集 → 全環境に自動反映 → 更新忘れが構造的に不可能
```

### 6. スケーラビリティ

環境が 3→10 に増えた場合:
- **md/xml**: 10ファイルに膨張。共通部分の変更コスト 10x
- **typos**: 1ファイルに `<:if:>` ブロック追加。共通部分の変更コスト 1x
