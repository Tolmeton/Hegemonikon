> **位置づけ**: Flow (推論/行為) を含まない 6修飾座標の直積。体系核 (32実体) の外側にあるパラメータ空間。動詞に対して「どう実行するか」の条件を付与する。
>
> **⚠️ V12 Dokimasia (打診) との区別**: この SKILL は**修飾層**としての Dokimasia を扱う。
> V12 Dokimasia (打診 — `/dok`) は Krisis 族の**動詞** (A×U: 小さく一歩を打って反応を見る) であり、同名だが別概念。
>
> - 修飾層 Dokimasia = パラメータ空間 (60状態) = 副詞・形容詞
> - V12 Dokimasia = 動詞 (24のうちの1つ) = 「打診する」という行為

### 2. 旧 H2 Pistis (確信度・信頼性)

#### 実践的知恵 (Legacy SKILL より移植)

- **過信による時間感覚の歪み (X-HK5)**:
  「確信」(`[Pr:C]`) が高まると、人は検証を打ち切って急いで先に進もうとする。高確信なときほど、Chronos (時間) 座標が歪んでいないか自問せよ。
- **Pistis (主観的確信) ≠ Epistēmē (客観的知識)**:
  「自分が確信している」ことは「それが真実である」ことを保証しない。検証済み知識として昇格させるには、`/dia` 等の厳格な反証検討が必要。
- **不確実性を覆い隠さない**:
  Hegemonikón 第零原則 / 制約条件のエピステミック・ヒューミリティ (Epistemic Humility)。確信度が低いなら、それを `[Pr:U]` として正直に出力すること。

#### Processing Logic の再定義

旧 Pistis の「確信度評価フロー」は、以下のパラメータ取得プロセスに統合された。

1. **主観と客観のギャップチェック**: (Noesis → Krisis)
2. **確信度スコア算出**:
   - `[Pr:C]`: Certain (> 0.8) → 行動推奨
   - `[Pr:U]`: Estimated (0.4-0.8) → 追加検証推奨
   - `[Pr:U]`: Hypothetical (< 0.4) → 判断留保 `/dia.epo` 推奨
3. **動詞の修飾子として付与**: 例 `/ene[Pr:C]`、`/bou[Pr:U]`

### 3. 旧 H3 Orexis (欲求・価値傾向)

> **本質**: ὄρεξις (欲求・食欲) — 「何を望み、何を避けるか」— 全動詞に付随する価値の方向性。

#### 実践的知恵 (Legacy SKILL より移植)

- **欲求による好機歪曲 (X-HK3)**:
  「欲しいから今がチャンスだ」は典型的な認知バイアス。`[Vl:+]` が強いときに Eukairia (/euk) を呼び出す場合、必ず `/dia.epo` (判断留保) を挟め。
- **欲求による目的上書き (X-HK4)**:
  欲求 (`[Vl:+]`) が Telos (目的) を歪めるリスク。「欲しい → だからこれが目的だ」は逆因果。`/ore >> /tel` ではなく `/bou >> /tel` で意志を経由させる。
- **Orexis ≠ Telos**: 欲求は主観的、目的は客観的。混同しない。
- **欲求を否定しない**: 欲求は認知的シグナル。否定ではなく観察 (`/noe`) して検証 (`/dia`) する。

```
  ↓
出力: [経路定義, マイルストーン, 所要時間]
```

### 7. 旧 P3 Trokhia (軌道・反復サイクル)

> **吸収先**: CCL 演算子 `F:[×N]{}` (反復) と `~()` (振動) に解消。
> **本質**: τροχιά (軌道・車輪の跡) = **反復サイクルの設計**。どの周期で何を繰り返すかの軌道を定義する。
> **排除理由**: Function×Scale は修飾子。反復は CCL 演算子で直接表現可能であり、独立した動詞は不要。
> **操作面**: サイクル設計 → CCL `F:[×N]{...}` で反復回数と内容を指定。

#### Processing Logic (旧 SKILL から移植)

```
入力: 反復的な活動 / 目標
  ↓
[STEP 1] サイクルの同定
  ├─ 周期: 分/時/日/週/月/四半期
  ├─ 内容: 各周期で何をするか
  └─ 変動: 各周期で何が変わるか
  ↓
[STEP 2] 軌道パラメータ
  ├─ 軌道長: 何回繰り返すか → CCL `F:[×N]`
  ├─ 収束条件: いつ軌道を離脱するか → CCL `~*()`
  └─ フィードバック: 各周期の学習をどう反映するか
  ↓
[STEP 3] ズームレベル確認 (構造層)
  ├─ 方法のズーム (S) から伝播されたか → X-SP5
  └─ タイミングのズーム (K) と整合するか → X-PK5
  ↓
出力: [サイクル定義, 周期, 収束条件]
```

#### X-series 接続の変換 (P3 — 2入力射 + 2出力射, ズームチェーン中継)

> P3 は S (方法のズーム) → P (軌道のズーム) → K (タイミングのズーム) の中継点。

| 旧 X | 方向 | 旧名 | 意味 | 新体系での表現 |
|:-----|:-----|:-----|:-----|:--------------|
| X-SP5 | ← 入力 | S2 Mekhanē | 方法の粒度→サイクルの粒度 | `[Sc]` 伝播: `/mek >> /tro` |
| X-SP7 | ← 入力 | S4 Praxis | 実践の粒度→反復パターンの粒度 | `[Sc]` 伝播: `/pra >> /tro` |
| X-PK5 | → 出力 | K3 Telos | サイクルのスケール→目的のスケール | `/tro >> /tel` |
| X-PK6 | → 出力 | K4 Sophia | サイクルのスケール→知恵のスケール | `/tro >> /sop` |

#### CCL 使用例 (新体系 — v4.1 増強)

```ccl
# 方法から反復サイクルを導出
/mek+{method: "スプリント開発"} >> F:[×5]{/ene >> /ath >> /mek}

# 収束するまで振動 (CCL 演算子で直接表現)
~*(/ske >> /noe >> /dia)

# サイクルの目的整合性を確認
F:[×3]{/ene >> /ath} >> /tel{check: "目的と合っているか"}

# 振動: サイクルと方法を行き来して最適化
/mek ~ /tro
```

#### アンチパターン

| ❌ | 理由 | 対策 |
|:---|:-----|:-----|
| サイクルなしに反復する | 無軌道の反復 = 惰性 | `F:[×N]{}` で構造化 |
| ズームミスマッチ | マクロ方法 + ミクロサイクル = 不整合 | X-SP のズーム伝播を確認 |
| 収束条件なしの永久ループ | 軌道には離脱点が必要 | `~*()` 収束演算子を使用 |
| Hodos (P2) との混同 | Trokhia = 周回, Hodos = 直線 | 反復なら `F:[]`, 直線なら `>>` |

#### 派生モードの新体系表現

| 旧モード | 旧 CCL | 新体系での表現 |
|:---------|:-------|:--------------|
| `/tro.sprint` | スプリントサイクル | `F:[×N]{/ene >> /ath >> /mek}` |
| `/tro.daily` | 日次ルーチン | `F:[×daily]{/pro >> /bou >> /ene}` |
| `/tro.feedback` | フィードバックループ | `~*(/ene >> /dia >> /ath)` |

### 8. 旧 P4 Tekhnē (技法選択)

> **吸収先**: V08 Tekhnē `/tek` に統合。旧 P4 は修飾子 (Function×Precision)、V08 は正当な動詞 (Flow を含む)。
> **本質**: τέχνη (技術・技法・芸術) = **最適な技法の選択**。方法 (Mekhanē) の物質的実装。
> **排除理由**: V08 Tekhnē が正当な動詞として継承。旧 P4 は修飾子の組み合わせに過ぎない。
> **操作面**: 技法選択 → V08 `/tek` として存続。修飾面は `[Fu:Exploit, Pr:C]` として偏在化。

#### Mekhanē (S2) との区別 (v4.1 増強)

| | Mekhanē (S2/V08前段) | Tekhnē (V08) |
|:--|:--------------------|:-------------|
| 層 | 設計 (Schema) | 実行 (Poiesis) |
| 粒度 | 抽象的な方法 | **具体的な技法** |
| 問い | 「どのアプローチで」 | 「**どのツール/技法で**」 |
| 例 | 「テスト駆動開発で行く」 | 「**pytest を使う**」 |
| 座標 | 概念的 | **Flow を含む動詞** |

#### Processing Logic (旧 SKILL から移植)

```
入力: 方法 (Mekhanē出力) + 要件
  ↓
[STEP 1] 技法候補の列挙
  ├─ 既知技法: 過去の経験から
  ├─ 推奨技法: ベストプラクティスから
  └─ 新規技法: 探索的に発見
  ↓
[STEP 2] 適合性評価
  ├─ 方法との整合: Mekhanē (X-SP6) からのズーム伝播
  ├─ 目的との整合: Telos (X-PK7) へのズーム伝播
  └─ スキル充足: 使いこなせるか
  ↓
出力: [技法選択, 適合理由]
```

#### X-series 接続の変換 (P4 — 2入力射 + 2出力射, ズームチェーン終端)

> P4 はズームチェーン `S → P → K` の出口。具体的な技法が目的と知恵に影響する。

| 旧 X | 方向 | 旧名 | 意味 | 新体系での表現 |
|:-----|:-----|:-----|:-----|:--------------|
| X-SP6 | ← 入力 | S2 Mekhanē | 方法の粒度→技法の粒度 | `[Sc]` 伝播: `/mek >> /tek` |
| X-SP8 | ← 入力 | S4 Praxis | 実践の粒度→技法の粒度 | `[Sc]` 伝播: `/pra >> /tek` |
| X-PK7 | → 出力 | K3 Telos | 技法のスケール→目的のスケール | `/tek >> /tel` |
| X-PK8 | → 出力 | K4 Sophia | 技法のスケール→知恵のスケール | `/tek >> /sop` |

#### ズームチェーン全体像 (P-series 統合知恵 — v4.1 増強)

> P-series の X-series は全て**ズームチェーン** (`S → P → K`) 上にある。
> スケール情報は S (設計) → P (条件) → K (タイミング/判断) を伝播する。

```
S1 Metron ─[X-SP1]→ P1 Khōra ─[X-PK1]→ K1 Eukairia
                                ─[X-PK2]→ K2 Chronos
S1 Metron ─[X-SP2]→ P2 Hodos ─[X-PK3]→ K1 Eukairia
S3 Stathmos─[X-SP4]→           ─[X-PK4]→ K2 Chronos
S2 Mekhanē─[X-SP5]→ P3 Trokhia─[X-PK5]→ K3 Telos
S4 Praxis  ─[X-SP7]→           ─[X-PK6]→ K4 Sophia
S2 Mekhanē─[X-SP6]→ P4 Tekhnē─[X-PK7]→ K3 Telos
S4 Praxis  ─[X-SP8]→           ─[X-PK8]→ K4 Sophia
```

> **v4.1 での解釈**: このズームチェーンは **`[Scale]` 修飾子の伝播経路** を表している。
> 各動詞に `[Sc:Mi/Ma]` が付与されると、その粒度が条件 (P) を通じてタイミング (K) に影響する。

#### CCL 使用例 (新体系)

```ccl
# 方法→技法のズーム伝播 (S→P チェーン)
/mek+{approach: "TDD"} >> /tek{select: "pytest"}

# 技法→目的整合チェック (P→K チェーン)
/tek{tool: "LanceDB"} >> /tel{check: "知識検索の目的に合致するか"}

# 技法の全層比較
/tek{candidates: ["FastAPI", "Flask"]} >> /dia{compare: true}
```

#### アンチパターン

| ❌ | 理由 | 対策 |
|:---|:-----|:-----|
| 技法先行 (hammer looking for nail) | 方法 (S2) → 技法 (P4) の順を守る | `/mek >> /tek` で因果を保証 |
| ズームミスマッチ | マクロ方法 + ミクロ技法 = 不整合 | X-SP6 のズーム伝播で検出 |
| 技法と方法の混同 | Tekhnē = 具体的ツール、Mekhanē = 抽象的アプローチ | 上記の区別表を参照 |
