## 0. 動機: 何が問題か

### コード検索の現状

既存のコード検索は 2 つの極端に偏っている:

| 手法 | 仕組み | 限界 |
| :-- | :-- | :-- |
| **テキスト検索** (grep, ripgrep) | 文字列マッチング | 構造を見ない。名前の一致だけ |
| **ベクトル検索** (embedding) | テキストを高次元ベクトルに射影 | 名前 (対象) を保存し、構造 (射) を忘却する |

どちらも **U_arrow (n=1 射の忘却)** を犯している (Aletheia §2.1):

- 対象 (変数名、関数名) だけを保存し
- 射 (入出力関係、合成構造、制御フローパターン) を捨てている

### 構造的同型の再利用価値

プログラミングにおいて最も再利用価値が高いのは **構造的同型** である:

```python
# 関数 A: ユーザーを名前でソート
def sort_by_name(users):
    return sorted(users, key=lambda u: u.name)

# 関数 B: 学生をスコアでランク
def rank_by_score(students):
    return sorted(students, key=lambda s: s.score)
```

対象 (名前) は完全に異なる。しかし構造は **同型**: `collection >> F:[each]{extract_key} >> sort`。

**問い**: この構造的同型を検出できる検索を、どう実装するか？

## 1. 核心命題: CCL は圏論の構文的実現である

### §1.1 CCL 演算子と圏論操作の対応

| CCL 演算子 | 圏論の操作 | 対応の水準 |
| :-- | :-- | :-- |
| `>>` (シーケンス) | 射の合成 ∘ | **A: 定義的同一** |
| `*` (融合/マージ) | テンソル積 ⊗ | **A: 定義的同一** |
| `%` (展開/直積) | 直積 × | **A: 定義的同一** |
| `F:[]{...}` (反復/マップ) | 関手の適用 F(−) | **A: 定義的同一** |
| `~` (双方向バインド) | 随伴 ⊣ | **B+: 概念的同型** |
| `<<` (逆算) | 反対圏 C^op | **B+: 概念的同型** |
| `\` (双対生成) | 双対関手 (−)* | **B+: 概念的同型** |
| `^` (メタ/リフレクション) | 2-cell (自然変換) | **B: 概念的同型** |
| `I:[] E:{}` (条件分岐) | 余積 (coproduct) | **B+: 概念的同型** |
| `V:{}` (フィルタ) | 部分対象分類器 (subobject classifier) | **B: 概念的同型** |
| `C:{}` (エラー処理) | モナド (Maybe/Either) | **B: 概念的同型** |
| `\|\|` (随伴宣言) | 随伴対 F⊣G の宣言 | **A: 定義的同一** |
| `\|>` (右随伴算出) | 右随伴 G の算出 | **B+: 概念的同型** |
| `<\|` (左随伴算出) | 左随伴 F の算出 | **B+: 概念的同型** |

**水準 A (定義的同一)** が 5 つ、**水準 B/B+ (概念的同型)** が 9 つ。
合計 14 演算子のうち 14 が圏論的対応を持つ。 **CCL に圏論的対応を持たない演算子はない。**

### §1.2 なぜ「構文的実現」と言えるか

圏論は通常、数学的記号 (∘, ⊗, ⊣, ⇒ 等) で記述される。
CCL はこれらの操作に **読みやすい構文** を与えている:

```text
圏論:  g ∘ f           CCL:  f >> g
圏論:  F(a) ⊗ G(b)     CCL:  F:{a} * G:{b}
圏論:  F ⊣ G           CCL:  F ~ G
圏論:  C^op            CCL:  << C
圏論:  α: F ⇒ G        CCL:  F ^ G
```

CCL は圏論に「名前を付けただけ」ではない。CCL は圏論の操作を:

1. **実行可能** にした (Hermeneus パーサーで AST に変換できる)
2. **人間が読める** ようにした (APL の情報密度 + 自然言語の可読性)
3. **ドメイン非依存** にした (認知でもコードでもデータでも同じ構文)

### §1.3 CCL 自由性原則

> **CCL の構文代数 (AST) は自由に保つ。マクロは等式 (同一視) ではなくパス (変換) として扱う。**
> — f611a2de セッションで確立

これにより CCL は **自由モノイダル圏** (またはそれに近い構造) を持つ [推定 65%]。

自由構成の普遍性により:

- 任意の圏 C に対して、CCL の生成元を C の対象/射に写す関手が **一意に** 存在する
- つまり CCL は圏論に対する **最も一般的な構文** として機能する

### §1.4 CCL は内部言語 (internal language) を持つ圏である

圏論において、圏が「内部言語」を持つとは、その圏の射と対象を**構文的に記述する言語**が存在すること。

| 概念 | 圏論 | CCL |
| :-- | :-- | :-- |
| 対象 | ob(C) | `/noe`, `/bou`, `/zet`, ... (24 動詞) |
| 射 | Hom(A,B) | 演算子 (`>>`, `*`, `~`, ...) |
| 射の合成 | g∘f | `f >> g` or `f_g` |
| 恒等射 | id_A | 深度なし `/noe` (自分自身への射) |
| 内部言語 | ラムダ計算 → CCC | **CCL 構文 → 自由モノイダル圏 (に近い構造)** |

CCL が内部言語である根拠:

1. **型規則が存在する**: ccl_grammar.md §5.2 で S/T/P/M の4型システムを定義
2. **well-formedness が構文的に判定可能**: BNF 生成規則 (§5.1) で有効な式を記述
3. **意味論的解釈が複数存在する**: 同じ構文が認知圏 (Cog) にも構造圏 (Struct) にもコード圏 (Code) にも射影可能

### §1.5 CCL は多圏的 (polycategorical) である

CCL の構造は単一の圏論的構造ではない。**複数の圏論的構造が互いに噛み合う系**:

| 圏論的構造 | CCL での実現 | 相互作用 |
| :-- | :-- | :-- |
| モノイダル圏 (⊗, I) | `*` (融合), `%` (展開) | `*` ⊣ `%` (随伴的双対) |
| 閉圏 (内部 Hom) | `L:[x]{...}` (ラムダ) | カリー化 |
| 2-圏 (自然変換) | `^` (メタ), `~` (振動) | 2-射としての認知操作 |
| 反対圏 (C^op) | `<<` (逆射), `\` (双対) | 圏全体の反転 |
| モナド (T, η, μ) | `C:{}` (エラー処理) | Kleisli 合成 |
| 随伴対 (F⊣G) | `~` (振動/双方向) | 最も CCL 固有の構造 |

**CCL は「複数の数学分野」を構文的に統一している。**

これは偶然ではない。CCL が認知操作 (FEP の能動推論) を記述するために設計され、認知操作自体が圏論的構造を持つため:

> FEP の能動推論 → 認知操作 → CCL → 圏論

**圏論は「構造の数学」であり、CCL は「構造の言語」。同じ対象を別の角度から記述している。**

### §1.6 CCL-PL: プログラミング言語としての CCL

PoC 実装 (§7 参照) により、CCL がプログラミング言語として機能することが実証された:

```text
CCL 入力:   /noe_/bou_F:[×3]{/ene~/ele}_/kat
Python 出力: v1=noe(); v2=bou(v1); for _ in range(3): v3=oscillate(ene,ele)(v2); v4=kat(v3)
```

CCL-PL の**他言語にない固有機能**:

| 機能 | 構文 | 意味 | 近似言語 |
| :-- | :-- | :-- | :-- |
| **自動双対** | `\encode` | 定義した関数の逆変換を自動取得 | なし |
| **逆射** | `goal << fn` | 目標から入力を逆算 | Janus (限定的) |
| **振動** | `A ~ B` | 2視点を交互に適用し平衡点を探索 | なし (AI ランタイム必要) |
| **深度修飾** | `fn+`, `fn-` | 同じ関数の深浅バリエーション | なし |

CCL-PL の位置づけ:

```text
         実行可能性
            ↑
  Python ─── CCL-PL ─── 数学記号
            ↑
      Haskell, APL
```

CCL-PL は Python の実行可能性と数学記号の表現力の中間に位置する。

### §1.7 CCL Grammar — 合成の well-formedness

CCL の文法は ccl_grammar.md で体系化されている (248 行)。概要:

**四則演算のアナロジー**:

| 算術 | CCL | 意味 |
| :-- | :-- | :-- |
| `+` 蓄積 | `_` (sequence) | 次にこれを |
| `×` 交差 | `*` (融合) | 一つにまとめて |
| `-` 除去 | `~` (振動) | 行き来して |
| `÷` 分離 | `>>` (射) | 変えて |

**合成の判定基準** (Kalon 判定):

- ◎ = 合成から3つ以上の認知パターンが展開可能
- ◯ = 有効だが自明
- ✗ = 冪等・不安定・情報損失が大きすぎる

例: `A >> B ~ C >> D` = ◎ (変換→振動→変換 = FEP サイクル) / `A * B * C * D` = ✗ (情報損失の累積)

**型システム** (§5.2):

```text
S (状態), T (変換), P (プロセス), M (メタ)
_ : S×S → S / >> : S→T→S / * : S×S → S / ~ : S×S → P / ^ : S→M / \ : T→T*
```

## 2. 理論的基盤: Aletheia のコード検索への適用

### §2.1 忘却関手としての CCL 変換

Aletheia (aletheia.md §1) の定理 U0: `F[U(q)] ≥ F[q]` — 忘却は VFE を増大させる。

Code → CCL 変換は **制御された忘却** (controlled forgetting):

```text
U_ccl: Code → CCL
  忘れるもの:  変数名、型名、コメント、空白、スタイル
  残すもの:    合成構造、制御フロー、データフローパターン
```

Aletheia のフィルトレーション (§3) をコード検索に転写:

| Level | Aletheia | CCL で保存されるか | 保存方法 |
| :-- | :-- | :-- | :-- |
| n=0 (対象) | — | ❌ 意図的に忘却 | 名前を捨てる |
| **n=1 (射)** | U_arrow | **✅ 保存** | `>>` で合成構造を明示 |
| **n=1.5 (合成)** | U_compose | **✅ 保存** | 関数呼出の連鎖がそのまま `>>` に |
| **n=2 (自然変換)** | U_depth | **◯ 部分保存** | `^` でメタ操作、`~` で双方向 |
| n=3 (豊穣) | U_precision | ❌ 未対応 | 将来: 確信度ラベルを付与 |

**CCL は n=0 を忘却し、n=1〜2 を保存する。** これは Aletheia フィルトレーションの
「ゴルディロックスゾーン」— 表層を捨て、構造を残す。

### §2.2 U⊣N 随伴としてのコード検索

```text
U: Code → CCL    (忘却関手: コードから CCL 構造式を抽出)
N: CCL → Code    (回復関手: CCL 構造式から候補コードを検索)

U ⊣ N: 構造的検索の随伴対
```

**N∘U 剰余 ρ ≥ 0 の意味** (Aletheia §5.5.3):

`sort_by_name` の CCL 構造式で検索 →
`sort_by_name` だけでなく `rank_by_score` も見つかる →
**元のコードにはなかった構造的同型を発見する** = 剰余 ρ > 0

**→ 再利用の数学的根拠: U⊣N の単位 η が非自明であること自体が、検索の価値を保証する。**

### §2.3 VFE の測定可能性

各 n レベルの構造を embedding に含めるか含めないかで、検索精度 (= VFE の逆数) が変化する。
これを実験的に測定できる:

```text
VFE(n=0, テキスト検索) > VFE(n=1, CCL 構造式) > VFE(n=1.5, CCL+呼出グラフ)
```

測定方法: 構造的に同型なコード対の検出率 (recall@k) を各レベルで比較。

## 3. CCL が圏論の数式化を「迂回する」メカニズム

### §3.1 直接経路 vs CCL 経路

```text
直接経路:  Code ─────────→ Category Theory
                ↑
           Curry-Howard + 依存型 + HoTT
           = 50年以上の研究問題

CCL 経路:  Code ──→ CCL ──→ Category Theory
              (AST変換)   (§1.1 の対応表)
              = 実装可能   = ほぼ自明
```

### §3.2 なぜ CCL 経路が可能か

1. **Code → CCL は AST 変換**: 有限個の変換ルールで機械的に実行可能
2. **CCL → Category Theory は §1.1 の対応表**: 構文的な読み替えだけ
3. **CCL の自由性**: 自由モノイダル圏の普遍性により、変換の一意性が保証される

### §3.3 既存アプローチとの比較

| アプローチ | 方向 | PL依存性 | 双方向性 | 実装状態 |
| :-- | :-- | :-- | :-- | :-- |
| Curry-Howard | 型 ↔ 命題 | Haskell/Agda/Coq のみ | なし | 成熟 |
| Moggi のモナド | 副作用 ↔ モナド | Haskell 中心 | なし | 成熟 |
| String diagrams | 視覚表現 ↔ モノイダル圏 | PL 非依存 | なし | 研究中 |
| Catlab.jl | Julia ↔ ACT | Julia のみ | なし | 開発中 |
| **CCL-IR** | **任意 PL ↔ 圏論的構造式** | **PL 非依存** | **✅ ~, <<, \\** | **パーサー既存** |

Lēthē (CCL-IR) の新規性:

1. **PL 非依存**: AST があれば任意の PL から変換可能
2. **双方向性が構文レベル**: `~` (随伴), `<<` (反対圏), `\` (双対) — 他のどのアプローチにもない
3. **実行可能**: Hermeneus パーサーが既に存在。形式化と実行が同一言語

## 4. コード検索への応用

### §4.1 検索パイプライン

```text
[ソースコード]
    │
    ▼ U_ccl (忘却関手: Code → CCL)
[CCL 構造式]
    │

```

### §4.2 変換ルール (Python → CCL)

| Python 構文 | CCL 構造式 | 圏論的対応 |
| :-- | :-- | :-- |
| `f(g(x))` | `x >> g >> f` | 合成 f∘g |
| `for x in xs: ...` | `xs >> F:[each]{...}` | 関手適用 F(−) |
| `if p(x): a else: b` | `x >> I:[p]{a} E:{b}` | 余積 |
| `[f(x) for x in xs if p(x)]` | `xs >> F:[each]{f} >> V:{p}` | フィルタ + マップ |
| `try: ... except: ...` | `... >> C:{...}` | Either モナド |
| `a, b = f(x), g(x)` | `x >> (f % g)` | 直積 |
| `yield x` | `>> !x` | Stream / コモナド |
| `lambda x: ...` | `L:[x]{...}` | 内部 Hom |
| `x = f(x)` (再代入) | `x >> f >> x` (暗黙移動) | 線形型的射 |

### §4.3 具体例: 構造的同型の検出

```python
# === 関数 A ===
def process_orders(orders):
    valid = [o for o in orders if o.status == "active"]
    totals = [calculate_total(o) for o in valid]
    return sum(totals) / len(totals)

# === 関数 B ===
def average_score(students):
    enrolled = [s for s in students if s.enrolled]
    scores = [s.grade for s in enrolled]
    return sum(scores) / len(scores)
```

CCL 変換後 (U_ccl):

```text
# 両方とも同一の CCL 構造式に変換される:
collection >> V:{predicate} >> F:[each]{extract} >> aggregate
```

→ embedding が一致 → 構造的同型を検出 → 再利用候補として提示

### §6.1 忘却の設計判断: 何を忘れ、何を残すか

| 対象 | 変換先 | 操作 | 根拠 |
| :-- | :-- | :-- | :-- |
| 引数 (`data`, `config`) | `¥` | **出自保存** (C3v2) | MB 外部入力 — データの出自は n=1 構造 |
| ローカル変数 (`result`, `temp`) | `#` | **出自保存** (C3v2) | MB 内部状態 — 名前は忘却、出自は保存 |
| ユーザー関数名 (`calc_total`) | `fn` | 忘却 | 名前は構造でない |
| 定数値 (`42`, `"hi"`) | `num_`, `str_` | 型のみ保存 | 値は構造でない |
| 組込関数 (`sorted`, `len`) | `sorted`, `len` | **保存** | 操作パターン = 射のラベル |
| 標準メソッド (`.append`, `.get`) | `.append`, `.get` | **保存** | 構造的に意味がある |
| 制御フロー | `>>`, `I:[]{}`, `F:[]{}` | **完全保存** | 射の構造そのもの |
| 空入力 (引数なし呼出等) | `()` | ユニット | CCL `_` (シーケンス演算子) との衝突回避 |

**核心**: 「名前 vs 構造」の二分法ではない。**構造的に意味のある名前は保存する** — `sorted` は関数名ではなく射のラベル。C3v2 では変数名の出自 (外部/内部) も n=1 構造として保存する。

#### §6.2.1 C3v2: ¥/# 変数トークン体系 (2026-03-22)

**問題**: C3 (`_a0`, `_a1`) は ρ を悪化させた (0.8784→0.8574)。原因は 2 つ:
1. `_aN` は対象の個別性 (n=0) を再導入し、忘却関手の設計に矛盾
2. `_` は CCL のシーケンス演算子 (operators.md §1.1 L143) であり、変数トークンとしての流用は仕様違反

**解決**: `¥` (MB 外部入力) / `#` (MB 内部状態) の 2 トークン体系を導入 (operators.md §1.5.5)。
- `¥` = 引数。Markov blanket を横断して入ってくるデータ
- `#` = ローカル変数。MB 内部で生成されるデータ
- `()` = 空入力 (ユニット)。`_` との衝突を回避

**圏論的意味**: 名前 (n=0) を忘却し、データの出自 (n=1) を射として保存する。VISION §2.1 の忘却関手設計に整合。

### §6.4 実装上の理論的洞察

**A. `not` は双対関手 `\` に対応**

Python の `not` → CCL の `\` (双対)。論理否定が圏論的双対関手として自然に解釈される。§1.1 の `\ = 双対関手 (−)*` の実装的確認。

**B. `and`/`or` は条件付き接続と選択型**

~~(旧: `and` → `*`, `or` → `+`)~~ — CCL に `+` は二項演算子として存在しない (単項「深化」のみ)。`and` → `*` は Dict の融合と衝突する。

operators.md 精読 (v7.6) からの修正:

| Python | CCL | 根拠 |
| :-- | :-- | :-- |
| `and` | `&` (条件付き接続) | `_` (単純逐次) と区別。`a and b` は条件付き (a が truthy なら b を評価) |
| `or` | `\|` (選択型) | CCL §1.5 に `A\|B` = Union が既存。「A または B」= 選択 |

`&` は CCL 正式演算子にはないが (cf. `&&` = 並列)、code→CCL の構造抽象として導入。`_` (単純逐次: `a; b`) と `&` (条件付き接続: `a and b`) の**構造的差異を保持する**ことが目的。`|` は CCL 型記法 (§1.5) に既存であり、Union セマンティクスが Python `or` と一致する。

**C. Dict は融合 `*`、Tuple は直積 `%`**

`{k1:v1, k2:v2}` → `(k1 % v1) * (k2 % v2)`。データ構造の種類がモノイダル構造の選択に自然に対応する。

#### A. CCL 構文と Python の対応は機械的に可能 (P2 部分検証)

| CCL | Python | 実装上の障壁 |
| :-- | :-- | :-- |
| `A_B_C` | `v1=A(); v2=B(v1); v3=C(v2)` | なし |
| `A*B` | `merge(A(), B())` | なし |
| `A%B` | `product(A(), B())` | なし |
| `A~B` / `A~*B` / `A~!B` | `oscillate(A, B)` / desugar: `for` ループ | なし (desugar で解消) |
| `A^` | `meta(A)(items)` / desugar: `[A(x) for x in items]` | なし (desugar で解消) |

**結論**: 全演算子が Python に変換可能。desugar モードではランタイム依存もゼロ。

#### B. 双対 (`\`) は言語レベルの新機能 (P1 の強化)

既存のプログラミング言語で「関数を定義すれば逆変換が自動的に利用可能」にする言語は存在しない [推定 85%]。

```python
# CCL-PL 固有の機能
@dual_of(encode)
def decode(data): ...

# \encode → 自動的に decode() が呼ばれる
# \(encode_compress_encrypt) → decrypt >> decompress >> decode (自動逆パイプライン)
```

これは **随伴対 (F⊣G) そのものの構文化**: 「左随伴を定義すれば右随伴が使える」。

他の言語での近似:

- Haskell: 型クラスで `inverse` を手動定義 (自動ではない)
- Prolog: 双方向推論はあるが関数レベルではない
- Janus: 可逆言語だがドメインが制限的

**→ `\` は CCL-PL を構造的に差別化する核心機能。**

#### C. 逆射 (`<<`) はプログラミングの新パラダイム (P1 の拡張)

`>>` (目標に向かって順方向に実行) の双対として `<<` (目標から逆算) を実装した。

```text
>> の意味: A >> condition → 条件に到達するまで A を繰り返す
<< の意味: goal << A → goal を得るために A の入力を推定する
```

これは **圏論の反対圏 C^op の構文的実現** であり:

1. テスト駆動開発 (目標出力から入力を逆算) の構造的表現
2. 逆問題 (inverse problem) の一般的な構文
3. 微分可能プログラミングの反伝播との構造的類似

#### D. 全演算子は構文的に表現可能 (Creator の洞察による修正)

> **旧結論 (v0.2)**: `~` (振動) は「AI が必要な意味論的操作」であり、静的コード変換の限界を示す。
> **修正 (v0.3)**: Creator の洞察により否定された。全演算子は構文層で表現可能。

Creator の発見: `~` は `F:[×N]{a_b}` で desugar できる。`^` はネスト構造 (MB の入れ子) に対応する。

```text
# 旧: ~ は AI ランタイムが必要
/noe ~ /ele  →  oscillate(noe, ele)  # ランタイム依存

# 新: ~ は F: で desugar 可能
/noe ~ /ele  →  for i in range(5):   # 純粋 Python
                    state = noe(state)  # 往路
                    state = ele(state)  # 復路
```

**修正後の二層モデル**:

| 層 | 内容 | 担当 |
| :-- | :-- | :-- |
| **構造** (全演算子) | `>>`, `*`, `~`, `^`, `<<`, `\` | 機械的変換 (desugar で実証) |
| **パラメータ** (判断) | 何回？ 収束した？ どの分岐？ | AI or 人間 |

**→ 旧§1.1 の A/B 水準区分は「演算子の分割」ではなく「構造 vs 精度 (S-III)」に修正。** AI が必要なのは演算子の実行ではなく、パラメータ (precision) の最適設定。

#### E. 演算子の合成: 事実上無限の表現力 (新知見)

実装の過程で、CCL 演算子の合成可能性が確認された:

```text
\(A ~* B)    → diverge(A, B)     # 収束の双対 = 発散
\(A * B)     → product(A, B)     # 融合の双対 = 展開
goal << \A   → dual(dual(A))(goal) = A(goal)  # 二重双対 = 恒等
```

演算子の合成は **自由モノイダル圏の普遍性** (§1.3) を実装レベルで確認するものであり、P4 の部分的検証となる。

| **構文層** (Syntax) | 最小の数学的構造を保証 | 循環 (cycle): A→B→A→B→... | 融合: Dict の merge |
| **意味層** (Semantics) | 入力が条件を満たすときに豊かな構造を付与 | ソレノイダル循環 (Helmholtz Q 成分) | ← 随伴 `* ⊣ %` が成立するなら Limit |

**核心**: 構文層は「何も仮定しない」。意味層は「入力が随伴性を持つなら、構造がリッチになる」。

### §9.3 `~` = Helmholtz Q 成分

HGK の K₆ 二重テンソル場における Q (非均衡的循環) と CCL の `~` が対応:

- 構文的: `A~B` = A と B の間の循環 (回数は収束条件 `*` で制御)
- 意味的: A と B が随伴対 F⊣G を持つなら、`~` はソレノイダル循環 (Helmholtz 分解の Q 成分)
- 体系的: G (均衡的結合) は `_` (逐次合成) に対応。Q (循環) は `~` に対応

```
  → 白色化は忘却の不均一性そのものを消し去る
```

### §19.5 脱圏化カスケードと Aletheia フィルトレーション

「力とは忘却である」§4.6h の脱圏化カスケードと §2.1 の Aletheia フィルトレーションは**同一構造**のそれぞれ抽象的/操作的表現である。

```
脱圏化カスケード (§4.6h):              Aletheia (§2.1):
──────────────────────────            ──────────────────────────
Level 0: 状態 (1-cell)                n=0: ソースコード (名前+構造)
         ↓ U (脱圏化)                       ↓ U_label (名前を忘れる)
Level 1: 不均一 (0-cell)              n=1: 射の構造 (U_arrow)
         ↓ F (再圏化)                       ↓ U_compose (合成を忘れる)
Level 2: 構造 (1-cell')              n=1.5: 基本パターン (U_compose)
         ↓ U' (脱圏化)                      ↓ U_depth (深さを忘れる)
Level 3: 概念/力 (0-cell')           n=2: 型パターン (S/T/P/M)
```

| 対応 | 脱圏化カスケード | Aletheia |
|:--|:--|:--|
| 最も豊かな表現 | 状態 (1-cell) = 場 $\phi(x)$ | n=0: ソースコード全文 |
| 最初の忘却 | U: 関係を忘れ点的な値 | U_label: 名前を忘れ構造を保存 |
| パターンの発見 | F: 0-cell 間のパターンを検出 | Aletheia U0: 忘却後の検索で recall 増加 |
| さらなる忘却 | U': 再び差分を取る | U_depth: 実装深度を忘却 |

> **Aletheia U0 (§2.1) の再解釈**: 「$F[U(q)] \geq F[q]$」(忘却後の検索は忘却前以上) は、脱圏化カスケードの Level 0 → Level 1 遷移で**ノイズ (名前) を除去し、信号 (構造) を増幅する**ことと同値。脱圏化は情報を捨てるが、捨てる対象が**ノイズ方向**ならば、信号方向の S/N は上がる。
>
> **Ξ との接続**: Ξ > 0 は「忘却が不均一 = ノイズ方向だけを選択的に忘却」を意味する。したがって Aletheia U0 が成立する必要十分条件は Ξ > 0 (忘却が不均一であること) [推定 70%]。
