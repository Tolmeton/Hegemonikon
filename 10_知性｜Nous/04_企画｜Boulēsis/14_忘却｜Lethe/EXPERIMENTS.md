### §11.5 VFE 測定法

Aletheia フィルトレーション (§2 (ビジョン.md).1) の各 n レベルで recall@k を測定:

```text
recall@k(n) = |{正例 ∩ top-k 結果}| / |正例|
```

| n | 検索方法 | 期待される性質 |
| :-- | :-- | :-- |
| 0 | テキスト全文 | 名前一致には強いが構造的同型に弱い |
| 1 | CCL U_arrow | 射構造を保存 → 構造同型に強い |
| 1.5 | CCL U_compose | 合成パターン保存 → P3a で実証済み |
| 2 | CCL + S(e) 加重 | 座標情報追加 → さらに精密 [仮説] |

**VFE の操作的定義**:

$$\text{VFE}(n) = 1 - \text{recall@10}(n)$$

VFE が低いほど構造保存が良い。Aletheia U0 (F[U(q)] ≥ F[q]) により:

$$\text{VFE}(0) \geq \text{VFE}(1) \geq \text{VFE}(1.5) \geq \text{VFE}(2)$$

この不等式が実験的に成立するか検証する = **Aletheia 理論の実験的実証**。

### §11.9 構造理解 vs 文字列一致 (アブレーション実験結果)

**問題**: CodeBERT (ρ=0.532) と TF-IDF (ρ=0.067–0.521) がなぜ同等の ρ を示すか。2つの解釈:

| 解釈 | 意味 | 含意 |
| :-- | :-- | :-- |
| A: CCL の構造が文字列レベルで情報を持つ | CodeBERT は構造トークンを「理解」している | embedder は構造を活用できる。改善余地あり |
| B: 両者とも文字列一致に依存 | 「構造理解」は幻想 | CCL 表現自体の改善が必要 |

**アブレーション実験結果** (n=200関数, 500ペア, 2026-03-22):

| 条件 | CodeBERT ρ | Δρ | TF-IDF ρ | Δρ | **反応差** |
| :-- | :-- | :-- | :-- | :-- | :-- |
| C0 (baseline) | **0.532** | — | 0.067 | — | — |
| C1 (構造トークン破壊) | 0.423 | **-0.110** ⬇️ | 0.268 | +0.201 ⬆️ | **-0.310** |
| C2 (トークン順序シャッフル) | 0.391 | **-0.141** ⬇️ | 0.067 | ±0.000 | **-0.141** |
| C3 (非構造トークン除去) | 0.379 | -0.153 ⬇️ | -0.052 | -0.119 ⬇️ | -0.034 |

**結論: 解釈 A 優勢 (A:6 vs B:0)**

核心的発見は **C1 の反応差 = -0.310**:
- CodeBERT: 構造トークン (`>>`, `_`, `%`) 破壊で ρ **低下** (-0.110) → 構造トークンを理解している
- TF-IDF: 同じ操作で ρ **上昇** (+0.201) → 構造トークンはストップワード的ノイズ
- **同じ ρ でもメカニズムが完全に異なる** — TF-IDF ≈ CodeBERT 問題は表面的類似に過ぎなかった

補足知見:
1. **C2 (シャッフル)**: CodeBERT は順序構造を捉えている (-0.141)。TF-IDF は bag-of-words なので完全不変 (±0.000) — sanity check 成功
2. **C3 (正規化)**: 非構造トークン (関数名 `pred`, `fn` 等) も情報を持つ。CodeBERT (-0.153) も TF-IDF (-0.119) も低下 → 関数名が構造距離と弱い相関を持つ (構造的に似た関数は似た名前を使う傾向)
3. **統計的安定性 (置換検定)**: 自己完結型スクリプト (n=500 permutations) による検証で、構造破壊 (C1) がもたらす相関変動の効果は乱数シードにほとんど依存せず極めて安定していること ($\sigma < 0.002$) が実証された。抽出される構造情報は偶然の産物ではない。

## 13. CodeBERT 反転 — 「ある が 取り出せない」(2026-03-20)

> **構造情報は LLM の内部に「ある」。しかし単一ベクトルでは「取り出せない」。**
> **Phase B の反転パターンは、Phase C の設計哲学を根本から転換する。**
> — confound_removal_analysis.md + Phase B2 attentive probing

### §13.1 反転の発見

Phase B (Structural Probe) では3モデルの hidden states から構造類似度を予測した。結果は2つの段階で劇的に変化した:

#### Stage 1: 単一ベクトル線形プローブ (structural_probe.py)

最終トークンの hidden state 1本を用い、交絡因子 (コード長・トークン重複・識別子重複・ASTノード数・インデント深度) を OLS 残差法で除去した偏相関:

| モデル | raw ρ ピーク | 偏ρ ピーク | 偏ρ ピーク層 | p (置換検定) |
| :-- | :-- | :-- | :-- | :-- |
| CodeBERT (125M) | 0.87 (L6) | **0.45** (L6) | L6/12 | — |
| CodeLlama (7B) | 0.35 (L3) | **0.17** (L14) | L14/32 | 0.046 |
| Mistral (7B) | 0.38 (L2) | **0.20** (L18) | L18/32 | 0.012 |

**層シフト**: 交絡因子を除去すると、raw ρ が高い浅層 (L2-3) から中深層 (L14-18) にピークがシフトする。浅層の相関は表層的 (コード長等に駆動)、中深層に圧縮された構造情報が浮かび上がる。

**分散分解 (R²)**: CodeLlama で構造の残留寄与は **7.4%**、Mistral で **8.9%**。「ゼロではない」が正当な主張だが、大きいとは言えない。

**暫定結論**: 構造情報は「ある」が、単一ベクトルからの読み出し量は限定的。

#### Stage 2: Attentive Probing (nonlinear_probe.py) — 反転

全トークンの hidden states にアテンション重み付き集約 + MLP プローブを適用:

| モデル | 単一ベクトル偏ρ | Attentive 偏ρ | 倍率 |
| :-- | :-- | :-- | :-- |
| CodeBERT (L12) | 0.45 | **0.74** | ×1.6 |

**反転の核心**: 偏ρ が 0.45 → 0.74 に跳ね上がる。5-fold CV 全 fold で p < 10⁻⁷。

これは「構造情報は単一トークンに集約されていないが、複数トークンに分散して存在している」ことを意味する。Attentive Probing が学習したアテンション重みは、構造情報を持つトークン位置を選択的に重み付けしている。

### §13.2 llm_body_draft.md との接続

この発見は [llm_body_draft.md](file:///home/makaron8426/Sync/oikos/01_ヘゲモニコン｜Hegemonikon/10_知性｜Nous/04_企画｜Boulēsis/12_遊学｜Yugaku/01_研究論文｜Papers/llm_body_draft.md) §7 (ビジョン.md) の「人間中心的忘却関手」と直結する:

| 概念 | llm_body_draft の用法 | Lēthē での実験的対応 |
| :-- | :-- | :-- |
| U_anthropocentric | 人間が「身体がない」と判断する忘却関手 | 単一ベクトル probe = 人間的な「1点で読む」方法 |
| implicit structure | LLM が暗黙に獲得した構造情報 | Attentive probe で検出される分散的構造表現 |
| access ≠ possession | 持っていることとアクセスできることは別 | 偏ρ 0.45 (1点) vs 0.74 (分散) |

**U_anthropocentric の操作的定義**: 構造情報の有無を「1つの観測点から読み出せるか」で判断する関手。Attentive Probing は、この関手を通さない直接的アクセスを提供する。

### §13.3 Phase C 設計哲学の転換 — ハイブリッド設計 (v2: 2026-03-22)

この発見は Phase C (§12 (ROADMAP.md)) の設計哲学を根本から変える。ただし **偏ρ = 0.74 は R² ≈ 0.55 に相当し、分散の 45% は未説明**。「構造情報は既にある」は半分正しいが、「0.74 しかない」も同時に正しい。

**→ Phase C は「明示化 AND 注入」のハイブリッド設計が必要。**

| 側面 | 旧 (§12 (ROADMAP.md) 執筆時) | v1 (反転直後) | **v2 (ハイブリッド)** |

## 15. 忘却関手の4評価関係 — ρ を超えて (2026-03-22)

> **R2/R3 が未検証の状態で ρ を最適化しても、忘却関手の品質改善にはならない。**

### §15.1 4つの関係

VISION §0 (ビジョン.md) (構造的同型の検出) と §2 (ビジョン.md).1 (忘却の設計判断) から、忘却関手 U が満たすべき4つの性質を導出する。

| 関係 | 定義 | 検証する性質 | 圏論的対応 |
| :-- | :-- | :-- | :-- |
| **R2 不変性** | 表層変換 σ に対し `U(f) = U(σ(f))` | 忘れるべきを忘れているか | U∘σ = U (自然変換に対する不変) |
| **R3 感度** | 構造変換 τ に対し `U(f) ≠ U(τ(f))` | 残すべきを残しているか | U が faithful (忠実) |
| **R1 同型** | `f ≠ g` かつ `U(f) ≅ U(g)` | 構造的同型を検出できるか | U の kernel が非自明 |
| **R4 粒度** | `d_struct(f,g) > d_struct(f,h) ⇒ d_CCL(f,g) > d_CCL(f,h)` | 構造差の順序を保存するか | U が距離の単調写像 |

### §15.2 検証階層

R4 は R2 + R3 の連続的拡張であり、R2/R3 が前提:

```text
L1: R2 (不変性)    ← 前提なし。最初に検証
L2: R3 (感度)      ← R2 が前提 (不変性がないと感度を測れない)
L3: R1 (同型)      ← R2 + R3 が前提
L4: R4 (粒度)      ← R2 + R3 + 適切な GT。= ρ の改善版
```

```text

```

**ρ は LQS=1 の世界でのみ意味がある。LQS < 1 なら ρ の値は信頼できない。**

> ⚠️ **Caveat**: R1 同型 = 同一 CCL を GT とする測定は、CCL Levenshtein 距離が完全分離を達成することを**自明に保証**する。より挑戦的なテスト (近似 CCL ペアに対する AST TED の相関) は将来の課題 (→ §15.7 ロードマップ)。

**近似 CCL ペアの構造相関テスト (P22)** — 746 ペア (2026-03-23)

P21 の「完全一致 (d=0) は自明」という批判に応えるため、完全一致以外のペアにおいて「CCL の距離が AST の距離を予測できるか」をテストした。

| ゾーン (d_CCL) | ペア数 | 平均 CCL 距離 | 平均 AST TED | ゾーン内 ρ |
|:---|---:|---:|---:|---:|
| 完全一致 (d=0) | 100 | 0.000 | 0.483 | N/A |
| **近似** (0 < d ≤ 0.3) | 301 | 0.203 | 0.607 | **0.126** |
| **中距離** (0.3 < d ≤ 0.6) | 199 | 0.491 | 0.696 | 0.156 |
| **遠距離** (d > 0.6) | 146 | 0.781 | 0.776 | 0.465 |

**解釈**: 
1. **大域的な単調性**: ゾーンごとの平均 AST TED は `0.483 → 0.607 → 0.696 → 0.776` と、CCL 距離の増加に伴い綺麗に**単調増加**している。CCL は関数間の大域的な構造の遠近感を正しくマッピングできている。
2. **近傍での相関の弱さ (ρ=0.126)**: 近似領域 (d≤0.3) においては、CCL 距離は AST 距離の差をほとんど予測できない。これは欠陥ではなく**忘却関手 (L14 Lethe) の本質**である。AST における大きな変化 (例えば長大な IF 文の中にさらにログ出力を足すなど) は、構造の骨格が変わらなければ CCL では吸収され (d_CCL が小さいまま)、逆に骨格が少しでも変われば d_CCL は敏感に反応する。この非線形な「丸め込み」があるからこそ、汎化的な検索が成立する。

## 16. CCL 文字列の自己充足性 — 木構造は冗長 (2026-03-22)

> **CCL 文字列は既に構造を直接エンコードしている。木変換は情報を増やさず、ノイズを加えるだけ。**
> — Creator × Claude, 662e0fc2

### §16.1 実験的根拠 (Wave 2: TED 実験)

CCL 文字列を木構造に変換し Tree Edit Distance (TED) を計算、Levenshtein 距離と比較した。

| メトリクス | ρ | 備考 |
| :-- | :-: | :-- |
| Levenshtein (トークン列) | 0.7244 | CCL 文字列そのままの比較 |
| TED (最初の木) | 0.3646 | `()` `[]` `{}` 全てをネスト化 — 過剰な木構造 |
| TED (括弧 flatten) | 0.5576 | `()` を平坦化、`{}` のみネスト |
| ハイブリッド (α=0.63) | **0.7408** | α×Lev + (1-α)×TED — TED の寄与は +2.3% |
| Siamese Network (Wave 9) | **0.8438** | 文字列から学習距離を直接学習 |

**解釈**: TED の最良結果 (ハイブリッド α=0.63) でも Levenshtein に対し +2.3% の改善に留まる。Wood の純寄与は極めて小さい。

### §16.2 なぜ木は冗長か

CCL は **構造の線形化された表現** として設計されている (§1 (ビジョン.md))。文字列自体が構造:

| CCL 構造要素 | 圏論的意味 | 文字列でのエンコード | 木変換で増える情報 |
| :-- | :-- | :-- | :-- |
| `>>` | 射の合成 ∘ | トークン順序 = 合成順序 | **なし** (兄弟順 or 二分木、どちらも同じ) |
| `{...}` | 関手適用 F(−) | 括弧ペア = 階層 | **なし** (括弧と同構造) |
| `I:[]{} E:{}` | 余積 (分岐) | ラベル+括弧 = 分岐 | **なし** |
| `F:[each]{...}` | 自然変換適用 | ラベル+括弧 = 反復 | **なし** |

→ CCL の設計が優秀すぎて、木にする余地がない。追加情報は 0、変換のノイズだけが加わる。

### §16.3 §11.9 との接続

本節は §11.9 の **解釈 A** を支持し、§11.9 アブレーション実験がその精密化を提供する:

> A: CCL 文字列は構造情報を直接エンコードしている。CodeBERT はこれを活用するが TF-IDF は文字列一致のみ。

§16 (TED 冗長) + §11.9 (アブレーション) の統合: CCL 文字列は**構造の線形化表現として自己充足**しており、木構造への変換は不要。一方で CodeBERT は文字列の中の構造トークンと順序を活用しており (C1: -0.110, C2: -0.141)、TF-IDF の bag-of-words とは質的に異なるメカニズムで動作する。

### §16.4 含意: 改善の方向

| 方向 | 根拠 | §11.9 からの追加根拠 | 優先度 |
| :-- | :-- | :-- | :-- |
| ❌ 木変換の改良 | 冗長な変換をいくら改良しても情報は増えない | — | 棄却 |
| ✅ CCL 生成の改善 (C3v2 等) | `¥`/`#` トークンでデータフロー・内部状態を明示化 | C3 (非構造トークン除去) で CodeBERT/TF-IDF 共に ρ 低下 → 関数名にも構造情報あり。CCL に構造情報を**増やす**余地がある | **最高** |
| ✅ 学習距離 (Siamese) | 文字列から直接非線形距離を学習 — ρ=0.84 達成済み | CodeBERT が順序・構造を活用 (C1/C2) → Siamese も同様の特徴を学習している可能性 | **高** |

## 17. 連続 CCL — 3つの研究方向 (2026-03-23)

> **起源**: CALM 論文 (Shao et al. 2025, arXiv:2510.27688) の @read から着想。
> CALM は「離散トークン → 連続ベクトル」(対象 n=0 の連続化) を行った。
> CCL の連続化は「射 n=1 と射の合成 n=1.5 の連続化」を意味し、本質的に異なる。
>
> **核心洞察**: CCL は圏論の構文的実現 (§1 (ビジョン.md))。CCL の連続化 = 構造そのものの連続化。
> 「ほぼ同型だが少し違う」構造の類似度を、離散一致ではなく **連続的な距離** として測定可能になる。
>
> **具体例**: 離散 CCL では `collection >> V:{predicate} >> F:[each]{extract} >> aggregate` は同型検出のみ。
> 連続 CCL では `collection >> V:{0.8·predicate} >> F:[each]{0.9·extract + 0.1·transform} >> aggregate` のような
> **準同型の類似度** を測地線距離として厳密定義可能。

### §17.1 方向 A: 正方形モデル L2 としての理論的定式化

> **何**: CCL 式の空間に [0,1]-豊穣構造を導入し、射の連続化を理論的に定式化する

HGK の正方形モデル (axiom_hierarchy.md v4.2.0) は既に「離散→連続」の移行を記述:

| 正方形モデル | Hom 値 | CCL での意味 |
|:---|:---|:---|
| L1 (前順序圏) | {0,1} | CCL 式は離散的。同型 or 非同型 |
| **L2 ([0,1]-豊穣圏)** | **[0,1]** | **CCL 式間に連続的な「距離」が存在** |
| L3 (豊穣 bicategory) | [0,1] + 2-cell | CCL 式間の「経路」(連続変形) が存在 |

やること:
1. CCL 式の集合 E に [0,1]-Hom を定義: d(e₁, e₂) = 構造的距離
2. 三角不等式・対称性の検証 → Lawvere 距離空間として定式化
3. Drift と CCL 距離の関係: Drift(G∘F) = d(e, G∘F(e)) として統一可能か

**接続先**: §2 (ビジョン.md) (Aletheia フィルトレーション), §9 (ビジョン.md) (2層意味論の意味層), §15 (ρ の理論的基盤)

**リスク**: [0,1]-Hom の具体的構成が CCL の構文構造と整合するか不明 [仮説 55%]

### §17.2 方向 B: CALM 的 Autoencoder による CCL ベクトル空間の実装

> **何**: CCL 式を連続ベクトル空間に射影する Autoencoder を構築する

CALM の core insight の CCL への適用:

| CALM | CCL 版 |
|:---|:---|
| K 個のトークン → 1 連続ベクトル | CCL 式 (可変長) → 固定次元連続ベクトル |
| Context-free Autoencoder | Context-aware Autoencoder (CCL は構造的文脈を持つ) |
| VAE + KL clipping | VAE + 構造保存正則化 (射の合成順序を保存) |
| 99.9% トークン再構築 | 目標: 高忠実度 CCL 再構築 (演算子・動詞・修飾を復元) |

CALM の「再構築が良い ≠ 潜在空間が良い」の教訓を活かす:
- 再構築精度 (Q1 関手性) と潜在空間ロバスト性 (Q2 随伴品質) を独立に最適化
- VAE 正則化 = 忘却関手 G の「滑らかさ」を確保する操作

**接続先**: §4 (ビジョン.md) (コード検索パイプライン — embedding 層の置き換え), §6 (ビジョン.md) (code_ingest.py の CCL → ベクトル化)

**利点**: §8 (ROADMAP.md).1 で P3 を実証した cosine 距離を、理論的に基礎づけられた CCL ベクトル空間距離に置換可能

**リスク**: CCL 式のデータ量が十分か (学習に必要な CCL 式の多様性) [仮説 45%]

### §17.3 方向 C: Energy Score による構造品質の Likelihood-Free 測定

> **何**: strictly proper scoring rule で CCL 式の品質を bluff 不可能な形で測定する

CALM が開発した Energy Score + BrierLM フレームワークの直接応用:

| 用途 | 定式化 | 意義 |
|:---|:---|:---|
| **Drift 測定** | Drift(G∘F) = Energy Score (P_before, P_after) | 単一数値 → **分布としての Drift 推定** (不確実性付き) |
| **Kalon 判定** | Kalon(x) ⟺ Energy Score (x, G∘F(x)) が最大 | strictly proper → **偽 kalon が構造的に排除** |
| **CCL 品質** | 生成モデル P(e) vs 真の分布 Q(e) の Energy Score | Code → CCL 変換の品質を likelihood-free で測定 |
| **温度制御** | Rejection-based sampling for CCL generation | precision parameter (Pr:C vs Pr:U) の操作化 |

Energy Score の 2 項 (diversity + fidelity) は VFE の 2 項 (Accuracy - Complexity) と構造的に双対:

```text
Energy Score: E[‖x-y‖] - ½E[‖x-x'‖]
             fidelity     diversity

VFE:        -Accuracy   + Complexity
            fidelity     diversity (≈ model simplicity)
```

**接続先**: §10 (ビジョン.md) (忘却スコア S(e) — S(e) は离散的。Energy Score は連続版), §9 (ビジョン.md) (2層意味論)

**利点**: 即座に実験可能。既存の CCL embedding + サンプリングで Energy Score を計算するだけ

**リスク**: 低い。Energy Score の計算自体は単純。解釈の妥当性が検証対象 [推定 75%]

### §17.4 3方向の関係

```text
         (A) 理論的定式化
         [0,1]-豊穣圏
             ↑
    数学的基盤を提供
             |
(C) Energy Score ←──── 測定道具 ────→ (B) Autoencoder
    品質測定             距離空間        ベクトル空間
    即時着手可能          の具体化        の構築
```

- A → B: 理論が実装の設計を制約する (どの構造を保存すべきか)
- A → C: 理論が測定の解釈を与える (Energy Score は何を測っているか)
- B → C: Autoencoder が Energy Score の入力を生成する
- C → A: 実験結果が理論の妥当性を検証する
- C → B: Energy Score が Autoencoder の訓練目標になる

**3方向は直交かつ相互波及可能。どこから着手しても他に波及する。**

## 18. Code→Code 構造検索 — U⊣N 随伴の N を実装する (2026-03-23)

> **離散 CCL 式の Levenshtein/cosine 距離では「近い構造」の順序は測れるが、「構造そのものの類似」は測れない。**
> **43d CCL 特徴量ベクトルは CCL 式の構造的骨格を直接エンコードし、名前を完全忘却した状態で検索を可能にする。**
> — Creator × Claude, 0ff85a02 セッション

### §18.1 Code→Code 構造検索の定義

Code→Code 構造検索 = 「あるコード片に構造的に類似するコード片を、名前やコメントに依存せず発見する」操作。

Lēthē の枠組みでは:

| 概念 | Lēthē の対応 | 実装 |
| :-- | :-- | :-- |
| 忘却関手 U | Code → CCL → 43d ベクトル | `python_to_ccl()` → `ccl_feature_vector()` |
| 回復関手 N | CCL 空間での最近傍 → 元コード候補 | `ccl_feature_index.search_similar()` |
| 剰余 ρ | N∘U(f) ⊇ {f} — 自分自身以外の同型も発見 | E2E テスト: `python_to_ccl` → `_collect_assignments`(0.879) |
| フィルトレーション | n=0 忘却 (名前), n=1 保存 (射), n=1.5 保存 (合成) | 43d = n=1〜1.5 の構造情報のみ |

**§0 (ビジョン.md) の問題**: 既存のコード検索は U_arrow を犯している (テキスト検索 = 名前一致のみ、ベクトル検索 = 名前を保存し構造を忘却)。
**§18 の解**: U_arrow を**意図的に適用**し、名前を忘却した状態で射の構造を直接検索する。忘却は欠陥ではなく武器。

## 20. CCL 型推論 — Aletheia フィルトレーション n=2 (TypeSeq) (2026-03-24)

> **操作を忘れ、計算の流れパターンを保存する。**
> **これは U_type: CCL → TypeSeq — Aletheia フィルトレーションの新レベル。**
> — Creator × Claude, 2026-03-24

### §20.1 動機: さらなる忘却

§18 の 43d 特徴量は CCL 式のトークン分布を数値化するが、**何の型のトークンか**を区別しない。VISION §1 (ビジョン.md).6 の型システム:

```text
S (状態), T (変換), P (プロセス), M (メタ)
_ : S×S → S / >> : S→T→S / * : S×S → S / ~ : S×S → P / ^ : S→M / \ : T→T*
```

この型情報を CCL 式から**自動推論**し、計算の流れパターン (TypeSeq) として保存する = **操作を忘却し、流れのみを残す忘却関手 U_type**。

### §20.2 Aletheia フィルトレーション拡張

```text
filtration:  Code → CCL → TypeSeq
             n=0    n=1    n=2
             名前   構造   流れ
```

| n | 忘却 | 保存 | 表現 | 実装 |
|:--|:--|:--|:--|:--|
| 0 | なし | 全て | ソースコード | — |
| 1 | 名前 | 射構造 | CCL (U_ccl) | `python_to_ccl()` |
| 1.5 | 名前+対象 | 合成パターン | 43d 特徴量 | `ccl_features()` |
| **2** | **名前+対象+操作** | **計算の流れ** | **TypeSeq** | **`ccl_type_seq()`** |

### §20.3 型推論ルール

| トークン/パターン | 型 | 根拠 |
|:--|:--|:--|
| `¥`, `#`, `str_`, `num_`, `nil_`, `bool_`, `()` | **S** | データ = 状態 |
| `fn`, 組込関数名, `.method`, `pred`, `\`, 制御構造 | **T** | 関数/射 = 変換 |
| `~` (振動) | **P** | `~ : S×S → P` |
| `^`, `[def]` | **M** | `^ : S→M`, メタ定義 |
| `>>`, 括弧, 構造子 | (型付けしない) | 合成子 |

**セグメント (>> 間) の型判定** — 優先順位: P > M > T > S。

### §20.4 TypeSeq の具体例

```text
入力 CCL: "¥ >> fn >> V:{pred} >> F:[each]{fn} >> return"
TypeSeq:  [S, T, T, T, S]
意味:     入力→変換→変換→変換→出力 (4段パイプライン)
```

TypeSeq が同一の関数は、**操作が異なっても計算の流れが同じ** = n=2 レベルの構造的同型。

**type_flow の意味**:

### §27.3 P42 の理論的説明 — 確定

> **P42: 幾何的注入 > テキスト注入。理由 = 忘却関手 ker(U_F) の差。**

Paper XI §3.1-3.2 の枠組みで:
- F₁ (自然言語/CCL テキスト) = full だが faithful でない → 構造情報が忘却される
- F₂ (49d 幾何) = faithful だが full でない → 構造情報が保持される
- 差 = ker(U_F₁) > ker(U_F₂)。F₁ は射の区別を失う (>> 支配、木構造消失)、F₂ は射の区別を保つ

**「テキスト注入が幾何注入に劣る」は注入方式の優劣ではない。** 忘却関手の核 (kernel) の大きさの差。構造情報はどちらにも含まれるが (構造はあらゆる具体に存在する)、テキスト化のパイプラインで忘却される量が異なる。

C-mini (125M, ρ=0.963) が C v2 13B (ρ=0.857) に勝つのは:
- C-mini: 49d → Structural Attention → hidden (忘却関手を 0 回通過)
- C v2: CCL テキスト → トークナイザ → attention → hidden (忘却関手 F₁ を 1 回通過)
- F₁ 1 回の通過で Δρ = 0.106 の構造情報が失われる

1. **Level 5 (49d 直接注入) は天井** — アーキテクチャ変更が必要だが ρ=0.963
2. **Level 2-3 (テキスト注入) は 85% を獲得可能** — ただし 13B パラメータが必要
3. **Level 2.5 (トポロジー抽出) は 33% を回復** — zero-cost だが天井が低い
4. **Level 5 + 2.5 は R@10 で Level 5 単独を超える** — faithful × full の積で相補的

**v3 の設計空間:**
- 道 A: Level 5 の改善 — 49d を超える幾何的表現 (structural_formula_v2 の S4/S5 方向)
- 道 B: Level 2.5→3 の引き上げ — トポロジー特徴をより faithful に (v2 改善方向)
