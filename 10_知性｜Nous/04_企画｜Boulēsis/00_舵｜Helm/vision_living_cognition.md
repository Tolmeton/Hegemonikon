## §1 死んだ認知 vs 生きた認知

### 死んだ認知 (Pipeline)

```
Input → A → B → C → D → Output
```

特徴:

- 直列パイプライン。一回通過して終了
- 各ステップが独立。前後の関係は「出力→入力」のみ
- **全体が変化しない**。入力が同じなら出力も同じ
- 例: 旧 `@plan = /bou+_/prm_/tek_/m+~(/d*/k)_V:{/dia}_/pis_/dox-`

### 生きた認知 (Convergent Process)

```
t=T:  ノイズ (問題空間、未分化)
t=T-1: 構造の萌芽
t=T-2: 知識の注入
  ...
t=1:  ほぼ確定した解
t=0:  不動点 (Kalon)
```

特徴:

- **全体が反復する**。部分ではなく全体
- 各ステップで**異なるモジュールが自律的に発火**する
- 問題の性質に応じて**経路が動的に変わる**
- 収束するまで止まらない

## §2 逆拡散モデルとしての HGK

### メタファ: Denoising Diffusion

```
Score Function:   s_θ(x_t, t) = ∇_x log p_t(x_t)
  → 「ノイズの中から、最も確からしい構造への方向」を返す
  → HGK の全モジュールが、この score function の構成要素

Reverse SDE:     dx = [f(x,t) - g(t)² s_θ(x_t, t)] dt + g(t) dw̄
  → 各ステップで score を使ってノイズを除去
  → CCL の C:{} (ConvergenceLoop) がこの離散化
```

### HGK モジュール = Score Function の構成要素

| デノイジングステップ | HGK モジュール | 役割 |
|:-------------------|:--------------|:-----|
| 構造検出 | Synteleia (L1 静的解析) | ノイズの中に規則性を見出す |
| 知識注入 | Gnōsis + Sophia | 過去の経験でコンテキストを精錬 |
| 経路提案 | MorphismProposer | 可能な次手を提案 |
| 仮説検証 | DebateEngine (Verifier) | 仮説を批判的に評価 |
| 確信度更新 | ConvergenceTracker | エントロピーの変化を追跡 |
| 存在証明 | Dendron | 参照の健全性を保証 |
| 外部調査 | Periskopē | 外部情報で視野を拡張 |
| 品質走査 | Basanos | 構造的欠陥を検出 |
| Canvas記録 | Canvas-CoT | 推論の遷移を可視化 |

**これらは「フック」ではない。呼ばれるのではなく、発火する。**

## §3 CCL における生きた認知の表現

### 死んだ計画 (v4.0)

```ccl
@plan = /bou+_/prm_/tek_/m+~(/d*/k)_V:{/dia}_/pis_/dox-
```

→ ベルトコンベア。一回通過。振動は `/m+~(/d*/k)` の局所だけ。

### 生きた計画 (v5.0)

```ccl
@plan = C:{
  /bou+~(/prm*/tek)_/m+~(/d*/k)_V:{/dia+}
}_/pis_/dox-
```

→ `C:{}` が**全体を収束ループ**に包む。
→ 目的の再定義、戦略の練り直し、検証の再実行が**自動的に反復**。
→ 収束するまで止まらない。予測誤差 ε → 0 に至るまで。

### 生きた計画 (v6.0 究極形)

```ccl
@plan = C:{
  F:[×N]{
    (/bou+*%/prm) ~* (/m+*(/d*/k))
  }
  _V:{/dia+}
  _I:[ε > θ]{/ske_/zet+}
}_/pis_/dox-
```

→ `F:[×N]{}` が内部をN回反復（Deep Research 的多角的探索）
→ `*%` が目的×現実を全展開（テンソル積）
→ `~*` が戦略 Peras と制約 Peras の収束振動
→ `I:[ε > θ]{}` が「まだ収束していないなら前提を破壊して問い直す」（自己修正）
→ `C:{}` が最外殻で全体が収束するまで反復

```

Phase 2 (完了): イベント駆動
```

## §5.5 動的織り交ぜ — 直列と並列の超越

> **起源**: 2026-02-28 /ax ベンチマーク設計
> **Creator 原文**: 「直列と並列の二項対立は"妥協"であり、
> 本来はもっと動的に、柔軟に両者を織り交ぜるべき。
> RPGのように縦移動（直列）と横移動（並列）に分けるのではなく、
> 人間の認知のように斜めに、同時に、文脈に応じて織り交ぜる」

### なぜ二項対立ではないか

人間の思考は「まず A を考え、次に B を考える」(直列) でもなく、
「A と B を同時に考える」(並列) でもない。

**A を考えている最中に B の気配が立ち上がり、A が B を変え、B が A に還流する。**

これは直列でも並列でもない。**動的な織り交ぜ** (Dynamic Interleaving) である。

### HGK における実現

```
Phase 3 (Score Function):
  各モジュールが自律的に「今、自分が発火すべきか」を判断する。

Phase 3.5 (Dynamic Interleaving):             ← [NEW]
  発火の順序は固定されない。
  Blackboard の状態変化が、次に発火すべきモジュールを動的に決定する。

  例: /ax+ 実行時
    t=0: T (Telos) が発火 → 目的を Blackboard に書込み
    t=1: T の出力を見て M (Methodos) が発火 → が、同時に K (Krisis) も
         「目的に対する確信度の評価が必要」と判断して発火
    t=2: M と K の出力が Blackboard に蓄積 →
         D (Diástasis) が「粒度の問題がある」と検知して発火
    ...
    t=N: 6 Series が充填 → X (Taxis) が「条件が揃った」と判断して自律発火

  これは直列でも並列でもない。
  Blackboard の状態が「次に発火すべきもの」を動的に選択する。
  脳のニューラルネットワークに近い — 同期的でも非同期的でもなく、
  必要なときに必要なものが動く。

Phase 4 (VISION):
  Phase 3.5 が完成形に達したとき、それが Phase 4 である。
```

### 実装の指針

| 妥協 | 本来 |
|:-----|:-----|
| `for sub in subscribers:` (直列) | Blackboard の状態変化が次の発火を**誘発** |
| `asyncio.gather(*tasks)` (並列) | Score Function が「今」の必要性を評価して動的に選択 |
| 固定順序 | **順序は状態に従属する** — 状態が変われば順序も変わる |
