# 状態型分類 (Paper X Type 1/2/3)

※これは `09_能動｜Ergon` プロジェクトの実装文書である。

## 1. なぜ状態型分類が必要か

旧文書では MCP ツールを $\mu / a / s / \eta$ の 4 状態空間で分類した。これは**存在論**としては正しいが、ハーネス強度を切り替えるには不十分である。

Paper X の核心は次である。

> **最適な忘却強度は状態の関数である。**

したがって v2 では、「ツールがどの状態空間に属するか」ではなく、**今どの状態型にいるか**で `boot/bye` の強度を変える。

## 2. Type 1 / 2 / 3 の定義

| Type | 状態 | 典型症状 | 最適ハーネス |
|:---|:---|:---|:---|
| **Type 1** | 有用な手がかりが最近あり、局所作業が進む | 軽微修正、既知手法、短い coding task | **軽量**。弱い忘却と軽量 boot |
| **Type 2** | デッドエンドや違反反復に入っている | 同じ失敗の反復、規約逸脱、視野狭窄 | **重量**。強い忘却と重量ハーネス |
| **Type 3** | 仮説は概ね正しいが行動がずれる | 方針は良いのに実装や sequencing が外す | **中 + 摂動**。中程度の制約 + 行動補正 |

## 3. 各 Type に対する処方

### Type 1

- 一言でいえば **軽量状態** である
- `boot`: 最小 kernel だけロード
- `bye`: 短い Belief Update で十分
- 主戦略: **Reduce**
- 主な risk: 不要な重装備で速度を殺すこと

### Type 2

- 一言でいえば **重量状態** である
- `boot`: 強い `C` をロードし、Hook も厚くする
- `bye`: 原因ではなく固定点を残す。再発防止条件を強く残す
- 主戦略: **Reduce + Offload**
- 主な risk: 全状態へ誤適用して prior を肥大化させること

### Type 3

- 一言でいえば **中 + 摂動状態** である
- `boot`: 制約は中程度、ただし `M` 軸を強め、必要な摂動だけを入れる
- `bye`: 何を考え違えたかより、**どの行為がずれたか**を残す
- 主戦略: **Isolate + Hook 補正**
- 主な risk: 説明を増やしても実行面が変わらないこと

## 4. 観測指標

| 指標 | Type 1 | Type 2 | Type 3 |
|:---|:---|:---|:---|
| 同一失敗の反復 | 低 | 高 | 中 |
| 手がかりの鮮度 | 高 | 低 | 中 |
| 方針の妥当性 | 高 | 低 | 高 |
| 実行のズレ | 低 | 中 | 高 |
| 推奨する Hook 強度 | 弱 | 強 | 中〜強 |

## 5. `boot/bye` への写像

| Type | boot profile | bye profile | 主な 3 原則 |
|:---|:---|:---|:---|
| Type 1 | light | short | Reduce |
| Type 2 | heavy | structured | Reduce + Offload |
| Type 3 | medium + mechanism-heavy + perturbative | action-corrective | Isolate + Reduce |

ここで重要なのは、`heavy` が常に良いわけではないことだ。Type 2 には効くが、Type 1 には過剰、Type 3 には的外れになりうる。

## 6. 旧 4 状態分類との関係

旧文書の $\mu / a / s / \eta$ 分類は廃棄しない。ただし役割を変える。

- **4 状態分類**: ツールや処理が「どこに属するか」を示す存在論
- **Type 1/2/3**: 今どの強度のハーネスをかけるべきかを示す制御論

旧分類は新分類の部分集合として残る。たとえば:

- `ask_with_tools` が $\mu \to a$ 境界で危険、という知見はそのまま有効
- ただし、その危険にどう対処するかは Type 1/2/3 に応じて変わる

## 7. 現行 HGK への診断

現行 HGK は実質的に **Type 2 default** である。

- 反復違反や視野狭窄には効く
- しかし軽作業や既知作業にも同じ重さをかける
- 結果として Type 1 の速度と Type 3 の行為補正が犠牲になる

v2 では、この一律運用をやめる必要がある。
