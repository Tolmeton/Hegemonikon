## 4. HGK 現行ハーネス批評

理論が正しいことと、現行ハーネスがその理論に適っていることは別問題である。ここでは現行 HGK を批評する。

### 4.1 Core prior ~45K tokens と E_H 166KB の平坦化

`01_markov_blanket.md` が指摘する通り、現行 HGK は **45K tokens 級 prior** を `boot` で毎回広く載せている。これは情報量の多寡の問題ではない。

さらに忘却論の観点では、ここに **Rules 166KB 級の E_H** が重なることで、`Core prior ~45K tokens` の問題が単独ではなく**符号化面の過大投資**として増幅される。`C_H` が針のように効くべきところで、`E_H` が毛布のように広がってしまう。

問題は次の 2 点である。

1. **均一性**
   - prior が広く薄く分布すると、Paper I の意味で忘却場 $\Phi_H$ が平坦化する
   - 平坦な場は力を生まない
2. **逆 U 字の上限張り付き**
   - `45K / 1M ≈ 4.5%` は、Theory 01 で仮置きした最適占有率 `2-5%` の上限に近い
   - 状態依存ロードがないまま全状態へ適用すると、Type 1 に対しては過重である

### 4.2 3 層防御の重量配分逆転

現行 HGK では、3 層防御の**説明**は厚いが、**環境強制**は相対的に薄い。

- `E` 軸:
  - Typos v8.4 構文、ギリシャ語術語、3 層防御表など、記述は豊富
- `C` 軸:
  - capability contraction や state-dependent load は十分に分離されていない
- `M` 軸:
  - Hook / validator / tool filter への移植がまだ局所的

これは「守るべきことが書かれている」のに、「どうやって必ず守らせるか」が相対的に軽い状態である。**E が厚く、C/M が薄い**。これをここでは「重量配分逆転」と呼ぶ。

### 4.3 Skills 84 と τ-Invariance による圧縮可能性

現行ハーネスの批評点は、skill 数が多いことそのものではなく、**Skills 84 規模でもなお τ-Invariance による圧縮余地が大きい**ことである。名前や局所的な文体が異なっていても、行為可能性を保存する核が同じなら、商を取って surface を減らせる。

この問題は単なる一覧の長さではない。

- `boot` で読み込まれる候補空間が広がる
- ルーティング時の選択複雑性が上がる
- 類似 skill 間の境界が曖昧だと、`C` 軸でなく `E` 軸の言い換えとして増殖する

つまり skill 増加は、Paper I 的には `|Hom(a,\eta)|` とルーティング複雑性の両方を増やす。**選択肢の豊富さがそのまま行為可能性の結晶化を弱める**。したがって v2 の課題は skill を感情的に削ることではなく、**τ-Invariance を使って保存すべき核と捨ててよい表面を分離すること**にある。

### 4.4 Hooks 7 登録を最少核として見る

忘却論に立てば、`M` 軸は「多ければ強い」ではない。むしろ **Hooks 7 登録程度を最少核として保ち、その 7 本に高密度の制約を載せる**方がよい。

- Hook は説明を増やす場所ではなく、`C` を環境へ埋め込む場所である
- 登録数を無制限に増やすと、今度は `M` 自体が肥大し、Type 1 を詰まらせる
- したがって最小 hook 集合を保ったまま、Rules 側の `C` 成分をそこへ移す方が設計として鋭い

ここでも問題は Hook が少ないことではない。**Hook が最少核である一方、Rules/E_H が過大であるため、3 層防御の重量が逆転している**ことである。

### 4.5 総合診断

現行 HGK は、**Type 2（デッドエンドや違反反復に効く重量ハーネス）を全状態へデフォルト適用している**。

これは Type 2 には効くが、

- Type 1 には過重
- Type 3 には説明過多で行為補正が弱い

という歪みを生む。
