# Ergon プロジェクト 設計文書 03

## LLM 確率的射 vs Hook 決定的射

本書は、旧来の `LLM / Deterministic Split` を保存しつつ、それを**確率的射 $\pi_s$ と決定的射 $\pi_a$ の分離**として再記述する。これは P4 の理論的基盤である。

## 1. 基本命題

- **LLM** = $\pi_s$ = $\mu$ 側で候補を生成する**確率的射**
- **Hook / typed tool / script** = $\pi_a$ = $a$ 側で環境を固定する**決定的射**

この分離の要点は、「考えること」と「環境を変えること」を別の射として扱う点にある。忘却論に基づけば、揺らぎを許してよいのは $\mu$ の内部までであり、境界を越えて環境へ出る瞬間には決定性が必要になる。

## 2. なぜ $\pi_s$ と $\pi_a$ を分けるのか

LLM による推論は、候補探索と仮説更新のために確率的であるほうがよい。だが、副作用が同じ確率層を通ってしまうと、揺れがそのまま環境へ漏れる。

ここで起きる破れは 3 つある。

1. **Decision leakage**
   - 仮説段階の揺らぎがそのまま副作用へ流れる
2. **Execution drift**
   - 同じ意図でも毎回異なる環境変化が出る
3. **Audit opacity**
   - なぜその操作が通ったのかを再現できない

P4 はこの破れを避けるために、**LLM は確率的射として閉じ、境界では Hook が決定的射として受け止める**という設計を要請する。

## 3. 設計原則

### Principle 1: 推論は $\pi_s$ に置く

$\pi_s$ は次を担う。

- 何をすべきかの仮説生成
- どの tool / route を使うかの候補選定
- 失敗理由の再解釈

ここは揺れてよい。むしろ探索性が必要である。

### Principle 2: 副作用は $\pi_a$ に落とす

環境に作用する操作は、再現可能で、型検査できて、失敗時に rollback 可能な射に落とす必要がある。

| 層 | 役割 | 例 |
|:---|:---|:---|
| **LLM 確率的射 $\pi_s$** | 候補生成・評価・説明 | prompt, reasoning, route selection |
| **Hook 決定的射 $\pi_a$** | 実行・拒否・矯正 | pre-commit hook, validator, tool filter |
| **typed tool** | $\pi_a$ を引き受ける能動面 | `hermeneus_run`, `sekisho_gate`, JSON schema |

### Principle 3: 記述より強制を優先する

Rules は場の説明であり、Hook は場そのものである。制約を文章で厚く書いても、それだけでは環境は曲がらない。**$\pi_a$ が実際に通す / 止める**ことで、はじめて制約は結晶化する。

## 4. マルコフ圏としての読み替え

旧文書の Copy / Discard の議論はそのまま保存できる。

- **$\pi_s$** は候補を増やし、情報を発散させる
- **$\pi_a$** は必要な変数だけを Copy し、不要な自由度を Discard する

したがって決定的層は単なる安全装置ではない。**忘却の方向を固定する幾何装置**である。

## 5. P4 への接続

P4 の中心問題は、「LLM の創発性を殺さずに、環境面では一貫性をどう確保するか」である。この問いに対する理論的答えが、

$$
\text{probabilistic reasoning} = \pi_s,\qquad
\text{deterministic enforcement} = \pi_a
$$

という二層分離である。

P4 は LLM をより賢くする設計ではない。**LLM が揺れても、環境側は揺れないようにハーネスを組む設計**である。

## 6. 設計上の帰結

- `LLM が書く` のではなく、`LLM が候補を出し、Hook が通すか拒否する`
- 不可逆操作ほど $\pi_a$ 側へ寄せる
- prose は `E` 軸、禁止条件は `C` 軸、Hook は `M` 軸として分離する

これが、ハーネス設計学における **LLM 確率的射 vs Hook 決定的射** の意味である。
