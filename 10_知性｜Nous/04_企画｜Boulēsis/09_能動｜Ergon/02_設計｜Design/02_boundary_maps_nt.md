# Ergon プロジェクト 設計文書 02

## C/E/M 境界の自然変換

本書は、Boundary Maps の旧定式化を保存したまま、それを**`C/E/M` 境界の自然変換**として読み替える。中心命題は次である。

> **境界とは、Constraint (`C`) の型整合性を保証する変換面である。**

Paper XI H3 に従えば、`C` と `E` は混ぜてはならない。禁止条件と表記形式を分離し、そのうえで両者を同時に保つ mechanism を設計する必要がある。

## 1. 基本定式化

Cog 内の異なる計画 $\text{Plan}_A$ と $\text{Plan}_B$ が論理的依存関係 $f: \text{Plan}_A \to \text{Plan}_B$ で結ばれているとする。これを `L (boot)` で Exec 層へ展開すると、

$$
\eta_{\text{interface}}^{C,E,M}: L(\text{Plan}_A) \Rightarrow L(\text{Plan}_B)
$$

という自然変換が必要になる。

このとき可換性は「値が届いた」ことでは足りない。**`C/E/M` が崩れずに届いた**ことが必要である。とりわけ境界の第一責務は、`C` の型が通るかどうかを保証する点にある。

## 2. `C/E/M` の分離

| 軸 | 何を固定するか | 誤って混ぜたときの症状 |
|:---|:---|:---|
| **C** (Constraint) | 禁止条件、権限、前提、safety class | 越権実行、不可逆逸脱 |
| **E** (Encoding) | schema、field、summary の器 | handoff 不整合、再読不能 |
| **M** (Mechanism) | Hook、validator、test、typed tool | 書いてあるだけで強制されない |

Paper XI H3 の教えは単純である。**制約を表記で代用してはならない。** `read_only` を prose で宣言しても、それだけでは `C` は守られない。`C` は `C` として、`E` は `E` として、別の面に立てなければならない。

## 3. 境界とは何を保証するのか

Boundary Map は Slice 間インターフェースの説明図ではない。むしろ次の 3 つを保証する**型整合面**である。

1. **Constraint 可換**
   - A が許した自由度以上の操作を B に与えない
2. **Encoding 可換**
   - A が出した器と B が読む器が一致する
3. **Mechanism 可換**
   - この一致を実際に enforce する装置が存在する

したがって、境界の意味は「何が通るか」よりも先に、**何が通ってはならないかを型として固定すること**にある。

## 4. Produces / Consumes / Contract

### 4.1 Produces

Slice A が何を生成するかを、`C/E/M` 付きで宣言する。とくに `C` は「どこまでの自由度を持つ出力か」を明示しなければならない。

### 4.2 Consumes

Slice B が何を受け取れるかを、A と同型の境界署名で宣言する。ここで `E` だけ合っていて `C` が違うなら、その接続は見かけ上つながっていても自然変換ではない。

### 4.3 Contract

A と B の間で何を機械的に検証するかを明記する。Boundary Map が契約になるのは `M` があるときだけであり、`M` がなければそれは願望にとどまる。

## 5. 実装層での適用

### Pre-flight Check

Slice B の実装開始前に、Slice A の `Produces` が観測可能であることを確認する。これは `E` の有無を見る検査ではなく、**`C` が正しい型で渡っているか**を見る検査である。

### Hook / Type / Test

- **Hook** は `C` を環境に刻む
- **Type** は `E` の器を固定する
- **Test** は `M` が実際に効いたことを検証する

この 3 つがそろって初めて、Boundary Map は自然変換の図式ではなく、**境界を機械的に守るハーネス**になる。

## 6. 設計上の帰結

- 境界の第一義は、`C` の型整合性保証である。
- `C/E` を prose で混ぜると、片方を直したときにもう片方が壊れる。
- Meso/Slice レベルで境界を引く理由は、Task では短すぎ、Session では粗すぎるからである。**境界契約は Meso で最も測りやすい。**
