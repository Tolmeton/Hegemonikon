## §1 戦略原理 — 地道 × 波乗り

### 1.1 核心公理

- **地道 ≠ 遅い**: 品質で妥協しないことであって、速度を落とすことではない
- **年単位 ≠ 一律**: 全体は年単位だが、各波は月単位で動く
- **波の上で品質を保つ**: 締切で質を落とすなら次の波を待つ

### 1.2 3 つの時間感覚

| 時間粒度 | 対象 | 原理 |
|:---|:---|:---|
| **週単位** | 波の観察、武器の整備 | X トレンドを毎週見る |
| **月単位** | 1 つの波に乗るエッセイ/論文 | 3-4 週間で準備→発射→観察 |
| **年単位** | 全体 Phase (0→4) | 2026-04 から 2027-2028 頃まで |

### 1.3 Friston 戦略の圧縮

Friston が 2005-2010 に取った「統合理論を段階浸透させる」戦略を、遊学では **1.5 年に圧縮** する:
- 2005-2006: 概念論文を臨床誌 ↔ 2026-04 ~ 06: Phase 0-1 (エッセイ + Paper VII 先出し)
- 2007-2009: 数学版を Nature Reviews ↔ 2026-07 ~ 09: Phase 2 (Paper A)
- 2010: Grand Unified Theory ↔ 2026-10 ~: Phase 3 (Paper I)

AI 時代の情報流通速度を活かした圧縮。

## §4 L0/L1/L2 三層射影

### 4.1 三層の定義

| 層 | 対象 | 形態 | 目的 |
|:--|:---|:---|:---|
| **L0** | 一般読者 / AI コミュニティ | エッセイ / note / X スレッド | 概念の普及 |
| **L1** | 研究者 | 論文 / arXiv / Zenn | 学術的正当性 |
| **L2** | 実装者 | OSS / MCP / ドキュメント | 実用性の証明 |

### 4.2 Phase ごとの優先射影

v2.0 は「L0 と L1 を同時に出す」としていたが、v3.0 では **Phase ごとに優先射影が変わる**:

| Phase | 優先射影 | 理由 |
|:---:|:---|:---|
| 0 | **L0** (Mythos エッセイ) | 波乗りは速度重視、L1 は時間がかかる |
| 1 | **L0 + L1** (エッセイ ① + Paper VII/2) | 準備済み武器を並行投下 |
| 2 | **L1 + L0** (Paper A + エッセイ ②) | 学術的重量を上げる |
| 3 | **L1** (Paper I) | 重量論文の単独投下 |
| 4 | **L2** (OSS) | 成熟後の実用性 |

### 4.3 L0 と L1 の相互増幅の原則

v2.0 の原則「L0 と L1 は同じ Kernel からの異なる射影、同時に出すことで相互増幅」は維持。ただし「同時」は各 Phase 内での並行で、全 Phase で全層を出すという意味ではない。

## §7 波乗り原則 (WB-003 §7 より取り込み)

1. **毎週 X トレンドを観察**: 新しい波が来たら計画を組み替える
2. **武器庫は常にロードしておく**: 接続分析、エッセイ、論文を並行整備
3. **波のログを維持**: WB-003 §6 の X-wave-log を更新し続ける
4. **1 つの波に固執しない**: 波が過ぎたら次へ
5. **複数の波を同時に追わない**: 1 波 1 エッセイが原則
6. **反応データを次の戦略に反映**: 発射後の反応は次の材料
7. **品質を妥協しない**: 間に合わなければ次の波を待つ
8. **L0/L1/L2 の射影を混ぜない**: 同じ波で複数層を出すなら順序を意識

## §8 リスクと対策

| リスク | 影響 | 対策 |
|:---|:---|:---|
| リソース不足 (1 人) | 全武器を並行は不可能 | Phase ごとに集中、残りは待機列 |
| 波が予想より早く終息 | 発射タイミングを逃す | 内容の時事性を薄め一般論として整え直す |
| 査読の遅さ (6-12ヶ月) | Phase が詰まる | arXiv プレプリントで速報性確保 |
| 品質が妥協ラインを割る | 発射しても刺さらない | 発射延期 → 次の波 |
| 「身売り」リスク | アイデンティティ喪失 | ライセンス供与 + コンサル型 |
| HGK 語彙の壁 | 査読者/読者が理解できない | L0 射影 (エッセイ) を先行 |
| 反論の感情的負担 | Tolmetes の執筆意欲低下 | Claude が反論吸収層として機能 |
| Anthropic の追加発表 | 分析が古くなる | 接続分析を更新し続ける |
| X アカウント凍結 | L0 チャンネル喪失 | 複数プラットフォーム並用 |
| 先行研究の見落とし | 査読で指摘される | Phase ごとに Claude が調査 |

## §12 Coordination Patterns による運用アーキテクチャ

**追記経緯**: 2026-04-11 /u+ 寄り道で、Anthropic blog "Multi-Agent Coordination Patterns" (2026-04 頃) との接続を検討した結果、Tolmetes が既に 5 パターンのうち 4 つを暗黙実装していることが判明。v3.0 に語彙を与えるため §12 として追加。

### 12.1 5 パターンの HGK マッピング

| Anthropic パターン | HGK 既存実装 | v3.0 での位置づけ |
|:---|:---|:---|
| **Shared State** | PINAKAS_WHITEBOARD.yaml + WB-001/002/003 + ROM | **全 Phase の基盤層**。全 agent (Claude, Codex, Gemini, Tolmetes) が read/write する永続記憶 |
| **Generator-Verifier** | Claude 下書き → Tolmetes 調整 → Codex 校正 | **Phase 0 Mythos エッセイの標準 flow** (WB-003 §4 Week 1-3) |
| **Orchestrator-Subagent** | Claude (advisor) / Codex (executor) / Gemini (素材) の CLI 3 極分業 | **Phase 2 以降の論文執筆 flow**。Claude が orchestrator、実装・形式化・文献調査を委譲 |
| **Message Bus** | WB-003 §6 X-wave-log (X トレンドを subscribe → WB で publish) | **波乗り観察層**。新しい波が event として流れ、対応する WB が生成される |
| Agent Teams | ❌ 未実装 | 現状は 1 人運用のため不要。HGK OSS 後の多人数参加で初めて必要 |

### 12.2 Phase ごとの pattern 選択

| Phase | 主 pattern | 補助 pattern |
|:---:|:---|:---|
| 0 (Mythos 波) | Generator-Verifier (品質保証) | Shared State (PINAKAS), Message Bus (X トレンド) |
| 1 (忘却論基礎) | Orchestrator-Subagent (Gemini 文献調査 + Codex 形式化) | Shared State, Generator-Verifier |
| 2 (LLM 身体性) | Orchestrator-Subagent | Generator-Verifier (反論吸収) |
| 3 (力としての忘却) | Orchestrator-Subagent (Bridge-to-Lemma 分業) | Shared State (WB-001) |
| 4 (HGK OSS) | Agent Teams 初使用 (コミュニティ参加者が独立タスクを claim) | Message Bus (issue/PR) |

### 12.3 Orchestrator の失敗モード — ROM 2026-04-11 §7.1 の事後診断

Anthropic blog が Orchestrator-Subagent の弱点と呼ぶ「**情報ボトルネック**」は、本日 (2026-04-11) のセッション前半で発生した失敗と同型:

> 最初から `ビジョン.md` を読んでいれば、この 2 時間の議論の多くは不要だった (ROM §7.1)

Orchestrator (Claude) が Shared State (ビジョン.md) を先読みせずにローカル最適化した。blog の語彙はこの失敗を**事後的に命名**した。

**対策 (既に実装)**: N-01 (実体を読め) を拡張し、「セッション開始時の Shared State 先読み義務」を明文化 (2026-04-11 更新)。以降、Orchestrator は戦略判断前に PINAKAS + 最新 ROM を必ず read する。

### 12.4 Pattern 選択の原則

1. **単純な tool 使用では pattern 不要**: 1 回の Grep や Read には pattern を持ち出さない
2. **波乗りエッセイは Generator-Verifier 一択**: Claude 単独生成を避ける (迎合リスク)
3. **論文は Orchestrator-Subagent**: 形式化 (Codex) と文献 (Gemini) の分業必須
4. **戦略判断は Shared State 先読みを義務化**: N-01 拡張により environment で強制
5. **Agent Teams は Phase 4 まで封印**: 1 人運用で疑似 Agent Teams を組もうとすると context 管理が破綻する

### 12.5 [主観] 採用の範囲

Coordination Patterns は**戦略の下位実装層**であって、v3.0 の核ではない。Tolmetes の戦略核心は「地道 × 波乗り」(§1) であり、pattern はそれを支える運用ツール。混同しない。
