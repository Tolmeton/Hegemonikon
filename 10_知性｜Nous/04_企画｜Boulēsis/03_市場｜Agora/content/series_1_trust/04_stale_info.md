# 【最終回】AIが「Heroku無料枠」を勧めてくる恐怖：情報の「鮮度」を見抜くプロトコル

「ポートフォリオサイトを作りたいんですが、おすすめのホスティングサービスはありますか？」

もしあなたが今、ChatGPTやClaudeにこう尋ねたとします。AIは自信満々にこう答えるかもしれません。
**「Herokuがおすすめです。無料プラン（Free Tier）があり、手軽にデプロイできます」**

そしてあなたはHerokuに登録し、デプロイの最終段階で「クレジットカード情報の入力」を求められ、月額料金が発生することを知って愕然とするのです。

ご存知の通り、**Herokuの無料プランは2022年11月に廃止されました。**

しかし、AIにとってこの事実は「つい最近のこと」あるいは「知らないこと」です。AIの学習データには**賞味期限**があります。どれほど流暢な日本語で回答されたとしても、その中身が「2021年の常識」で止まっているとしたら、技術選定においては致命的です。

シリーズ「AI信頼構築」の最終回となる第4回は、AIが抱える**「情報の鮮度（Freshness）」**の問題と、それを見抜くための技術的アプローチについて解説します。

## なぜAIは「古い情報」を捨てられないのか

AIが古い情報を自信満々に語るのには、主に2つの原因があります。

### 1. 訓練データのカットオフ（Knowledge Cutoff）
すべてのLLM（大規模言語モデル）には「知識のカットオフ日」が存在します。例えば、2023年4月までのデータで学習されたモデルは、それ以降に起きたライブラリの破壊的変更やサービスの終了を知りません。

### 2. 頻度バイアス（Frequency Bias）
これがより厄介な問題です。たとえ最新のWeb検索機能を持っていたとしても、AIは**「ネット上に大量に存在する情報」を優先する傾向**があります。

*   **Herokuは無料だ** と書かれた記事：2010年〜2022年の間に数百万件蓄積
*   **Herokuは有料化した** と書かれた記事：2022年以降の数万件

AIの学習ウェイトにおいて、圧倒的な「過去の多数派」が「現在の少数派（真実）」を押しつぶしてしまうのです。これを防ぐには、人間側が明示的に介入する必要があります。

## 技術の墓場：よくある「ゾンビ情報」リスト

開発現場でAIを利用する際、特に以下のトピックについては「AIは嘘をつく（古い情報を言う）」という前提で接する必要があります。

| カテゴリ | 具体的な「ゾンビ情報」の例 | 現実（2024年時点） |
| :--- | :--- | :--- |
| **インフラ** | Heroku Free Tier | 2022年11月廃止 |
| **AIモデル** | OpenAI `text-davinci-003` | 2024年1月終了 (API廃止) |
| **Python** | Python 3.7 / 3.8 | EOL（セキュリティサポート終了） |
| **JSライブラリ** | `moment.js` の新規採用 | 非推奨（Maintenance Mode） |
| **CSS** | ベンダープレフィックス (`-webkit-`) | 現代のブラウザではほぼ不要 |

これらをAIが推奨してきた場合、そのAIは「過去の記憶」で喋っています。

## Protocol D：外部サービス検証（Verification）

では、どうすればこの「ゾンビ情報」を回避できるでしょうか。ここで導入するのが、本シリーズで提唱する**Protocol D（Dependency Check）**です。

技術的な推奨事項（ライブラリ、SaaS、API）が含まれる回答に対し、以下の検証プロセスを挟みます。

### アプローチ1：検索クエリの強制付与

AIにいきなり答えさせるのではなく、**必ず「終了」「非推奨」というキーワードを含めて検索させる**手法です。

RAG（検索拡張生成）や、ChatGPTのWebブラウジング機能を使う場合、以下のようなプロンプト設計が有効です。

```markdown
# Bad Prompt
Herokuの無料枠の使い方を教えて。

# Good Prompt (Protocol D)
Herokuの利用を検討しています。
回答する前に、必ず以下のクエリでWeb検索を実行し、その結果を元に回答してください。
検索クエリ: "Heroku pricing free tier shutdown 2024"
```

「使い方を教えて」と聞くと、AIは学習済みの（古い）知識を検索せずに吐き出しがちです。「現状を確認せよ」という指示（Directive）を与えることで、推論の方向性を矯正します。

### アプローチ2：Freshness Directive（鮮度指示）

システムプロンプトやカスタム指示（Custom Instructions）に、以下のルールを組み込むことで、情報の鮮度に対する感度を高めることができます。

> **System Prompt Example:**
> あなたはシニアエンジニアです。技術的なアドバイスを行う際は、その情報が「現在も有効か」を常に疑ってください。
> 特に、ライブラリのバージョン、SaaSの料金プラン、APIの仕様については、あなたの学習データが古い可能性があることを認識し、回答の冒頭に「※202X年X月時点の情報である可能性があります」と注釈を入れてください。

## 出力の「鮮度」を見抜く3つの質問

ツールを使わずに、チャットのやり取りだけで相手（AI）が古い情報を喋っているか見抜くには、以下の3つの質問を「壁打ち」として投げかけてください。

1.  **「その情報は、いつ時点のものですか？」**
    *   AIに自分の知識のカットオフを自覚させます。
2.  **「公式ドキュメントの『Migration Guide』や『Deprecation』のページを確認しましたか？」**
    *   ハルシネーション（もっともらしい嘘）を抑制し、自信がない場合は「確認できません」と言わせる効果があります。
3.  **「現在もそのAPI/プランは新規契約可能ですか？」**
    *   「存在する」ことと「新規に使える」ことは別です。レガシーな機能を紹介されていないか確認します。

## 自動化の実装例：Pythonによる鮮度チェック

もしあなたがAIエージェントを開発しているなら、出力されたキーワードに対して自動的に鮮度チェックを行う関数を挟むべきです。

以下は、推奨された技術スタックが「生存しているか」を簡易チェックする擬似コードです。

```python
def check_tech_freshness(tech_name):
    """
    推奨された技術が「終了」していないか検索して検証する
    """
    search_query = f"{tech_name} deprecated shutdown end-of-life 2024"
    results = search_web(search_query) # 外部検索API
    
    risk_keywords = ["sunset", "discontinued", "archive", "migration guide", "eol"]
    
    for result in results:
        if any(keyword in result.text.lower() for keyword in risk_keywords):
            return f"⚠️ 警告: {tech_name} は終了または非推奨の可能性があります。最新情報を確認してください。"
            
    return "✅ 鮮度チェック通過（Web検索上での明らかな終了告知なし）"

# AIの回答から技術名を抽出してチェック
# >>> check_tech_freshness("text-davinci-003")
# Output: "⚠️ 警告: text-davinci-003 は終了または非推奨の可能性があります..."
```

## シリーズまとめ：AIと信頼関係を築くために

全4回にわたり「AI信頼構築」について解説してきました。最後に、私たちがAIと付き合う上で持つべき「3つの検証軸」を振り返ります。

1.  **確信度の検証**（第1回・第2回）
    *   AIの「自信満々」は信じない。思考の過程（Chain of Thought）を出力させ、論理の飛躍がないか確認する。
2.  **情報源の汚染度**（第3回）
    *   AIに入力するコンテキストにノイズが混じっていないか。RAGにおける検索精度の重要性。
3.  **情報の鮮度**（今回）
    *   訓練データは過去の遺産である。「今」通用するかは、必ず外部情報（Web検索・ドキュメント）と突き合わせる。

AIは魔法の箱ではなく、**「過去のデータの集合体」**です。その特性を理解し、適切なプロトコル（手順）で接することで、AIは初めて「信頼できるパートナー」になります。
