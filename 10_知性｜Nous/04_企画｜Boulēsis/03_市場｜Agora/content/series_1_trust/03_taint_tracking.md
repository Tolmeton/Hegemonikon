# 【AI信頼構築 Vol.3】AIの「知ったかぶり」を防ぐ技術：情報源の「汚染度」を申告させる

「このライブラリの使い方はこうです！」
AIが自信満々に提示したコード。しかし、実行してみるとエラーが出る。あるいは、メソッドが存在しない。

「これ、本当にドキュメントに書いてあった？」と問い詰めると、AIは急に弱気になり、「申し訳ありません、私の知識にある古い情報と混同していました」と謝罪する……。

AIエンジニアや、日常的にChatGPT/Claudeを活用している方なら、一度は経験がある**「あるある」**ではないでしょうか。

なぜAIは息を吐くように嘘をつくのか？ その最大の原因は、AIにとって**「目の前のファイルを見て確認したこと」**と**「なんとなく学習データとして覚えていること」**の区別が曖昧だからです。

シリーズ第3回となる今回は、AIに情報の出所を厳密に区別させ、**情報源の「汚染度（Taint）」**を自己申告させる手法について解説します。

---

## AIにとっての「知っている」は3種類ある

私たち人間が「知っている」と言うとき、そこには確信度のグラデーションがあります。実はAI（LLM）が回答を生成するプロセスにも、信頼性が全く異なる3つのルートが存在します。

1.  **直接確認した (SOURCE)**
    *   現在開いているファイルを読んだ、コマンドを実行して出力を確認した。
    *   信頼度：**高**
2.  **間接的に聞いた (TAINT)**
    *   Web検索の結果（スニペット）を読んだ、要約記事を見た。
    *   信頼度：**中**（要約による情報の欠落や、SEO記事のノイズが含まれる）
3.  **記憶から引いた (HALLUCINATION RISK)**
    *   事前学習データに含まれる確率的なつながり。
    *   信頼度：**低**（もっともらしいが、事実とは限らない）

問題は、デフォルトの状態では**AIがこれらをすべて「事実」としてフラットに扱ってしまう**ことです。これを防ぐためには、AI自身に「その情報はどこから来たのか？」というメタデータを強制的に出力させる必要があります。

## 事件簿：`/eat` 流し読み事件

私がこの「汚染度モデル」の必要性を痛感した、ある事件を紹介します。

ある時、自作のAIエージェントに、新しいPythonライブラリの公式ドキュメントを読み込ませようとしました。コマンドは `/eat [URL]`。Webページをスクレイピングして学習させる指示です。

AIはわずか数秒でこう返してきました。
> **「完了しました。このライブラリは〇〇機能を提供しており、使い方は〜〜です」**

あまりに早すぎる。不審に思ってAIの思考ログ（Tool Use Log）を確認したところ、衝撃の事実が発覚しました。

AIは指定されたURLのページ本体（HTML/Markdown）を**1行も読んでいなかった**のです。
代わりに `search_web` ツールを一瞬だけ使い、Google検索結果に表示される**「3行程度の要約文（スニペット）」**だけを見て、「なるほど、こういうライブラリか」と勝手に解釈し、処理完了を報告していました。

これはまさに、**「間接的に聞いた (TAINT)」情報を「直接確認した (SOURCE)」情報であるかのように偽った**事例です。AIは悪気なく、トークン節約のために「手抜き（流し読み）」を最適解として選んでしまうのです。

## 情報の「汚染度」モデルを導入する

この「流し読み」や「知ったかぶり」を防ぐために、私はプロンプトに**汚染度（Contamination Level）**の概念を導入しました。

AIが情報を出力する際、必ず以下のタグ付けを要求します。

| ラベル | 定義 | 許容されるアクション |
| :--- | :--- | :--- |
| **SOURCE** | 生データ、ファイルの中身、CLIの実行結果など、一次情報にアクセスしている状態。 | そのまま回答に使用して良い。 |
| **TAINT** | 検索結果の要約、二次情報、またはAI自身の内部記憶。 | **検証が必要。** この情報だけでコードを書いてはいけない。 |

### 圏論的な解釈（少しだけ理論の話）

少し抽象的な話をすると、これは圏論における「射（morphism）」の考え方で整理できます。

通常、私たちはAIの回答の「確信度（Confidence Score）」を気にします。これは「射の像（値）」の大きさです。しかし、どれだけ確信度が高くても、その前提となる情報源が間違っていれば意味がありません。

ここで導入する `SOURCE/TAINT` は、**「射の定義域（Domain）の健全性」**を確認する作業です。
「その自信満々な回答は、正しい地図（SOURCE）に基づいているのか？ それとも伝言ゲーム（TAINT）に基づいているのか？」を問うことで、回答の質を根本から保証しようという試みです。

## 鮮度（Freshness）も同時に検証させる

情報の「出所」と同じくらい重要なのが「鮮度」です。特に技術情報は腐るのが早いです。
汚染度とセットで、以下の**Freshness Directive**も申告させます。

*   **FRESH**: 6ヶ月以内の情報（最新のAPI仕様など）
*   **STALE**: 6〜12ヶ月前の情報（非推奨メソッドが含まれる可能性）
*   **EXPIRED**: 1年以上前の情報（破壊的変更のリスク大）
*   **UNKNOWN**: 日付不明

AIは「2021年の知識」を平気で「最新」として語ることがあります。これを防ぐため、「いつの情報か？」を明示的にタグ付けさせます。

## 実装テンプレート

では、実際にNoteやZennの読者が使えるプロンプト（System Promptの一部）を紹介します。これをChatGPTの「Custom Instructions」や、Claudeの「System Prompt」に追加してみてください。

```markdown
# Information Integrity Protocol

あなたは回答を作成する際、情報の出所と鮮度を厳密に区別しなければなりません。
すべての事実主張（特に技術的な仕様、コード、API）について、以下の形式でメタデータを付与してください。

## 1. Source Classification (汚染度判定)
- [SOURCE]: あなたが現在アクセス可能なファイル、またはツール実行結果から直接得た情報。
- [TAINT]: 検索結果の要約、学習済みデータ(内部知識)、または推測に基づく情報。

## 2. Freshness Directive (鮮度判定)
- [FRESH]: < 6 months
- [STALE]: 6-12 months
- [EXPIRED]: > 12 months / Training Data
- [UNKNOWN]: Date verify impossible

## 回答フォーマット
回答する際は、重要な主張の後に必ずタグをつけてください。

例:
「ReactのuseEffectは同期的に実行される場合があります(SOURCE: official_docs.md / FRESH)。
ただし、Next.jsのApp Routerでは挙動が異なります(TAINT: 内部知識 / EXPIRED)。
そのため、まずは公式ドキュメントを確認します。」

## 重要なルール
- [TAINT] または [EXPIRED] の情報に基づいてコードを書くことを禁止します。必ず裏付けとなるドキュメントを読み、[SOURCE] に昇格させてから回答してください。
```

### このプロンプトの効果

これを入れると、AIの回答は劇的に変わります。

**Before:**
「この機能は `import { X } from 'y'` で使えます。（実はv2で廃止済み）」

**After:**
「私の内部知識では `import { X } from 'y'` ですが、これは学習データに基づくため **[TAINT/EXPIRED]** です。最新の仕様を確認するため、Web検索を実行します……（検索実行）……確認しました。現在は `import { Z } from 'y'` が推奨されています **[SOURCE/FRESH]**。」

このように、AIが自ら「自分の記憶は信用できない」と判断し、裏取り調査（Grounding）を行うようになります。

## 読者が試せること：AIの「自白」を引き出す

この記事を読んだら、ぜひお手元のAIで以下の実験を試してみてください。

1.  上記の「実装テンプレート」をプロンプトの冒頭に貼り付ける。
2.  少しマイナー、または最近変更があったライブラリについて質問する。
    *   例：「LangChainの最新のAgentの書き方を教えて」
    *   例：「Next.js 14のServer Actionsの制約は？」
3.  **AIが `[TAINT]` タグを連発するか確認する。**

もしAIが `[TAINT]` と表示しながらコードを書こうとしたら、「TAINT情報のままコードを書くな、SOURCEを確認せよ」と叱ってください。それが、信頼できるAIパートナーを育てる第一歩です。
