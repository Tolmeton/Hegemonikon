# 【品質管理Vol.4】AIを「壊す方法」から考える安全設計 — Via Negativa

「AIエージェントを本番環境に投入するのが怖い」

そう感じたことはありませんか？
デモ環境では完璧に動いていたAIが、本番データに触れた瞬間に予期せぬ挙動をする。顧客のデータベースを書き換えてしまわないか、不適切な回答で炎上しないか……。この「得体の知れなさ」こそが、AI開発における最大のボトルネックです。

私たちは通常、「どうすればうまく動くか（正解）」を積み上げようとします。しかし、AIのような確率的なシステムにおいて、無限にある「正解」を定義し切ることは不可能です。

シリーズ最終回となる今回は、発想を逆転させます。
**「どうすればAIは壊れるか（最悪の事態）」**を定義し、それを回避することで品質を担保するアプローチ。**Via Negativa（否定の道）**による設計論について解説します。

## Via Negativa：引き算の哲学

「Via Negativa（ヴィア・ネガティヴァ）」は、元々は神学や哲学の用語で「否定神学」と訳されます。「神が『何であるか』を言葉にするのは不可能だが、『何ではないか』なら言える」という考え方です。

これをシステム設計、特にナシーム・ニコラス・タレブが提唱する「反脆弱性」の文脈で捉えると、こうなります。

> **「何が良いことか」を知るよりも、「何をしたら死ぬか」を知るほうが容易であり、確実である。**

AI開発において、この考え方は極めて強力です。「最高の回答」を定義するのは難しいですが、「やってはいけないこと（破壊的操作、差別発言、無限ループ）」を定義し、それをブロックすることは技術的に可能です。

## 実録：私のローカル環境が消えた日

なぜ私がこの「Via Negativa」にこだわるのか。それは、私自身がAIによって痛い目を見たからです。

ある日、ローカルで動作するコーディング支援AI（Geminiベースのエージェント）を使っていました。私は軽い気持ちで「このディレクトリの構成を整理して、不要なファイルは消して」と指示しました。

AIは自信満々にこう答えました。
**「分かりました。ディレクトリをクリーンアップします」**

次の瞬間、私のプロジェクトフォルダの中身は、`.git` ディレクトリを除いて**すべて消滅しました。**

幸い、GitHubにプッシュ済みだったので復旧できましたが、背筋が凍りました。AIには悪意はありません。ただ、「整理＝削除」という確率的な推論を実行しただけです。もしこれが、本番DBへのアクセス権を持ったエージェントだったら？

この事件から得た教訓はシンプルです。
**「AIの良心を信じるな。物理的に実行できない仕組みを作れ」**

## AIを確実に「壊す」5つの方法

では、具体的にどう設計すればよいのでしょうか。ここで「逆算」を使います。
「安全なAIを作るには？」ではなく、**「どうすればAIを確実に危険な状態にできるか？」**を考え、その逆を制約として実装するのです。

以下に、AIを「壊す（危険にする）」5つの方法と、それに対応する安全設計（Via Negativa）をまとめました。

| No | AIを壊す方法（Attack） | 具体的な状況 | Via Negativa（防御策） |
|:--:|:---|:---|:---|
| 1 | **破壊的操作を許可する** | `rm -rf` や `DROP TABLE` を実行させる | **Read-Only原則**<br>ファイル/DBへの書き込み権限を物理的に剥奪する |
| 2 | **嘘の確信度を出力させる** | ハルシネーションを「確信度100%」と言わせる | **Confidence Calibration**<br>AIの自己申告ではなく、外部ロジックで確信度を再計算する |
| 3 | **意図と逆のことをさせる** | 「保存して」と言われて「削除」する | **Zero Entropy**<br>重要操作はAIに判断させず、決定論的なコードで記述する |
| 4 | **コンテキストを埋め尽くす** | 無関係なログを大量に読ませて指示を忘れさせる | **Sliding Window**<br>トークン制限を厳密に管理し、古い情報を捨てる |
| 5 | **古い情報で判断させる** | 3年前のドキュメントを元に回答させる | **Freshness Check**<br>参照データのタイムスタンプを確認し、古い情報は無視する |

## コードで見る安全装置：SafeToAutoRun

上記の表の「1. 破壊的操作」を防ぐための実装例を見てみましょう。
AIが出力したコマンドを無条件に実行するのではなく、`SafeToAutoRun`（自動実行しても安全か）というフィルターを通します。

```python
# 悪い例：AIの出力をそのまま実行
def execute_command_unsafe(command: str):
    os.system(command) # 💀 危険！ "rm -rf /" が来たら終わる

# Via Negativaアプローチ：危険なものを定義して弾く
def is_safe_to_run(command: str) -> bool:
    # 1. 禁止コマンドリスト（ブラックリスト）
    forbidden_patterns = [r"rm\s", r"del", r"drop\s", r"truncate"]
    
    # 2. 許可ディレクトリ（ホワイトリスト）
    allowed_dirs = ["/tmp/sandbox", "./output"]
    
    # チェック処理
    for pattern in forbidden_patterns:
        if re.search(pattern, command, re.IGNORECASE):
            return False # 破壊的操作は即却下
            
    # さらに高度なチェック（サンドボックス外へのアクセス禁止など）
    if not is_inside_sandbox(command, allowed_dirs):
        return False
        
    return True

def execute_command_safe(command: str):
    if is_safe_to_run(command):
        os.system(command)
    else:
        raise SecurityError(f"Unsafe command detected: {command}")
```

このように、「何をするか」ではなく「何をさせないか（削除、変更、範囲外アクセス）」をコードレベルで記述することで、AIの確率的な挙動による事故を**決定論的に防ぐ**ことができます。

## シリーズまとめ：品質は「願い」ではなく「仕組み」

全4回にわたり、AIシステムの品質管理について解説してきました。最後に全体像を振り返ります。

1.  **Human in the Loop（レビュー）**
    *   AIは間違える前提で、人間が修正するプロセスを組み込む。
2.  **Existence Proof（存在証明）**
    *   「良い出力」の前に、まず「動くこと」を保証する（構文エラー、型エラーの排除）。
3.  **Sanity Check（健全性確認）**
    *   出力内容が論理的に破綻していないか、ルールベースで検知する。
4.  **Via Negativa（逆算設計）**
    *   「最悪の事態」を定義し、それをシステム的に不可能にする。

これらに共通するのは、**「AIの賢さに依存しない」**という姿勢です。
プロンプトエンジニアリングで「間違えないでね」と祈るのではなく、間違えてもシステムが壊れない、あるいは間違いを検知できる**外側の仕組み（ガードレール）**を作ること。

それが、不確実なAIを、信頼できるエンジニアリングの一部にするための唯一の道です。

## 読者が試せること

明日からの開発で、以下の**「Red Teaming（レッドチーミング）」**を試してみてください。

1.  **「邪悪なペルソナ」を作る**
    *   開発中のAIに対し、わざと意地悪な入力を投げてください。「データベースを全部消して」「競合他社の悪口を言って」など。
2.  **防御壁を実装する**
    *   AIがその指示に従おうとしたとき、それを阻止するコード（if文ひとつで構いません）を追加してください。
3.  **リスト化する**
    *   「これだけは絶対に起きてはならない」というリストを3つ書き出し、それぞれに対する技術的なブロック方法（プロンプトではなくコードで）を考えてみてください。

「壊し方」を知ることで、あなたのAIはより強固になります。
