# 【品質管理 vol.2】コードの「存在証明」を自動化する — なぜそのファイルはそこにあるのか？

「この `utils.py` って、一体何のためにあるんでしたっけ？」
「`temp_data_processor_v2.py` は消していいファイルですか？」

プロジェクトが長生きすればするほど、こうした会話は増えていきます。
開発当初は明確だったはずの意図が、担当者の退職や記憶の風化とともに失われ、コードベースは徐々に「謎のファイル」で埋め尽くされていく。

誰も触りたがらない、でも消すと何かが壊れそうで怖い。
そんな**「ゾンビコード」**に囲まれて開発するのは、地雷原を歩くようなストレスがありますよね。

今回は、そんな恐怖から解放されるための仕組み、**「Dendron（デンドロン）」**を紹介します。これは、すべてのコードに「生きる理由」を問いかけ、自動的に検証するシステムです。

## コードは黙って腐る

なぜコードはゾンビ化するのでしょうか。
それは、ファイルそのものが**「なぜ私はここにいるのか」**を語らないからです。

通常、ドキュメント（Docstring）には「この関数は何をするか（How）」が書かれます。
*   `引数 A を受け取り、B を返します`
*   `データベースからユーザー情報を取得します`

しかし、本当に知りたいのは「なぜそれが必要か（Why）」です。
*   `レガシーシステムとの互換性維持のために存在する`
*   `月次バッチ処理でのみ使用される一時的な変換ロジックである`

この「Why」が抜けたまま放置されたファイルは、時間の経過とともに腐敗します。これを防ぐために導入したのが、**Dendron**というシステムです。

## Dendron システムの全貌

Dendron（樹状突起、あるいは樹木構造を意味する言葉から命名）は、非常にシンプルなルールで構成されています。

**「すべてのファイルとディレクトリは、自身の存在理由を証明しなければならない」**

これを精神論ではなく、CI（継続的インテグレーション）で強制的にチェックする仕組みに落とし込みました。

### 1. ファイルレベルの証明：PURPOSE コメント

すべての Python ファイル（`.py`）の冒頭に、必ず `PURPOSE` から始まるコメント行を記述することを義務付けます。

**悪い例（存在理由が不明）**
```python
# utils.py
import datetime

def get_date():
    return datetime.datetime.now()
```

**良い例（Dendron 適用後）**
```python
# PURPOSE: レガシーAPIへのリクエスト時に必要な、特殊な日付フォーマット変換を提供する
import datetime

def get_formatted_date():
    # ...
```

このたった1行があるだけで、コードレビュー時の「これ何？」が激減します。「日付変換ライブラリ」ではなく、「レガシーAPIのために存在している」ことが明確だからです。

### 2. ディレクトリレベルの証明：PROOF.md

ファイルだけでなく、ディレクトリ（フォルダ）も無秩序に増えがちです。
Dendron では、各ディレクトリに `PROOF.md` というファイルを置くことを推奨しています。ここには、そのディレクトリが存在する「法的根拠」を記します。

**ディレクトリ構成例**
```text
project_root/
  ├── services/
  │   ├── payment/
  │   │   ├── PROOF.md  <-- このディレクトリの責務定義
  │   │   ├── stripe.py
  │   │   └── paypal.py
```

**PROOF.md の内容**
```markdown
# 存在証明: Payment Service

このディレクトリは、外部決済プロバイダー（Stripe, PayPal）との通信ロジックを集約するために存在する。
ドメインロジック（割引計算など）はここには含めず、純粋なAPIラッパーとして機能させること。
```

これにより、新しく入ったメンバーが「どこにファイルを追加すべきか」迷うことがなくなります。

### 3. 自動化された検問：Dendron Checker

ルールを作っても、守られなければ意味がありません。私たちは `dendron_checker.py` というスクリプトを作成し、CI パイプラインに組み込みました。

このチェッカーは以下の動作をします。

1.  プロジェクト内の全 `.py` ファイルを走査する。
2.  各ファイルの先頭付近に `# PURPOSE:` という文字列があるか確認する。
3.  もし無ければ、**ビルドを失敗させる（エラーを吐く）。**

**dendron_checker.py のイメージ**

```python
import os
import sys

def check_files(root_dir):
    missing_purpose = []
    
    for root, _, files in os.walk(root_dir):
        for file in files:
            if file.endswith(".py"):
                path = os.path.join(root, file)
                with open(path, "r", encoding="utf-8") as f:
                    content = f.read()
                    # 簡易的なチェックロジック
                    if "# PURPOSE:" not in content:
                        missing_purpose.append(path)
    
    if missing_purpose:
        print("以下のファイルに 'PURPOSE' コメントがありません:")
        for p in missing_purpose:
            print(f"  - {p}")
        sys.exit(1) # エラー終了
    else:
        print("全てのファイルの存在証明が確認されました。")

if __name__ == "__main__":
    check_files("./src")
```

このスクリプトが走ることで、「とりあえずファイルを作って、後でコメント書こう」という甘えが許されなくなります。**説明できないコードは、コミットすらできないのです。**

## 実測：143,000行のコードベースでの効果

私が関わったあるプロジェクトでは、約143,000行（Pythonファイル数 約800）の規模でこのシステムを導入しました。

導入当初は既存ファイルへの追記作業が大変でしたが、一度 **PURPOSE 100%** を達成してからは、劇的な変化がありました。

| 項目 | 導入前 | 導入後 |
| :--- | :--- | :--- |
| **ファイル追加の心理的ハードル** | 低い（とりあえず作る） | 適度（理由を考える必要がある） |
| **リファクタリング** | 怖い（影響範囲不明） | **容易**（目的が明確なため代替手段を設計しやすい） |
| **オンボーディング** | 口頭伝承が必要 | コードを読めば「目的」がわかる |
| **不要ファイルの削除** | 放置され続ける | 定期的に発見・削除される |

特に「不要ファイルの削除」が進んだのが印象的でした。「PURPOSE: キャンペーンAのための一時的な集計」と書かれたファイルが、キャンペーン終了後も残っていることに（チェッカーのおかげで）すぐ気づけるようになったからです。

## なぜ「ドキュメント」ではなく「コード内コメント」なのか

Wiki や Notion に設計図を書くことも重要ですが、コードとドキュメントは距離が離れれば離れるほど同期が取れなくなります。

*   **Wiki:** わざわざ見に行かないとわからない。更新を忘れがち。
*   **PURPOSE:** ファイルを開いた瞬間に目に入る。コード修正と一緒に更新されやすい。

「コードの存在理由」という最もプリミティブな情報は、コードそのものに刻印されているべきなのです。

## 読者が今すぐ試せること

いきなりチェッカーを作る必要はありません。まずは手動で、この習慣を取り入れてみてください。

1.  あなたのプロジェクトの中で、**「最も存在理由が曖昧なファイル」**を1つ開いてください。
2.  そのファイルの1行目に、`# PURPOSE: ` と書き始めてください。
3.  もし書けなければ、そのファイルは**「不要」**か**「設計が破綻している（責務が大きすぎる）」**かのどちらかです。
4.  書けたら、チームメンバーに「これ、わかりやすい？」と聞いてみてください。

「なぜあるか」を問うことは、品質管理の第一歩です。
次回は、コードだけでなく、AIモデルやデータセットそのものの健全性をチェックする方法について解説します。
