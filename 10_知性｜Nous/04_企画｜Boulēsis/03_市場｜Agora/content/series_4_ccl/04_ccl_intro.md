# プロンプトエンジニアリングの「数式」化。CCL開発で気づいた、再現性を劇的に高める思考法【CCL全貌 #4】

「昨日はうまくいったプロンプトが、今日は通じない」
「同じ指示を出しているはずなのに、出力の品質が安定しない」

LLM（大規模言語モデル）を業務で使っていると、こんな**「AIガチャ」**のような状況に頭を抱えることはありませんか？

自然言語は柔軟で強力ですが、同時に**極めて曖昧**です。私たちは普段、文脈や空気感でその曖昧さを補完しあっていますが、確率的に言葉を紡ぐAIにとって、その曖昧さは「出力のブレ」に直結します。

シリーズ最終回となる今回は、これまでの連載で開発してきた**CCL（Cognitive Computation Language）**の運用を通じて得られた、**「プロンプトに代数的な構造を持たせる」**という知見について解説します。

これは、CCLという言語を使わなくても実践できる、**プロンプトエンジニアリングの新しい視点**です。

---

## 1. 自然言語の限界と「構造」の必要性

通常、私たちはAIに対して「手順書」のようなプロンプトを書きます。

> **【よくあるプロンプト】**
> 以下の文章を要約してください。そのあと、要約した内容を英語に翻訳し、最後にビジネスメールの形式に整えてください。

一見問題なさそうですが、ここには**「処理の境界線」**がありません。「要約」と「翻訳」が混ざったり、「メール形式」にする段階で元の意味が抜け落ちたりすることがあります。

CCLの開発過程で痛感したのは、**「自然言語の指示は、演算子（Operator）として定義されていない」**という問題でした。

### 曖昧さの正体

数学の世界では `1 + 2` は常に `3` です。しかし、プロンプトの世界では `A + B` が `AB` になることもあれば、`C` という全く別のものに変質することもあります。

再現性を確保するためには、プロンプトを「お願い」ではなく、**「入力・関数・出力」が明確な数式（あるいはコード）**として捉え直す必要があります。

## 2. CCLのアプローチ：思考を演算する

第3回で紹介したCCLの実装では、プロンプトを**パイプライン処理**として記述します。

```python
# CCL的な思考のイメージ
result = input_text >> summarize >> translate >> format_email
```

このように記述すると、AIは以下のことを強制されます。

1.  `summarize`（要約）を完了し、中間生成物を作る。
2.  その中間生成物を `translate`（翻訳）の入力とする。
3.  その結果を `format_email`（整形）する。

ここで重要なのは、**各ステップが独立した「関数」として機能し、前のステップの出力が次のステップの入力になるという「構造」**です。

## 3. プロンプトに宿る「代数的性質」

CCLを運用して面白かったのは、プロンプトが**代数的性質（Algebraic Properties）**を満たすように設計すると、精度が安定するという発見でした。少し数学的な話になりますが、直感的に理解できるはずです。

### 結合律 (Associativity)

数学では `(a + b) + c = a + (b + c)` が成り立ちます。プロンプトでも同様です。

*   **パターンA：** `(要約して翻訳) してから メールにする`
*   **パターンB：** `要約してから (翻訳してメールにする)`

この2つが同じ結果になるようにプロンプトを設計する必要があります。もし結果が食い違うなら、それは指示の中に「文脈依存の曖昧さ」が含まれている証拠です。

### 分配律 (Distributivity)

`a * (b + c) = a * b + a * c` のような性質です。

*   **一括処理：** 「商品Aと商品Bのレビューをまとめて分析して」
*   **個別処理：** 「商品Aを分析」＋「商品Bを分析」して合算

複雑なタスクの場合、**分配律に従ってタスクを分解（因数分解）してからAIに投げ、最後に結合する**方が、圧倒的に精度が高くなります。CCLではこれを `Map` 演算子として実装しましたが、通常のプロンプトでも「リストの各要素に対して以下の処理を行い、最後に統合せよ」と明記することで再現可能です。

## 4. 実測：構造化あり vs なし

実際に、同じタスクを「自然言語の羅列」と「構造化した指示」で行った場合の比較結果です（GPT-4使用、各10回試行）。

| 項目 | 自然言語プロンプト（非構造化） | 構造化プロンプト（代数的アプローチ） |
| :--- | :--- | :--- |
| **指示内容** | ニュース記事を読み、要約、感情分析、キーワード抽出を行い、JSONで出力する | Step1:要約, Step2:感情分析, Step3:抽出 と定義し、パイプライン処理させる |
| **フォーマット遵守率** | 80%（時々JSONが壊れる、項目が抜ける） | **100%** |
| **内容の一貫性** | 感情分析の結果が要約の内容に引きずられることがある | 各ステップが独立しているため、分析が鋭い |
| **修正の容易さ** | 指示を書き直すと全体バランスが崩れる | 特定のステップ（関数）だけを修正すれば良い |

構造化することで、**「前の処理のハルシネーション（幻覚）が次の処理に伝播する」リスクを遮断できる**ことがわかりました。

## 5. 一般化：明日から使える「構造化プロンプト」

CCLという言語を使わなくても、この知見はすぐに活かせます。プロンプトを書く際、以下のテンプレートを意識してみてください。

### 悪い例：線形な指示
```text
以下の会議録を読んで、重要な決定事項を箇条書きにして。あ、あと未決事項があったらそれも抜き出して。最後に、次の会議のアジェンダ案も作ってほしい。トーンは丁寧語で。
```

### 良い例：関数型プログラミング的指示
```markdown
# 入力データ
[会議録テキスト]

# 処理定義
1. **関数 extract_decisions**: 入力から決定事項を抽出し、リスト化する。
2. **関数 extract_pendings**: 入力から未決事項を抽出し、リスト化する。
3. **関数 draft_agenda**: 上記の「未決事項」をもとに、次回のアジェンダ案を作成する。

# 実行手順
Step 1: extract_decisions を実行
Step 2: extract_pendings を実行
Step 3: draft_agenda を実行

# 出力形式
それぞれの結果を以下のフォーマットで出力すること...
```

このように**「定義」と「実行」を分ける**だけで、AIは論理構造を理解しやすくなり、出力の精度が劇的に向上します。これが、CCLが目指した**「思考のプログラミング化」**のエッセンスです。

このプロジェクトを通じて確信したのは、**「AIへの指示は、もはや会話ではなくプログラミングである」**ということです。
