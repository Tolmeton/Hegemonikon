# 【CCL全貌 #1】思考の「行ったり来たり」を数式にする — 認知の振動演算子 `~`

「思考」は、決して一直線には進みません。

新しい技術を学ぶとき、あるいは複雑なバグの原因を探るときを思い出してください。仮説を立てて検証し、少し理解してはまた新たな疑問にぶつかり、最初の仮説を修正する……。私たちは常に**「認識」と「評価」の間を行ったり来たり**しています。

この「行ったり来たり」こそが、理解を深める本質的なプロセスです。しかし、従来のフローチャートや手続き型プログラミングのような直線的なモデルでは、このダイナミックな心の動きをうまく表現できませんでした。

本連載「CCL全貌」では、思考のプロセスを記述するための言語 **CCL (Cognitive Categorical Language)** を通じて、私たちの認知をコードのように書き下す方法を探求します。

記念すべき第1回は、思考のループを司る**振動演算子（Oscillation Operator） `~`** について解説します。

---

## 1. 思考のループを記号化する `~`

CCL（Cognitive Categorical Language）とは、HGK（Hyper-Geometric Knowledge）体系において提唱される、**認知プロセスを圏論的・代数的に記述するためのドメイン固有言語**です。

通常、プログラムは `Input -> Process -> Output` と流れます。しかし、人間の思考は `Input -> Process -> Re-Input -> Re-Process...` と循環します。

この循環を表現するために、CCLでは **振動演算子 `~`（チルダ）** を導入しました。

### 直線的な処理 vs 振動的な処理

| 種類 | 記法 (CCL) | 意味 | イメージ |
| :--- | :--- | :--- | :--- |
| **射 (Arrow)** | `A -> B` | AからBへ変換する | 一方通行の処理 |
| **振動 (Oscillation)** | `A ~ B` | AとBの間を往復し、状態を更新する | らせん階段を登る |

この `~` は、単なる無限ループではありません。「ある状態に収束するまで、相互作用を繰り返す」という、**動的な平衡状態**を目指すプロセスを表しています。

## 2. 具体例：直感と論理の対話 `/noe~/dia`

では、具体的な認知モデルで考えてみましょう。
HGKの文脈では、人間の認識を大きく2つの機能に分けます。

1.  **Noēsis (ノエシス / 直観的把握)**:
    対象をパッと見て「あ、これはアレだ」と全体像を掴む機能。パターン認識に近い入力処理。
2.  **Diakrisis (ディアクリシス / 分別・判断)**:
    掴んだ対象に対して「正しいか？」「どう分類されるか？」を論理的に評価・分析する機能。

思考が深まるとき、私たちはこの2つを高速で往復させています。

### 思考のトレース

1.  **Noēsis**: コードを見て「認証機能の実装だな」と直感する。
2.  **Diakrisis**: 細部を読む。「待てよ、トークンの有効期限チェックが抜けていないか？」と評価する。
3.  **Noēsis (再)**: その評価を持って再度コードを見る。「確かに抜けている。ここに追加すべきだ」と認識が更新される。
4.  **Diakrisis (再)**: 「追加した場合、副作用はないか？」と再評価する。

このプロセスをCCLでは以下のように記述します。

```ccl
// 認識(noe)と判断(dia)の振動
process DeepUnderstanding = /noe ~ /dia
```

この `/noe ~ /dia` というたった数文字の式が、**「納得いくまで考え抜く」** という複雑な精神活動を表現しているのです。

## 3. 実装：収束振動 `~*`

エンジニアの皆さんなら、「そのループ、いつ終わるの？」と気になるはずです。無限ループはバグの元ですからね。

CCLでは、振動が安定解（不動点）に達して停止することを明示する場合、**収束振動演算子 `~*`** を用います。

これをPython風の擬似コードで表現すると、以下のようになります。

```python
def oscillation_operator(state, analyze_func, evaluate_func, threshold=0.01):
    """
    CCLにおける `~*` (収束振動) の実装イメージ
    """
    current_state = state
    
    while True:
        # 1. 分析フェーズ (Noesis)
        perception = analyze_func(current_state)
        
        # 2. 評価フェーズ (Diakrisis)
        judgment, error = evaluate_func(perception)
        
        # 3. 収束判定
        # 予測誤差(error)が閾値を下回れば、認識が安定したとみなす
        if error < threshold:
            return judgment
        
        # 4. 状態更新（次のループへ）
        # 評価結果をフィードバックして、視点を更新する
        current_state = update_state(current_state, judgment)
```

このコードにおける `error` がゼロに近づくプロセスこそが、「分かった！」というアハ体験（Aha-experience）の正体です。

## 4. 理論的背景：FEPと圏論

なぜ、わざわざこのような演算子を定義するのでしょうか？ それは、現代の認知科学や数学の知見と整合させるためです。

### 自由エネルギー原理 (FEP) との接続

カール・フリストンが提唱する**自由エネルギー原理 (Free Energy Principle)** では、脳は「予測誤差（サプライズ）を最小化するように推論を更新し続ける」とされます。

CCLの `~` 演算子は、まさにこの**アクティブ・インファレンス（能動的推論）のサイクル**そのものです。
*   `/noe` = 感覚入力に基づく予測の生成
*   `/dia` = 予測誤差の計算とモデルの修正
*   `~*` = 自由エネルギーの最小化（平衡状態への到達）

### 圏論的視点：自己関手と不動点

数学（圏論）の言葉を使えば、振動は**自己関手（Endofunctor）の反復適用**として捉えられます。

ある認知状態の空間を $\mathcal{C}$ とし、思考プロセスを関手 $F: \mathcal{C} \to \mathcal{C}$ とします。
振動プロセスは、$X, F(X), F(F(X)), \dots$ という列を生成します。
そして、収束振動 `~*` が目指すのは、 $F(X) \cong X$ となる**不動点（Fixed Point）** $X$ を見つけることです。

「思考がまとまる」とは、数学的には「認知の不動点を見つける」ことと同義なのです。

## 5. 日常における「振動」の事例

このモデルは、プログラミングや数学以外にも応用できます。例えば「難解な技術論文を読む」シーンを考えてみましょう。

1.  **1周目 (Scan)**: 全体をざっと読む。数式は飛ばす。「なんとなくこういう話か」と `/noe` する。
    *   *誤差大*: 細部が矛盾しているように見える。
2.  **2周目 (Deep Dive)**: 疑問点（`/dia` での引っかかり）を中心に読み直す。用語の定義を確認する。
    *   *誤差中*: 「あ、この変数はこういう意味か」と `/noe` が更新される。
3.  **3周目 (Synthesize)**: 自分の言葉で説明できるか試す。
    *   *誤差小*: 記述内容と自分の理解が一致する（収束 `~*`）。

この「何周もして解像度を上げていく」プロセスを、CCLでは**「螺旋的推論（Spiral Inference）」**と呼び、`~` 演算子の主要な適用領域としています。

*   **Step 1**: 今、自分は「情報を集めている（`/noe`）」のか、「評価・判断している（`/dia`）」のか？

*   **Step 3**: もし空転しているなら、ループの条件（評価基準）を変える必要があるかもしれません。
