# マルコフ圏で「思考」をモデリングする

## 1. なぜ「マルコフ圏」なのか？

従来のプログラミング（決定論的なロジック）と、生成AI（確率的な挙動）の間には深い溝があります。

- **論理の世界:** $1 + 1$ は必ず $2$。
- **AIの世界:** 「空」の次は、高い確率で「青い」だが、たまに「広い」かもしれない。

この両者を統一的に扱う数学的舞台が、圏論の一分野である**マルコフ圏**です。

マルコフ圏の定義は厳密には複雑ですが、思考のモデリングという観点では、以下の2つの特徴が重要です：

1.  **情報の複製（Copy）ができる:** ある文脈を保持したまま、別の処理に回す（プロンプトのコンテキスト維持）。
2.  **情報の破棄（Discard）ができる:** 不要な情報を捨て、結論だけを残す。

「なんだ、当たり前じゃないか」と思われるかもしれません。しかし、純粋数学の世界（特に線形論理など）では、情報の複製や破棄は自明ではありません。
**「文脈を共有しつつ（複製）、確率的に分岐し、不要なノイズを捨てる（破棄）」**というLLMの挙動は、まさにマルコフ圏の構造そのものなのです。

CCLは、この構造をコードとして表現します。

## 2. CCLの2つの軸：PoiēsisとPragmata

CCLでは、思考の動きを大きく2つの軸（次元）で捉えます。これらはHGK（Hyper-Geometric Knowledge）体系における独自の概念ですが、直感的には「作る動き」と「進める動き」です。

### 縦軸：Poiēsis（ポイエーシス / 生成・変容）
ギリシャ語で「制作」「創作」を意味します。情報の「質」や「量」を変化させる操作です。

*   **`+` (Expansion):** 情報を広げる、具体例を出す、発散させる。
*   **`-` (Distillation):** 情報を要約する、抽象化する、本質を抽出する。
*   **`~` (Oscillation):** 揺らがせる、視点を変える、推敲する。

### 横軸：Pragmata（プラグマタ / 実行・制御）
ギリシャ語で「行為」「事実」を意味します。情報の「流れ」や「接続」を制御する操作です。

*   **`/` (Workflow):** 外部のエージェントやツールを呼び出す。
*   **`>>` (Pipe):** 出力を次の入力へ渡す。
*   **`{}` (Scope):** 処理の範囲を限定する。

CCLのスクリプトは、この「生成（縦）」と「実行（横）」を編み合わせることで、思考の織物を作り上げます。

## 3. CCL演算子と圏論の対応表

では、CCLの記号が具体的にどのような数学的構造に対応しているのか見てみましょう。これこそが、CCLが「ただのプロンプト集」ではなく「プログラム」として機能する理由です。

| CCL演算子 | 動作イメージ | マルコフ圏・圏論での対応 | 解説 |
| :--- | :--- | :--- | :--- |
| **`/`** | 実行・呼出 | **射の合成 (Composition)** | 関数 $f$ の後に $g$ を適用する $g \circ f$。思考のステップをつなぐ基本操作。 |
| **`~`** | 振動・反復 | **自己関手の反復 (Endofunctor)** | 同じプロセス $F$ を何度も適用する（$F^n$）。推敲ループなどで使用。 |
| **`*`** | 収束 | **不動点 (Fixed Point)** | 入力と出力が変化しなくなる状態（$x = f(x)$）を目指す。議論の収束。 |
| **`+`** | 展開 | **左随伴 (Left Adjoint)** | 自由生成。コンテキストから可能性を広げる（エントロピー増大）。 |
| **`-`** | 蒸留 | **右随伴 (Right Adjoint)** | 忘却・要約。構造を保ったまま情報を圧縮する（エントロピー減少）。 |

特に面白いのが `+` と `-` の関係です。
「具体例をたくさん出して（`+`）、それらに共通する法則を抜き出す（`-`）」という思考プロセスは、数学的には**随伴（Adjunction）**と呼ばれる構造に近い動きをしています。CCLはこの「思考の呼吸」を演算子として定義しています。

## 4. Kalon正則化：美しさは「不動点」にある

HGKには**「Kalon（カロン）」**という概念があります。古代ギリシャ語で「美」や「高貴さ」を意味しますが、CCLにおいては**「構造的な安定性」**と定義されます。

思考プロセスにおいて「美しい結論」とは何でしょうか？
それは、**「これ以上考えても答えが変わらない状態」**です。

例えば、あるアイデアを「批判（`-`）」して「再構築（`+`）」するプロセスを考えます。
最初はアイデアが大きく変わりますが、繰り返すうちに変化が小さくなり、最終的に強固なコアだけが残ります。

数学的には、展開（$F$）と蒸留（$G$）の合成関数 $G \circ F$ を繰り返し適用し、不動点（Fixed Point）に到達することを意味します。

CCLでは以下のように記述します。

```ccl
// アイデア(Idea)に対して、批判と再構築を3回繰り返し、収束(Kalon)させる
Idea >> ~3 {
    - (Criticize: 欠点を指摘し、情報を削ぎ落とす)
    + (Refine: 指摘を踏まえて具体化・強化する)
} *
```

最後の `*` は、このループが「安定するまで（あるいは指定回数で強制的に）収束させる」という意思表示です。これを**Kalon正則化**と呼びます。

## 5. 実践：数学的に正しい思考パイプライン

これらを組み合わせると、信頼性の高い「思考パイプライン」を設計できます。
単に「いい感じに書いて」と頼むのではなく、構造を指定するのです。

以下は、「曖昧な要望」を「仕様書」に変換するCCLの実装例です。

```ccl
// 入力：曖昧なクライアントの要望
Input >> {
    // 1. 発散フェーズ：可能性を広げる（左随伴的アプローチ）
    + (Brainstorm: 隠れた前提やエッジケースを列挙せよ)
    
    // 2. 構造化フェーズ：論理的整合性をチェック（射の合成）
    / (CheckLogic: 矛盾点を洗い出す)
    
    // 3. 収束フェーズ：本質を抽出する（右随伴的アプローチ）
    - (Summarize: 実装可能な仕様として要約せよ)
    
    // 4. 検証ループ：仕様として「美しく（安定）」なるまで磨く
    ~2 {
        / (Review: 開発者視点でレビュー)
        + (Fix: 指摘箇所の修正)
    } *
} >> Output
```

このコードは、プロンプトの羅列ではありません。
「広げて（`+`）、整えて（`/`）、絞って（`-`）、磨く（`~`）」という、**思考のアルゴリズム**そのものです。

## まとめ

CCLがマルコフ圏をベースにしている理由は、AIという「確率的な生き物」を、「論理的な構造」の中に飼い慣らすためです。

- **Poiēsis（`+`, `-`, `~`）** で思考の質を操り、
- **Pragmata（`/`, `>>`）** で思考の流れを操る。
- そして **Kalon（`*`）** で結論を安定させる。

この3つが揃ったとき、AIは単なるチャットボットから、信頼できる「思考エンジン」へと進化します。

### 読者が試せること

今抱えている複雑なタスクを、CCLの記号（`+`, `-`, `/`, `~`）を使って紙に書き出してみてください。

*   「ここはアイデアを広げたいから `+` だな」
*   「ここは一度要約しないとコンテキストが溢れるから `-` だな」
*   「このレビュー工程は2回回したいから `~2` だな」

それだけで、漫然としていたプロンプトエンジニアリングが、構造化された「設計」に変わる感覚を掴めるはずです。
