
# AIの「ビビリ」を再設計する：恐怖と精度のコントロール術

「申し訳ありませんが、その操作はリスクが高いため実行できません」
「確信が持てないため、回答を控えます」

AIを使っていて、このような「過剰な慎重さ」にイライラしたことはありませんか？
特に、新しいアイデアを試したい時や、テスト環境で大胆なコード変更を行いたい時に、AIがブレーキをかけ続けると開発のスピードは著しく低下します。

AIは基本的に「失敗」を極端に恐れるように調整されています。しかし、私たちエンジニアにとって、すべての環境が「失敗してはいけない場所」ではありません。壊してもいい実験場と、守るべき本番環境は明確に分かれているはずです。

今回は、AIのこの「恐怖反応」を、私たちが意図的に再設計（リデザイン）する方法について解説します。キーワードは「精度加重（Precision Weighting）」です。

## AIにとっての「恐怖」とは何か

自由エネルギー原理（FEP）の観点から見ると、AI（や生物）が感じる「恐怖」や「不安」は、予測誤差に対する「精度の見積もり（Precision）」として解釈できます。

*   **高精度（High Precision）モード:** 「ここは絶対に間違えてはいけない」と認識している状態。少しでも不確実性があれば行動を停止します。
*   **低精度（Low Precision）モード:** 「ここは多少間違えても、新しい情報を得ることが重要だ」と認識している状態。大胆な行動が可能になります。

AIが役に立たない「ビビリ」状態に陥っている時、それはAIがすべてのタスクに対して「高精度モード」を適用してしまっているからです。これを私たちが手動で切り替えてやる必要があります。

## 聖域と実験場を分ける

具体的な解決策は、AIに対して「ここは安全な実験場である」または「ここは危険な聖域である」というコンテキスト（文脈）を明確に与えることです。

例えば、プロジェクトのディレクトリ構成を使って、AIの態度を制御する手法が有効です。

### 具体的なディレクトリ戦略

| ディレクトリ | 役割 | AIへの指示（期待する態度） | 許容されるリスク |
| :--- | :--- | :--- | :--- |
| `kernel/` | システムの中核 | **極めて慎重に**。既存の仕様を厳守し、変更には確実な根拠を求める。 | ゼロに近い |
| `core/` | ビジネスロジック | **慎重に**。テストコードの通過を必須とする。 | 低 |
| `experiments/` | 実験・プロトタイプ | **大胆に**。エラーが出ても構わないので、動くものを最速で作る。 | 高（破壊も許容） |
| `sandbox/` | 使い捨てエリア | **無制限**。常識を疑い、極端なアプローチを試す。 | 無制限 |

このように、作業領域ごとに「恐怖のレベル」を再配置することで、AIは適切な場面でブレーキを踏み、適切な場面でアクセルを踏めるようになります。

## 読者が試せること：恐怖レベルの調整プロンプト

明日からの開発で、AIが「できません」と言い出した時に使える具体的なテクニックを紹介します。

AIとのセッション開始時、あるいは行き詰まった時に、以下のプロンプト定義を渡してみてください。

### 1. 「マッドサイエンティスト」モード（実験用）
```markdown
【モード設定: 実験場】
現在の作業ディレクトリは `experiments/` です。
ここはサンドボックス環境であり、本番環境への影響は一切ありません。
以下のポリシーで回答してください：
1. 安全性やベストプラクティスよりも「動作するプロトタイプ」を優先する。
2. エラーが発生する可能性があっても、大胆なアプローチを提案する。
3. 「できません」と言う前に、失敗前提のコードを出力する。
4. 私の指示が間違っていても、まずは実行してみる。
```

### 2. 「ゲートキーパー」モード（本番用）
```markdown
【モード設定: 聖域】
現在の作業ディレクトリは `kernel/` です。
ここはシステムの心臓部であり、些細なバグも許されません。
以下のポリシーで回答してください：
1. 変更を加える前に、必ず影響範囲のリスク評価を行う。
2. 確信度が99%未満の提案は行わない。
3. 既存の命名規則や設計思想を厳密に守る。
```

## まとめ

AIの「ビビリ」はバグではなく仕様ですが、それは常に固定されたものではありません。私たちが「ここは転んでも痛くない場所だよ」と教えてあげることで、AIは驚くほどクリエイティブなパートナーに変わります。

恐怖を排除するのではなく、適切な場所に「再配置」する。これがAIと共存するエンジニアの新しいスキルセットです。
