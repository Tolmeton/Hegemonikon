
# MCP で25ツールを統合する — AIに「手足」を与える苦難とGatewayパターン

AIモデルは非常に賢いですが、デフォルトでは「箱入り」です。インターネットの向こう側にあるデータベースを直接操作したり、ローカルのファイルシステムを自由に歩き回ったりすることはできません。

そこで注目されているのが **MCP (Model Context Protocol)** です。これはAIに外部ツールへの安全なアクセス権を与えるための標準規格です。

私はこのMCPを使って、自作のAIエージェントに **7つのサーバー、計25個のツール** を統合しました。これによってAIは、ファイル操作からDB検索、APIコールまでを自律的に行えるようになりました。しかし、そこに至る道は平坦ではありませんでした。

## 統合した25ツールの全貌

私が構築した環境では、AIは以下のツール群を自由に呼び出すことができます。

| サーバーカテゴリ | ツール数 | 主な機能 | 難易度 |
| :--- | :---: | :--- | :---: |
| **FileSystem** | 5 | ファイルの読み書き、ディレクトリ探索、差分確認 | 中 |
| **Database** | 8 | SQL実行、スキーマ確認、VectorDB検索 | 高 |
| **Web/API** | 6 | ブラウジング(Puppeteer)、外部APIリクエスト | 高 |
| **Utility** | 4 | 時間取得、乱数生成、Base64エンコード | 低 |
| **Git** | 2 | コミットログ取得、ブランチ確認 | 中 |

これらを統合することで、AIは「データベースからエラーログを検索し（DB）、関連するソースコードを読み（FileSystem）、修正案を作成してWebでドキュメントを確認する（Web）」といった一連の作業を完遂できるようになりました。

## 実装の壁：stdin/stdout汚染問題

MCPは、標準入出力（stdin/stdout）を使ってJSON-RPCという形式で通信を行います。これが最大の落とし穴でした。

開発中、多くのライブラリやツールは、デバッグ情報や警告を不用意に `print()` で標準出力（stdout）に吐き出します。しかし、MCPにおいて標準出力は「通信回線」そのものです。そこに「Warning: ...」のようなプレーンテキストが混ざると、JSONのパースエラーが発生し、通信が切断されてしまうのです。

### 解決策：Gateway Pattern と stderrへの逃がし

この問題を解決するために、私は **Gateway Pattern** を採用しました。

1.  **Gatewayサーバーの設置**:
    AIと個々のMCPサーバーの間に、通信を中継する「Gateway」プロセスを挟みます。
2.  **標準出力のハイジャック**:
    各ツールのプロセスを起動する際、PythonやNode.jsのレベルで `stdout` を厳密に管理。ライブラリが勝手に出力するログはすべて強制的に `stderr`（標準エラー出力）へリダイレクトさせました。
3.  **クリーンな通信**:
    Gatewayは、検証済みのJSON-RPCメッセージだけを通過させます。

これにより、行儀の悪いライブラリが含まれていても、MCPの通信は安定し、25個ものツールを同時に扱える堅牢性が確保されました。

## 読者が試せること：SQLiteサーバーを立ててみる

MCPの世界に触れるなら、まずは公式が提供しているシンプルな例から始めるのが一番です。

1.  **Cursor や Windsurf をインストール**
    最近のAIエディタはMCPをネイティブサポートしています。
2.  **SQLite MCP Server を設定**
    ローカルにあるSQLiteファイルをAIに操作させてみましょう。
    設定ファイルに以下のように記述するだけで、AIがDBの中身を見られるようになります。

    ```json
    "mcpServers": {
      "sqlite": {
        "command": "uvx",
        "args": ["mcp-server-sqlite", "--db-path", "./my-data.db"]
      }
    }
    ```

「AIが自分のデータベースの中身を知っている」という体験は、これまでのチャットボットとは一線を画す感動があります。ぜひ、最初の一歩を踏み出してみてください。
