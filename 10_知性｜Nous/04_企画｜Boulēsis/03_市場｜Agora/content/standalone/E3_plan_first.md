
## 記事4：AI時代の実装は2割でいい。「計画8割」にシフトするエンジニアの生存戦略

「エンジニアならコードを書け」「手を動かしながら考えろ」。
かつてはそれが正解でした。試行錯誤のコストが高く、動くものを作って初めて分かることが多かったからです。

しかし、生成AIの登場でゲームのルールは完全に逆転しました。コードを書くコストは劇的に下がり、実装スピードは人間の数十倍、数百倍になりました。

この時代において、人間が「実装」に時間をかけることは、もはや価値ではなくなりつつあります。代わりに急浮上しているのが「計画（Plan）」の重要性です。

今回は、「計画8割・実装2割」という新しい黄金比について解説します。AIを部下として使いこなし、圧倒的な成果を出すためのワークフロー変革を提案します。

### 実装コストの崩壊と、計画の価値高騰

従来のソフトウェア開発では、工程の多くを「実装（Coding）」と「デバッグ」が占めていました。
*   要件定義・設計：3割
*   実装・テスト：7割

しかし、GitHub CopilotやGPT-4などのAIツールを使えば、関数の実装や定型的なユニットテストは数秒で完了します。人間が1時間かけていた作業が、1分で終わるのです。

ここで浮いた時間をどこに投資すべきか？ それは**「何を作るか（What）」**と**「どう構成するか（Architecture）」**の設計、すなわち計画です。

AIは「指示されたもの」を作るのは天才的に上手いですが、「何を作るべきか」を自律的に決めることはできません（現時点では）。曖昧な指示を出せば、曖昧な（そして動かない）コードが高速で生成されるだけです。
AI時代の生産性は、「AIに渡す設計図の精度」で決まります。

### 新しい開発サイクル：/plan → /dev → /vet

AIネイティブな開発フローは、以下の3つのフェーズで回ります。

#### 1. /plan（計画・設計）：80%の時間
ここが人間の主戦場です。
*   **要件の明確化**: ユーザーは誰か？ 何を解決したいか？
*   **コンテキストの言語化**: 制約条件は？ 使用するライブラリは？
*   **疑似コード/プロンプト設計**: 自然言語でロジックを組み立てる。

ここではコードを一行も書きません。代わりに、AIに与えるための「完璧な指示書（コンテキストファイル）」を作り込みます。マークダウンで仕様をまとめ、データ構造を定義し、エッジケースを洗い出します。

#### 2. /dev（実装・開発）：10%の時間
ここはAI（と少しの人間）の出番です。
*   **コード生成**: `/plan` で作った指示書をAIに投げます。
*   **自動実装**: AIが一瞬でコードを吐き出します。
*   **微修正**: 生成されたコードの軽微な修正。

計画が完璧なら、このフェーズは驚くほどスムーズに進みます。「ボタンを押して待つだけ」に近い感覚です。

#### 3. /vet（検証・審査）：10%の時間
最後にもう一度人間が介入します。
*   **コードレビュー**: AIが書いたコードがセキュリティ基準を満たしているか？
*   **動作確認**: 意図通りに動くか？
*   **責任の担保**: 出力結果に対する責任を持つ。

### 具体事例：チャットボット開発の時間配分

「社内ドキュメントを検索できるチャットボット」を作るとします。

**【従来の方法】**
いきなりPythonファイルを書き始め、LangChainのドキュメントを読みながら試行錯誤。「あれ、PDFの読み込みどうやるんだっけ？」と調べながら実装。バグが出たらprintデバッグ。
→ **所要時間：3日**

**【AI時代の方法（計画8割）】**
最初の2日間はエディタを開かず、テキストファイルに以下を記述します。
*   「入力はPDFとMarkdown」
*   「ベクトルDBはChromaを使用」
*   「回答が見つからない場合は『分かりません』と答える」
*   「クラス構成は `Ingestor`, `Retriever`, `Bot` の3つ」
*   「各メソッドの入出力定義」

この詳細な設計書（プロンプト）をAIに入力します。AIは数分でほぼ完璧なコードを出力します。あとは動作確認するだけ。
→ **所要時間：2.1日（うち実装は0.1日）**

トータルの時間は短縮され、かつ設計書というドキュメントが残るため、メンテナンス性も向上します。

### 従来フロー vs AI時代のフロー

| 項目 | 従来（Human Driven） | AI時代（AI Driven） |
| :--- | :--- | :--- |
| **主役** | 手を動かすエンジニア | 設計するアーキテクト |
| **時間の使い方** | How（どう書くか）に悩む | What（何を作るか）を練る |
| **ボトルネック** | タイピング速度、構文知識 | 指示の明確さ、論理的整合性 |
| **失敗の原因** | バグ、実装ミス | 要件の曖昧さ、指示不足 |
| **成果物** | コード | 設計書 + コード |

### 今すぐ読者が試せること

次のタスクに取り掛かるとき、**「コードを書くのを我慢する」**というトレーニングをしてみてください。

1.  **エディタを開かない**: メモ帳やNotionだけを開く。
2.  **自然言語プログラミング**: やりたい処理、変数名、関数の役割、エラー処理の方針などを、すべて日本語（または英語）で書き出す。
3.  **自分への問いかけ**: 「この指示書を、何も知らない新人に渡して、迷わず実装できるか？」と自問する。
4.  **AIに投げる**: そのテキストをそのままChatGPTやCopilotに貼り付ける。

もしAIが一発で期待通りのコードを書けなかったら、それはAIのせいではなく、あなたの「計画（指示）」が甘かったからです。
「実装はAIに任せ、人間は思考に特化する」。このシフトチェンジこそが、これからのエンジニアの生存戦略であり、最もクリエイティブな働き方になるはずです。
