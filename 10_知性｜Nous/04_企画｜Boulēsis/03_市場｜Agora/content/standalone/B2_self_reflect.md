
# AIの「思い込み」を打破する：自己反省（Self-Reflection）プロンプトの実装法

## AIの「もっともらしい嘘」を見抜けますか？

「この契約書の条項にリスクはありますか？」
「この患者の症状から考えられる病名は？」

AIがこのように高度な判断を求められるシーンが増えています。しかし、大規模言語モデル（LLM）には、「ハルシネーション（幻覚）」という持病があります。事実ではないことを、さも事実であるかのように自信満々に語る現象です。

AIの回答を鵜呑みにして、後で痛い目を見た経験はないでしょうか？ 特に、ハイリスクな意思決定にAIを用いる場合、この問題は致命的です。

AIの精度を上げるためにモデル自体を再学習させるのはコストがかかります。しかし、プロンプトエンジニアリングの工夫だけで、AIに「自分の回答を疑わせる」ことは可能です。それが、今回紹介する**「自己反省（Self-Reflection）」**というテクニックです。

## AIに「考え直す」プロセスを強制する

人間でも、重要なメールを送る前には「書き間違いはないか？」「誤解を招く表現はないか？」と読み返します。これを「推敲」や「内省」と呼びます。

デフォルトのAIは、入力に対して確率的に最もありそうな単語を次々と出力する「書き殴り」モードで動いています。これに対し、自己反省プロンプトは、一度出した答えに対して「本当にそれでいいのか？」とツッコミを入れるプロセスを強制します。

具体的には、以下の2つの手法が有効です。

1.  **Fact-and-Reflection (FaR)**: 事実と反省の分離
2.  **T3内省**: 連続実行時のトレードオフ分析

## 1. Fact-and-Reflection (FaR) メソッド

FaRメソッドは、回答生成プロセスを3つのステップに分解する方法です。いきなり結論を出させるのではなく、まず材料を並べさせ、次に批判させ、最後に結論を出させます。

*   **Step F (Fact List)**: 確実な事実のみを箇条書きにする。ここでは推測を入れない。
*   **Step R (Reflection)**: Step Fに基づき、これから導き出そうとする回答に「潜在的な誤り」や「論理の飛躍」がないか、AI自身に批判的レビューを行わせる。
*   **Step C (Conclusion & Confidence)**: FとRを踏まえて結論を出し、その結論に対する「確信度（0-100%）」を数値で提示する。

### 具体事例：医療情報の要約

通常のAIに「頭痛と吐き気がある場合の病気は？」と聞くと、「片頭痛や風邪でしょう」と即答しがちです。FaRを適用するとこうなります。

*   **Step F（事実）**: ユーザーは頭痛と吐き気を訴えている。これらは非特異的な症状であり、片頭痛、緊張型頭痛、くも膜下出血、脳腫瘍、胃腸炎など多岐にわたる疾患で見られる。ユーザーの年齢、性別、発症の急激さに関する情報はない。
*   **Step R（反省）**: 情報が不足している中で特定の病名を挙げることは、診断ミス（見逃し）のリスクがある。特に「くも膜下出血」のような緊急性の高い疾患を除外せずに、一般的な「風邪」と断定するのは危険である。
*   **Step C（結論）**: 特定の病名は断定できません。緊急性の高い疾患の可能性も否定できないため、直ちに医師の診断を受けてください。特に「ハンマーで殴られたような痛み」がある場合は救急要請が必要です。（確信度：95% - 断定を避けたため高信頼）

このように、一度立ち止まって考えさせることで、安易な回答を防ぐことができます。

## 2. T3内省：設計意図の言語化

コーディングやシステム設計など、連続して判断が必要なタスクでは、「T3内省」が有効です。これは、AIが何かを生成した後に、以下の3つの質問（Three Questions）を自問自答させる手法です。

1.  **なぜこの設計にしたか？**（意図の明確化）
2.  **何を犠牲にしたか？**（トレードオフの認識）
3.  **犠牲にして良かったか？**（妥当性の検証）

例えば、AIにプログラムコードを書かせたとします。T3内省を行わせると、「処理速度を優先するために、可読性を犠牲にしました。今回の要件はリアルタイム処理なので、このトレードオフは正当です」といった注釈が付くようになります。これにより、ユーザーはAIの意図を理解し、採用するかどうかの判断が容易になります。

## プロセスによる出力品質の差

自己反省プロセスを経たAIとそうでないAIの出力プロセスの違いを表にまとめました。

| ステップ | 通常のAI（即答型） | 自己反省AI（FaR/T3適用） |
| :--- | :--- | :--- |
| **処理の流れ** | 入力 → [確率計算] → 出力 | 入力 → [事実抽出] → [自己批判] → [修正] → 出力 |
| **回答の傾向** | 一般論、平均的な回答、断定的。 | 限定的、条件付き、慎重、根拠が明示される。 |
| **ハルシネーション** | 発生しやすい（文脈を埋めるために嘘をつく）。 | 発生しにくい（「事実リスト」にないことは書かない）。 |
| **ユーザーの安心感** | 低い（根拠が不明）。 | 高い（思考プロセスが開示されているため）。 |

## 読者が試せること：FaR実装プロンプト

ChatGPTやClaudeで、重要な質問をする際に以下のプロンプトを追加してみてください。特に、正確性が求められるリサーチや分析タスクで威力を発揮します。

```markdown
# Instructions
あなたのタスクは、ユーザーの質問に対して極めて正確かつ慎重に回答することです。回答を出力する前に、以下の「Fact-and-Reflection (FaR)」プロセスを内部で実行し、その思考過程も含めて出力してください。

## FaR Process
1. **[Step F: Facts]**
   - 質問に関連する「客観的かつ確実な事実」のみをリストアップしてください。推測や意見は一切含めないでください。
   - 情報が不足している場合は、その事実も記載してください。

2. **[Step R: Reflection]**
   - Step Fに基づき、回答を構築する上で考えられる「論理の飛躍」「バイアス」「誤りの可能性」を挙げてください。
   - 「もし〜だとしたら、この結論は間違っている」という反証を行ってください。

3. **[Step C: Conclusion]**
   - 上記を踏まえた最終的な回答を記述してください。
   - 最後に、この回答の「確信度（0-100%）」と、確信度が100%でない理由を1行で添えてください。
```

このプロンプトを使うと、AIの回答が長くなりますが、その分「思考の深さ」が段違いになります。「AIは間違えるもの」という前提に立ち、その間違いをAI自身に検知させる仕組みを作ること。それが、AIを実務で使いこなすための鍵となります。
