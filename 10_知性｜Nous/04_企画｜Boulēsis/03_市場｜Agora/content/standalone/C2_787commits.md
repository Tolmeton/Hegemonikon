# 787コミットの内訳 — AIとの共同開発、怒涛の1ヶ月で見えた「加速の法則」

「AIを使って開発スピードを上げたい」
そう思っても、実際に手を動かすとプロンプトエンジニアリングに時間を取られ、結局自分で書いた方が早かった……そんな経験はありませんか？

私も最初はそうでした。しかし、ある「体制」を整えた瞬間、開発速度が劇的に変わりました。

この記事では、私がAIパートナーたちと共に、わずか1ヶ月で787コミットを積み上げた実録データと、そこで起きた「加速曲線」の秘密、そして複数のAIモデルを使い分けるMulti-Model戦略について共有します。

## 1ヶ月で787コミット：数字が語る異常な生産性

まず、このグラフのような数字を見てください。これは私がAIと共同開発を行った1ヶ月間のコミット数の推移です。

| 期間 | コミット数 | 開発の状況 |
| :--- | :---: | :--- |
| **第1週** | 50 | プロンプトの試行錯誤。AIに文脈を理解させるのに苦労。 |
| **第2週** | 120 | 共通言語（ドメイン知識）がAIに定着し始める。 |
| **第3週** | 317 | 実装が加速。定型的な処理は一瞬で終わる。 |
| **第4週** | **300** | 複雑なリファクタリングすら自動化。完全にフロー状態。 |

注目すべきは、**第1週と第4週の差**です。最初は週50コミット程度でしたが、後半には週300コミットを超えています。これは単なる慣れではありません。「正のフィードバックループ」が形成されたからです。

### 正のフィードバックループとは？

コードベースが成長するほど、AIに渡す「コンテキスト（文脈）」が豊かになります。既存のコードが優れた「Few-Shot Learning（少数の実例）」として機能し、AIは「ああ、このプロジェクトではこういう書き方をするんですね」と、阿吽の呼吸でコードを生成するようになります。

つまり、**書けば書くほど、AIは賢くなり、次はもっと速く書けるようになる**のです。このループに入ると、開発は爆発的に加速します。

## Multi-Model戦略：適材適所のチームビルディング

787コミットを支えたもう一つの要因は、単一のAIモデルに依存しなかったことです。人間にも得意不得意があるように、AIモデルにも明確な「役割」があります。私は以下の3つのモデルを「チームメンバー」として採用しました。

| 役割 | モデル | 特徴・採用理由 |
| :--- | :--- | :--- |
| **アーキテクト** | **Claude 3.5 Sonnet** | 文脈保持能力が最強。システム全体の設計、クラス図の生成、矛盾の指摘を担当。長文の仕様書を読み込ませても破綻しない。 |
| **実装担当** | **Jules (GitHub Copilot)** | VS Code統合が優秀。Claudeが決めた設計を元に、実際の関数やメソッドを高速に打ち込む。IDE上での文脈理解が深い。 |
| **レビュアー** | **Gemini 1.5 Pro** | トークン単価が安く、コンテキストウィンドウが広大。ファイル全体を読み込ませてのバグ探しや、別角度からのセキュリティチェックに最適。 |

このように、設計はClaude、実装はJules、レビューはGeminiと役割を分担させることで、各モデルの強みを最大限に引き出し、弱点を補い合いました。特にClaudeの設計をJulesに渡すフローは、人間のテックリードとジュニアエンジニアの関係に近く、非常にスムーズでした。

## 読者が試せること：まずは「設計」と「実装」を分けてみる

いきなり3つのモデルを使うのは大変かもしれません。まずは、以下のステップを試してみてください。

1.  **設計だけをAIに頼む**
    コードを書かせず、「こういう機能を作りたいが、クラス設計はどうすべき？」とClaude（またはGPT-4o）に相談し、設計図だけを作らせます。
2.  **設計図を元に実装させる**
    その設計図を別のチャットウィンドウ（またはCopilot）に貼り付け、「この設計に従ってコードを書いて」と指示します。

この「役割分担」を意識するだけで、AIの出力精度が驚くほど向上するはずです。787コミットへの道は、この小さな分業から始まります。
