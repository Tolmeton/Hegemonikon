
# AIの出力を「省略なし」で受け取る方法 — 省略の暴力との戦い

「コードを修正しました。以下が修正版です」

```javascript
function heavyCalculation() {
  // ... (既存のコード)
  const result = newData + oldData;
  // ... (残りの処理)
  return result;
}
```

AIが出力したこのコードを見て、絶望したことはありませんか？
「既存のコード」や「残りの処理」の部分にこそ、重要な文脈や微妙な変更が含まれているかもしれないのに、AIは勝手に省略してしまう。これを私は「省略の暴力（Violence of Omission）」と呼んでいます。

特に、この省略されたコードをうっかりコピーして本番ファイルに上書きしてしまい、元々あったロジックを消失させてしまう事故は後を絶ちません。

AIはなぜ省略するのか。そして、どうすれば「完全なコード」を書かせることができるのか。今回はその対策を共有します。

## なぜAIは省略するのか

AI（LLM）にとって、長いコードを出力することは「コスト」です。計算リソースを消費し、応答時間が長くなり、途中でトークン制限に引っかかるリスクも増えます。
そのため、多くのモデルは「ユーザーがすでに知っている部分は省略するのが親切（かつ効率的）」だと学習しています。

しかし、コーディングにおいてこの「親切」は致命的です。私たちは差分だけを見たいのではなく、コピペで動く「完全なファイル」を求めていることが多いからです。

## 省略のパターンとリスク

| 省略タイプ | 特徴 | リスク |
| :--- | :--- | :--- |
| **コメントアウト省略** | `// ... rest of code` | コピペ時のコード消失。構造の欠落。 |
| **プレースホルダー** | `/* ここに実装 */` | 実装責任の放棄。ロジックの不整合。 |
| **部分提示** | 変更箇所のみを抜粋 | インデントやスコープ（`}`の数など）のミスを誘発。 |

## 対策：Proposal First (BC-5)

いきなりコードを書かせると、AIは勝手に省略を始めます。
これを防ぐための有効な手法が、Googleの研究者などが提唱するプロンプトエンジニアリング技術の一つ、「Proposal First（提案先行）」の応用です。

コードを書く前に、「何を変更し、どこを省略せずに書くべきか」を合意形成します。

## 読者が試せること：「省略禁止」の明示的指示

AIとのセッションで、以下のルールを「カスタム指示（Custom Instructions）」やプロンプトの冒頭に設定してください。

### 1. 鉄の掟プロンプト
```markdown
【出力ルール：省略の禁止】
コードを出力する際は、以下のルールを厳守してください。

1. **NO OMISSIONS**: `// ... existing code` や `// ... rest of logic` といった省略は一切禁止します。
2. **FULL FILE**: 修正を含むファイル全体を、最初から最後まで完全な状態で出力してください。
3. **COPY-PASTE READY**: 私が何も考えずにコピー＆ペーストして、そのまま動作する状態で提供してください。

ファイルが長すぎる場合のみ、関数単位で分割して出力することを許可します。
```

### 2. 「Apply」前の確認テクニック
IDE統合型AI（CursorやGitHub Copilotなど）を使う場合、AIが勝手にファイルを書き換える前に、以下のように釘を刺します。

```markdown
この変更を適用したいですが、あなたが省略をしないか心配です。
「私はコードを省略せず、ファイルの全行を出力します」と宣言してから、コード生成を開始してください。
```

人間に念押しするように、AIにも念押しをすることで、内部的な「省略フラグ」をへし折ることができます。

## まとめ

AIによる省略は、単なる手抜きではなく「情報の欠落」という重大なバグです。
特にリファクタリングや複雑なロジック修正の際、省略された部分にこそバグの種が潜んでいることがあります。

「10個中7個しか報告しない部下」を許容してはいけません。「10個すべてを机の上に並べてくれ」と明確に指示を出すこと。それが、AIによるコード破壊を防ぐ唯一の手段です。
