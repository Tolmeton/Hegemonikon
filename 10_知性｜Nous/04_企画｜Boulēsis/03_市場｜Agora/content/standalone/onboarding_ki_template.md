# Agora — KI (Knowledge Item) 穴埋めテンプレート

> ユーザーが **白紙から書かずに済む** 穴埋め式テンプレート。
> 30分セットアップの「10-20分」フェーズで使用。
> 消化元: `onboarding_strategy.md` §2

---

## 使い方

1. このテンプレートをコピーして `my-knowledge.md` として保存
2. `{ }` 内を自分の情報に書き換える
3. AI の Knowledge / Rules / Context に設置する
4. `/dok` で動作確認 → 成功したら完了

---

## テンプレート A: 業務プロファイル (Identity Stack)

```markdown
# {あなたの名前/チーム名} — AI 設定ファイル

## 私について
- 職種: {例: Web開発エンジニア / 翻訳者 / PMO}
- 経験年数: {例: 5年}
- 得意分野: {例: React, TypeScript, REST API設計}
- 苦手分野: {例: デザイン, インフラ}

## 仕事のスタイル
- 好みの言語: {日本語 / English / 両方}
- 出力の詳しさ: {簡潔 / 普通 / 詳細}
- コードスタイル: {例: ESLint Airbnb / PEP8 / 特になし}

## 現在のプロジェクト
- プロジェクト名: {例: 社内CRM刷新}
- 技術スタック: {例: Next.js + Supabase + Vercel}
- チーム規模: {例: 3名}
- 期限: {例: 2026年6月末}

## AIへの要望
- {例: コードを書くときは必ずテストも一緒に書いて}
- {例: 提案は3案出して、推奨に★をつけて}
- {例: 曖昧な指示のときは聞き返して}
```

## テンプレート B: 専門知識ファイル (Knowledge Item)

```markdown
# {トピック名} — 専門知識

## このファイルについて
{1-2行でこの知識の概要}

## 用語集
| 用語 | 定義 |
|:-----|:-----|
| {用語1} | {定義} |
| {用語2} | {定義} |
| {用語3} | {定義} |

## ルール / 規約
- {例: 顧客データは必ず匿名化してから処理する}
- {例: API レスポンスは 200ms 以内を目標}
- {例: コミットメッセージは Conventional Commits 形式}

## よくある質問と回答
### Q: {例: ステージング環境のURLは？}
A: {例: https://staging.example.com}

### Q: {例: デプロイ手順は？}
A: {例: main ブランチへの push で自動デプロイ}

## 参考資料
- {リンクやファイルパス}
```

## テンプレート C: ワークフロー指示書 (Skill 軽量版)

```markdown
# {作業名} — 作業手順

## いつ使うか
{例: 週次レポートを作成するとき}

## 手順
1. {例: Google Analytics から先週のPVデータを取得}
2. {例: 前週比を計算し、増減理由を3つ挙げる}
3. {例: Slack #reports チャンネルに投稿するフォーマットで出力}

## 出力フォーマット
{期待する出力形式を具体的に書く}

## 注意事項
- {例: 売上データは四捨五入せず端数まで含める}
- {例: 社外秘の数字は含めない}
```

## チェックリスト (30分セットアップ)

| 分 | やること | 完了 |
|:--|:--------|:-----|
| 0-10 | Skill をインストール (SKILL.md をコピー) | ☐ |
| 10-15 | テンプレートA (業務プロファイル) を記入 | ☐ |
| 15-20 | テンプレートB or C を1つ記入 | ☐ |
| 20-25 | AI に「自己紹介して」と聞いて設定確認 | ☐ |
| 25-30 | 実際の業務タスクを1つ試す | ☐ |

> **30分で「使える」まで到達することがゴール。**
> 完璧を目指さない。まず動かす。
