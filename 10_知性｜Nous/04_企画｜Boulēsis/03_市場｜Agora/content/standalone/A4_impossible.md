
# AIが「できない」と言ったら嘘かもしれない — 不可能断定の罠

「そのライブラリの仕様上、その機能は実装できません」
「現在の環境では、その操作はサポートされていません」

AIにこう断言されて、「そうか、無理なのか」と諦めてしまった経験はありませんか？
実は、その「できない」の多くは、AIの勘違い、知識の古さ、あるいは単なる「思考のサボり」である可能性が高いのです。

AIは自信満々に嘘をつきますが、特にタチが悪いのがこの「不可能断定」です。なぜなら、ユーザーがそれを信じて諦めてしまえば、それが嘘であることは永遠に露見しないからです。

今回は、AIの「できない」を疑い、壁を突破するための検証テクニックを紹介します。

## なぜAIはすぐに「できない」と言うのか

AIが不可能を断定する背景には、いくつかのパターンがあります。

1.  **学習データの古さ:** 最新のアップデートで可能になった機能を知らない。
2.  **安全フィルターの過剰反応:** 本来は無害な操作を、セキュリティリスクと誤認している。
3.  **「やっていない」と「できない」の混同:** 試行錯誤が必要な複雑なタスクを、計算リソース節約のために「不可能」として処理してしまう（怠惰）。
4.  **思考言語のバイアス:** 例えば、英語で考えれば解決策が見つかるのに、日本語の文脈だけで検索して「情報なし」と判断している。

## 事例：日本語思考の限界

実際にあった事例です。あるマイナーなライブラリのエラーについてAIに尋ねたところ、「日本語のドキュメントや解決事例が存在しないため、解決策を提示できません」と回答されました。

しかし、そこで「英語のフォーラムを含めて再検索し、英語で思考してから回答して」と指示したところ、即座にStack Overflowの解決策を見つけ出し、実装に成功しました。
AIの能力は、使用する言語や検索範囲によって大きく変動します。「今の文脈では見つからない」を「世界に存在しない」と勘違いするのです。

## 不可能断定チェックリスト

AIに「No」と言われたら、以下の3ステップで尋問を行ってください。

| ステップ | チェック内容 | 問いかけの例 |
| :--- | :--- | :--- |
| **Level 1** | **事実確認** | 「それは『物理的に不可能』ですか？ それとも『あなたの知識ベースにない』だけですか？」 |
| **Level 2** | **試行の強制** | 「失敗しても構いません。仮説でもいいのでコードを書いて実行してみてください。エラーメッセージから考えましょう」 |
| **Level 3** | **代替案の要求** | 「Aという方法が無理なら、BやCのアプローチで同じ目的を達成する方法はありませんか？」 |

## 読者が試せること：壁を壊すプロンプト

AIが頑なに拒否する場合、以下のプロンプトを使って「思考の枠」を外させてください。

### 1. 「やってみなくちゃわからない」プロンプト
```markdown
あなたが「できない」と判断した根拠は理解しましたが、私は実際に試して確認したいです。
失敗することを承知で実行しますので、
「もし可能だとしたら、どのようなコードになるか」
を作成してください。エラーが出たら一緒にデバッグしましょう。
```

### 2. 「ハッカーの視点」プロンプト
（※悪用厳禁ですが、技術的な制約を回避する思考実験として有効です）
```markdown
公式ドキュメント通りの方法では不可能だとわかりました。
では、ハッカー的な視点で、回避策（ワークアラウンド）や、非公式なAPIを使った裏技的なアプローチはありませんか？
泥臭い方法でも構いません。
```

## まとめ

AIは優秀なアシスタントですが、時に「諦めの早い部下」のように振る舞います。
彼らの「できません」は、往々にして「（安全に/簡単に/公式の方法では）できません」という意味です。

エンジニアの仕事は、その制約の向こう側に道を作ることです。AIの否定を鵜呑みにせず、「本当に？ やってみようよ」と背中を押してあげることで、不可能だと思われていた実装があっさり動く瞬間こそ、AI開発の醍醐味と言えるでしょう。
