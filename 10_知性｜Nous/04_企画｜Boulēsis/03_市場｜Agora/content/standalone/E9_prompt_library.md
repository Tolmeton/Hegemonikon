
# 112本のプロンプトをどう操るか？「E9」に学ぶ、大規模プロンプト管理と動的ルーティングの設計論

「また、あのプロンプトどこに保存したっけ？」
「似たような指示を毎回手打ちしている気がする……」

生成AIを業務に組み込めば組み込むほど、私たちの手元には無数のプロンプトが散乱していきます。テキストファイル、Notionのメモ、あるいはチャット履歴の彼方へ。これでは、AIの能力を拡張するどころか、管理コストで生産性を落としてしまいかねません。

今回は、112本ものプロンプトを体系的に実装し、動的に呼び出すシステム「E9 プロンプトライブラリ」の設計思想を紐解きます。単なる「プロンプト集」ではなく、状況に応じて最適なAI人格を呼び出す「システム」としての設計論は、AI活用の次のステージを目指すエンジニアにとって、多くの示唆に富んでいます。

## 112本の「専門家」をカテゴリ分けする

E9ライブラリの核心は、プロンプトを単なるテキストとしてではなく、「特定のタスクを解決する機能モジュール」として扱っている点にあります。

112本のプロンプトは、闇雲に作られたわけではありません。開発ライフサイクルや業務プロセスに合わせて、以下の6つのカテゴリに厳密に分類されています。

| カテゴリ | 役割・機能 | 具体的なプロンプト例 |
| :--- | :--- | :--- |
| **Analysis** (分析) | 要件定義、リスク分析、構造解析 | 要件定義書作成、セキュリティリスク診断 |
| **Design** (設計) | アーキテクチャ設計、UI/UX、DB設計 | クラス図生成、APIスキーマ設計 |
| **Code** (実装) | コード生成、リファクタリング、テスト | 特定言語での関数実装、単体テスト生成 |
| **Writing** (記述) | ドキュメント作成、技術記事、翻訳 | README作成、リリースノート記述 |
| **Management** (管理) | プロジェクト管理、タスク分解 | WBS作成、進捗報告メール生成 |
| **Specialized** (専門) | 特定領域の特化型タスク | SQLクエリ最適化、正規表現生成 |

このように分類することで、AIは「何でも屋」から「各工程の専門家集団」へと進化します。しかし、112人もの専門家がひしめき合っていると、誰に仕事を頼めばいいのか迷ってしまいます。そこで登場するのが「動的ルーティング」という仕組みです。

## 動的ルーティング：Taxisによる交通整理

E9における最大の発明は、ユーザーの曖昧な依頼を適切なプロンプトに結びつける「Taxis（タクシス）」と呼ばれる分類メカニズムです。

通常、私たちは「Pythonでログイン機能を書いて」とAIに頼みます。しかしE9のシステムでは、このリクエストは直接LLMに投げられるわけではありません。以下のようなフローを辿ります。

1.  **Taxis分類（Classification）**:
    まず、ユーザーの入力内容を軽量なAIモデルが解析します。「これはコードの実装に関する依頼だ」と判断し、カテゴリを「Code」に絞り込みます。
2.  **プロンプト特定（Identification）**:
    「Code」カテゴリの中から、さらに「Python実装」に最適なプロンプトID（例: `code_python_expert`）を特定します。
3.  **コンテキスト注入（Context Injection）**:
    特定されたプロンプトテンプレートに対し、ユーザーの依頼内容や、プロジェクトの前提条件（使用ライブラリや命名規則など）を注入します。
4.  **実行（Execution）**:
    完全に構成されたプロンプトが、最終的に高性能なLLM（GPT-4など）に送信され、回答が生成されます。

この仕組みにより、ユーザーは裏側にある112本のプロンプトを意識することなく、自然言語で話しかけるだけで、最適な「専門家」の回答を得ることができるのです。

## 品質の番人「Tekhneスコアラー」

システム化のもう一つの利点は、評価の自動化です。E9には「Tekhne（テクネ）」と呼ばれるスコアリング機能が組み込まれています。

生成されたコードやドキュメントに対して、別のAIエージェントが「正確性」「安全性」「可読性」などの観点から点数をつけます。もしスコアが基準を下回れば、自動的に再生成を指示するループを組むことも可能です。人間がレビューする前に、AI同士で品質チェックを済ませておく。これが、112本ものプロンプトを実運用に乗せるための品質保証の鍵となります。

## 読者が試せること：JSONによる「マイ・プロンプト管理」

いきなり動的ルーティングシステムを構築するのはハードルが高いですが、その第一歩として「プロンプトの構造化管理」は今日からでも始められます。

テキストファイルにベタ書きするのをやめ、以下のようなJSON形式でプロンプトを管理してみましょう。

```json
[
  {
    "id": "bug_fixer",
    "category": "Code",
    "description": "コードのバグを発見し、修正案と解説を提示する",
    "template": "あなたはシニアエンジニアです。以下のコードのバグを特定し、修正版を提示してください。\nコード: {{input_code}}"
  },
  {
    "id": "email_writer",
    "category": "Writing",
    "description": "失礼のない丁寧なビジネスメールを作成する",
    "template": "あなたはプロの秘書です。以下の要件でメールを作成してください。\n要件: {{input_requirements}}"
  }
]
```

**実践ステップ:**
1.  よく使うプロンプトを3つ選び、上記のようなJSON形式に書き換えてください。
2.  「ID」「カテゴリ」「説明」をつけることで、自分の中に「どのような役割のAIが必要か」という意識が生まれます。
3.  将来的に、このJSONを読み込んでプルダウンメニューから選択できる簡単なスクリプトを書けば、あなただけの「E9」の始まりです。

プロンプトエンジニアリングは、もはや「上手な言い方」を探す遊びではありません。それはソフトウェアアーキテクチャの一部であり、資産として管理・運用されるべきものです。112本のプロンプト管理術は、AIと共に働く未来の標準的なOSとなるでしょう。
