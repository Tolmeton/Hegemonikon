
# 長い会話が壊れる理由 — Context Rot（コンテキストの腐敗）

「最初はあんなに賢かったのに、なぜ急にポンコツになったんだ？」

ChatGPTやClaudeを使っていて、こう感じたことはありませんか？
セッションの序盤では、こちらの意図を完璧に汲み取り、素晴らしいコードを書いていたAIが、会話が続くにつれて指示を無視し始めたり、直したはずのバグを再発させたりする。

この現象には名前があります。「Context Rot（コンテキストの腐敗）」です。
AIの記憶容量（コンテキストウィンドウ）は増えていますが、注意を向ける能力（アテンション）は無限ではありません。会話が長引けば長引くほど、重要な文脈はノイズに埋もれ、腐敗していきます。

今回は、この「見えない劣化」を検知し、適切に対処するための防衛術を解説します。

## 会話の寿命を知る：30-40-50の法則

AIとのチャットラリー（往復回数）には、明確な「賞味期限」があります。モデルによって多少の差はありますが、経験則として以下の閾値を意識してください。

| メッセージ数 | 状態 | 症状 | 推奨アクション |
| :--- | :--- | :--- | :--- |
| **0 - 30** | **健全 (Green)** | 指示通りに動く。文脈を正確に把握している。 | そのまま継続 |
| **30 - 40** | **注意 (Yellow)** | 細かい指示を忘れ始める。「あれ、さっき言ったよね？」が増える。 | セーブポイントの作成準備 |
| **40 - 50** | **危険 (Red)** | 幻覚（ハルシネーション）が増える。コードの整合性が崩れる。 | **即時移行** |
| **50以上** | **崩壊 (Black)** | 同じミスを無限に繰り返す。論理的な対話が困難。 | 強制終了 |

多くのエンジニアは「50以上」の崩壊ゾーンまで粘ってしまい、結果として時間を浪費します。重要なのは「Yellow」の段階で気付き、手を打つことです。

## Context Rotの兆候を見逃さない

メッセージ数を数えていなくても、AIの挙動から腐敗のサインを読み取ることができます。

1.  **同じファイルを何度も開く（Read）**
    AIがすでに内容を知っているはずのファイルを、何度も`read_file`しようとする場合、それは「記憶があやふやになっている」証拠です。人間で言えば、会話中に何度もメモを見返すようなものです。

2.  **命名規則の崩壊**
    最初は `getUserData` だった関数名が、突然 `fetchUser` に変わったり、スネークケースとキャメルケースが混在し始めたりしたら、コンテキストの一貫性が失われています。

3.  **ループする提案**
    「その方法はさっき試してダメだった」という提案を再びしてくる場合、AIは過去の失敗履歴へのアクセス（アテンション）を失っています。

## 対策：Handoff（ハンドオフ）戦略

会話が腐敗し始めたら、そのセッションを治療しようとしてはいけません。新しいセッションへ「引越し」をするのが唯一の正解です。これを「Handoff（ハンドオフ）」と呼びます。

### 読者が試せること：Handoffプロンプト

会話が「注意（Yellow）」ゾーンに入ったと感じたら、AI自身に引継ぎ資料を作らせましょう。

**ステップ1：現状のまとめを作成させる**
以下のプロンプトを現在の（腐りかけた）セッションに投げます。

```markdown
会話が長くなり、コンテキストの精度が落ちてきました。
新しいセッションに移行するため、以下の情報をまとめた「引継ぎドキュメント」を作成してください。

1. **現在のゴール**: 私たちが最終的に何を達成しようとしているか。
2. **達成したこと**: 既に完了した実装と、確定した仕様。
3. **未解決の課題**: 現在直面しているエラーや、次に着手すべきタスク。
4. **重要な制約**: 命名規則、ライブラリのバージョン、禁止事項など。
5. **直近のコード状態**: 修正中のファイルのパスと、最新の変更内容。

これをマークダウン形式で出力してください。
```

**ステップ2：新セッションへ移行**
出力されたドキュメントをコピーし、新しいチャット（New Chat）の冒頭に貼り付けます。
「これは前任者からの引継ぎ資料です。これを元に開発を継続してください」と添えれば、フレッシュな脳みそを持ったAIが、正確な文脈で作業を引き継いでくれます。

## まとめ

「AIが急に馬鹿になった」のではありません。「AIが疲れた」のです。
Context Rotは避けられない物理現象のようなものです。これに抗うのではなく、適切なタイミングで「選手交代」を行うことが、高品質なAIコーディングを維持する秘訣です。

30往復を超えたら、そろそろ引越しの準備を始めましょう。
