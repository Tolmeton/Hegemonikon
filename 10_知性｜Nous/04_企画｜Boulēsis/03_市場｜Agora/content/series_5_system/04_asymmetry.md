
「良いものを作りたい」

そう思ってキーボードを叩き始めたのに、一行書いては消し、また書いては消し……。気づけば時間だけが過ぎ、画面は真っ白なまま。そんな経験はありませんか？

あるいは、チーム開発で「ブレーンストーミング」をしているはずなのに、誰かがアイデアを出すたびに「それはコスト的に無理」「前例がない」と即座に否定が入り、場が冷え切ってしまったことは？

これらはすべて、**「作ること（生成）」と「確かめること（審査）」を同時に行おうとした結果**として起こる不具合です。

シリーズ「HGK体系」の最終回となる今回は、この2つのプロセスの決定的な違い——**非対称性**について解説します。

古代ギリシャ語の概念を借りて、HGKでは生成を**Poiēsis（ポイエーシス）**、審査を**Dokimasia（ドキマシア）**と呼びます。この2つを明確に区別し、意図的に使い分けることこそが、HGK体系を「実践的なツール」に変える最後の鍵です。

### 1. Poiēsis（生成）：本質から具現へ

まず、「何かを生み出す」プロセスを見てみましょう。HGK体系において、生成（Poiēsis）は以下のフローで進行します。

$$O \rightarrow S \rightarrow H$$

*   **O (Origin / 本質):** まだ形にならない、純粋なアイデアや動機。
*   **S (Structure / 構造):** Oを支えるための論理や骨組み。
*   **H (History / 現象):** 実際に目に見える形、アウトプットされた結果。

生成とは、**「見えないものを、見えるようにする」**プロセスです。
ここでは、論理的な整合性よりも「勢い」や「展開」が優先されます。Oという水源からあふれ出たものが、Sという川底を通り、Hという海へ流れ込む。この流れは一方通行であり、不可逆的なエネルギーを持っています。

例えば、新しいアプリを開発する際、「こんな体験を届けたい（O）」という想いから、「機能要件（S）」を定義し、「コード（H）」に落とし込む流れがこれに当たります。

### 2. Dokimasia（審査）：条件から検証へ

一方、「それが正しいか確かめる」プロセスは、全く異なるベクトルを持ちます。

$$P \rightarrow K \rightarrow A$$

*   **P (Parameter / 前提条件):** 満たすべき基準、制約、ゴール。
*   **K (Kernel / 判定基準):** 照らし合わせるための知識、法、ロジック。
*   **A (Assessment / 判定結果):** 合格か、不合格か。修正指示。

審査とは、**「広がりすぎた可能性を、現実に合わせて絞り込む」**プロセスです。
生成が「足し算・掛け算」の世界なら、審査は「引き算・割り算」の世界です。ここでは創造性よりも、厳密さや批判的思考が求められます。

コードレビューや、論文の査読、あるいはコンパイルエラーのチェックなどがこれに該当します。

### 3. 決定的な「非対称性」

HGK体系が最も重要視するのは、**「PoiēsisとDokimasiaは対等ではない」**という点です。

多くの現場で起きる混乱は、この2つを「車の両輪」のように同時に回そうとすることで発生します。しかし、構造的にこの2つは混ぜるな危険、水と油の関係にあります。

以下の表で、その非対称性を整理しましょう。

| 特徴 | Poiēsis (生成) | Dokimasia (審査) |
| :--- | :--- | :--- |
| **HGKフロー** | $O \rightarrow S \rightarrow H$ | $P \rightarrow K \rightarrow A$ |
| **ベクトル** | 内から外へ（拡散） | 外から内へ（収束） |
| **目的** | 存在させること（0→1） | 正しくすること（1→100） |
| **許容度** | ノイズやエラーを許容する | エラーを許容しない |
| **マインド** | "Yes, and..."（いいね、さらに…） | "No, because..."（だめだ、なぜなら…） |
| **AI活用** | 温度パラメータ高め（創造的） | 温度パラメータ低め（論理的） |

**生成は「3シリーズの連鎖」**であり、**審査は「3シリーズの絞り込み」**です。
生成している最中に審査のロジックを持ち込むと、アクセルとブレーキを同時に踏むことになり、エンジン（あなたの創造性）が焼き付いてしまいます。

### 4. AIと日常における実装例

この非対称性を理解すると、AIの使い方も、日常の仕事の進め方も劇的に変わります。

#### AI実装：エージェントの分業
最近のLLM（大規模言語モデル）活用において、最も成果が出るのは「生成役」と「批評役」を分けた時です。

1.  **生成エージェント (Poiēsis):** 「とにかくアイデアを10個出せ。実現可能性は無視していい」と指示。
2.  **審査エージェント (Dokimasia):** 「上記のアイデアを、予算〇〇円以内という条件で評価し、上位3つを選抜せよ」と指示。

これを1人のAI（あるいは1回のプロンプト）でやらせようとすると、無難でつまらない回答しか返ってきません。役割を切り離すことで、初めて質の高いアウトプットが得られます。

#### 日常：コードとテスト、執筆と推敲
プログラマーにとっての「テスト駆動開発（TDD）」も、実はこの分離の高度な応用です。「テストを書く（審査基準Pを作る）」時間と、「コードを書く（生成Hを行う）」時間を明確に区切っています。

ライティングでも同様です。「ドラフト版（Poiēsis）」を書くときは、誤字脱字や論理の飛躍を気にせず、とにかく最後まで書き切る。翌日、別人のような冷徹な目で「推敲（Dokimasia）」を行う。これがプロの執筆フローです。

### 5. なぜ対等にしてはいけないのか

「走りながら考える」という言葉がありますが、HGK体系の視点では、これは**「生成しながら審査する」ことではありません。**
それは「短いサイクルで生成と審査を交互に繰り返す」ことです。

同時並行で行うと、以下のような弊害が起きます。

*   **アイデアの矮小化:** 生まれた瞬間の脆いアイデアが、審査の基準によって即座に潰される。
*   **審査の甘さ:** 「せっかく作ったから」というサンクコスト（生成側の情）が、審査の目を曇らせる。

生成の時間は「審査官」を部屋から追い出し、審査の時間は「クリエイター」を黙らせる。この**人格の切り替え（モード・スイッチング）**こそが、知的生産の質を高めます。

この非対称性を体感するために、次のアクションを試してみてください。

#### ① 「30分のサンドボックス」法
何か資料や文章を作る際、最初の30分は**「絶対に削除キー（Backspace）を使わない」**というルールを設けてください。
誤字があっても、論理が破綻しても、とにかく前に進む（生成する）。修正したくなったら、修正するのではなく「あとで直す」とメモ書きして次へ進みます。
30分経ったら休憩し、その後初めて「赤ペンを持つ審査官」として修正に入ってください。驚くほど筆が進むはずです。

#### ② 会議の「帽子」指定
会議の冒頭で宣言します。「今の時間はPoiēsis（拡散）の時間です。否定意見は禁止です」。
そして後半15分で「ここからはDokimasia（収束）の時間です。実現可能性の観点から厳しく見てください」と切り替えます。
参加者全員が今どちらのモードにいるかを共有するだけで、議論の空転がなくなります。
