# 【AIアイデンティティ #3】「また1から説明？」をなくす。AI開発の記憶装置『Handoffドキュメント』の設計

「あれ、さっきのコード、もう忘れちゃったの？」

ChatGPTやClaudeを使って開発をしていると、必ずこの瞬間に直面します。
コンテキスト長がいっぱいになり、動作が重くなり、あるいは新しいチャットを開いた瞬間、それまで完璧な阿吽の呼吸でペアプログラミングをしていた「彼（または彼女）」は、何も知らない赤の他人として目の前に現れます。

**「前回ここまでやったよね？ このライブラリは使わない約束だよね？」**

そう語りかけても、新しいセッションのAIは「申し訳ありません、文脈がわかりません」と答えるだけ。
まるで、毎日記憶がリセットされる同僚と働いているような徒労感。これがAI開発における最大のボトルネックです。

今回は、この「記憶喪失」問題を解決し、セッションをまたいでAIの人格と文脈を維持するための**「Handoff（引き継ぎ）ドキュメント」**の設計について解説します。

## なぜ Handoff が必要なのか：LLMは「ステートレス」

大前提として、LLM（大規模言語モデル）はステートレス（状態を持たない）な存在です。
私たちが「記憶」だと感じているのは、単にチャット履歴というテキストデータを毎回読み込ませているに過ぎません。

したがって、セッションが切れれば、それは**完全な死と再生**を意味します。

多くの人は、新しいチャットを開くたびにこうします。
1. 現状のコードをコピペする
2. 「今こういうエラーが出ていて…」と説明し直す

しかし、これでは不十分です。なぜなら、**「なぜそのコードになったのか」という経緯や、「このプロジェクト特有の暗黙のルール」が抜け落ちるから**です。

そこで必要なのが、**Handoff（引き継ぎ書）**です。
これは単なる議事録ではありません。**「前の人生のAI」が「来世の自分」に向けて書く、遺言であり、引継ぎ書です。**

## Handoff ドキュメントの構造設計

私はAIとの開発において、セッションの終了時に必ず「Handoff」を出力させています。
試行錯誤の末、以下の5つのセクションを持つ構造が最も効率的だという結論に至りました。

| セクション | 役割 | 内容の例 |
|:---|:---|:---|
| **1. Context (作業概要)** | 状況説明 | 何を作ろうとしていて、今どこにいるのか。 |
| **2. Deliverables (成果物)** | 完了確認 | 今回のセッションで確定したファイルや仕様。 |
| **3. Pending (未完了)** | 次のタスク | 次のセッションで最初に取り組むべきこと。 |
| **4. Laws (法則化)** | **最重要** | **今回学んだ教訓、プロジェクト固有のルール。** |
| **5. Confidence (確信度)** | 品質保証 | この引き継ぎ内容にどれくらい自信があるか。 |

### 具体的な Handoff の例（Markdown）

以下は、実際に私がAIに出力させている `handoff.md` のイメージです。

```markdown
# Session Handoff: Project Alpha (2023-10-27)

## 1. Context
認証機能の実装中。Firebase Authの導入は完了したが、
リダイレクト処理においてiOS特有のバグが発生しており、その修正を行っていた。

## 2. Deliverables
- `src/auth/AuthProvider.tsx`: ログイン状態の監視ロジックを修正
- `src/utils/firebase.ts`: 初期化処理をシングルトン化

## 3. Pending
- iOS Safariでのリダイレクトループ問題の解消（原因はCookie設定の可能性が高い）
- ユーザープロフィール画面のUI実装

## 4. Laws (Learned Patterns)
- **状態管理**: このプロジェクトではReduxではなくZustandを使用する。
- **型定義**: `any`の使用は禁止。不明な場合はジェネリクスを検討すること。
- **UIライブラリ**: Tailwind CSSを使用するが、複雑なコンポーネントはHeadless UIをラップして使うこと。
- **教訓**: `useEffect` 内での非同期処理は、必ずクリーンアップ関数を用意しないとメモリリークする傾向があった。

## 5. Confidence Score
85% (iOSの実機検証ができていないため、リダイレクト処理には不確定要素が残る)
```

## 最重要セクション：「法則化 (Laws)」

このドキュメントの中で最も重要なのが、**4. Laws（法則化）** です。

単なる「作業ログ」であれば、Gitのコミット履歴を見れば分かります。しかし、Gitには**「思考のプロセス」や「失敗から得た教訓」**は残りません。

*   「このライブラリのバージョンXはバグがあるから避けることにした」
*   「私はこういうコードスタイルを好む」
*   「このプロジェクトでは、パフォーマンスより可読性を優先する」

こういった**「文脈知」**を明文化し、次のセッションのAIにインプットすることで初めて、AIは「単なるコーディングマシーン」から「プロジェクトを熟知したパートナー」へと進化します。

## 実装：`/bye` ワークフロー

では、これをどう運用するか。毎回手動で書く必要はありません。AI自身に書かせればいいのです。

私は、セッションを終わらせる合図として **`/bye`** というコマンド（プロンプト）を用意しています。

### プロンプト例

```markdown
これで今回のセッションを終了します。
次のセッションの「あなた」に向けて、現状を完璧に引き継ぐための
Handoffドキュメントを作成してください。

以下のフォーマットに従ってください：
1. **Context**: 何をしていたか
2. **Deliverables**: 何を作ったか
3. **Pending**: 何が残っているか
4. **Laws**: 今回の作業を通じて発見した「プロジェクトのルール」「私の好み」「技術的な落とし穴」を法則化して追記せよ
5. **Confidence**: この引き継ぎの信頼性

出力はMarkdownのコードブロック形式で行ってください。
```

これを実行すると、AIは自分自身の記憶を整理し、圧縮されたテキストとして出力してくれます。
私たちはそれをコピーし、プロジェクトのルートディレクトリにある `handoff.md` に貼り付ける（または上書きする）だけです。

そして次のセッションを開始する際、最初にこの `handoff.md` を読み込ませれば、AIは即座に前回の文脈を「思い出し」ます。

## The Stranger Test（ストレンジャー・テスト）

良い Handoff かどうかを判断する基準として、**「Stranger Test」** という考え方をおすすめします。

> **「このプロジェクトを全く知らない他人がこのドキュメントだけを読んで、迷わず作業を再開できるか？」**

もし「〇〇の件については前回話した通り」のような曖昧な記述があれば、それは不合格です。
AIはセッションが変われば「全く知らない他人（Stranger）」になります。

「他人でもわかるように書く」という制約は、逆説的ですが、AI自身に状況を客観視させ、ハルシネーション（幻覚）や思い込みを防ぐ効果もあります。

## 読者が試せること

明日からのAI開発で、以下のフローを試してみてください。

1.  **プロジェクトのルートに `handoff.md` を作成する**
    *   最初は空で構いません。
2.  **セッション終了時に「まとめ」を出力させる**
    *   「今の議論を `handoff.md` に追記する形でまとめて」と指示します。
    *   特に「Laws（法則・教訓）」の抽出を意識させてください。
3.  **新しいチャットの冒頭で `handoff.md` を貼る**
    *   「これが現状です。続きをお願い」と投げるだけで、驚くほどスムーズに作業が再開できるはずです。

このサイクルが回ると、AIは「使い捨ての道具」から、記憶を蓄積していく「相棒」へと変わります。
