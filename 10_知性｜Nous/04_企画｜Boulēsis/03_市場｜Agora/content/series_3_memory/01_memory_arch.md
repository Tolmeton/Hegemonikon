# AIに「昨日の続き」を覚えさせる。4層の記憶アーキテクチャ設計論【記憶スタック #1】

「昨日はあんなに良い議論ができたのに、今日のあなたは私のことを何も知らない」

ChatGPTやClaudeを使っていて、そんな「寂しさ」を感じたことはないでしょうか。
ブラウザのタブを閉じた瞬間、あるいは「New Chat」を押した瞬間、AIはリセットされます。毎回ゼロからコンテキストを説明し、同じ指示を繰り返す。これはまるで、記憶喪失のパートナーと毎日初対面の挨拶を交わしているようなものです。

AIそのものはステートレス（状態を持たない）な存在ですが、**外部に「記憶の仕組み」を設計してあげることで、AIはセッションを超えて成長し始めます。**

本連載「記憶スタック」では、私が実際に開発・運用しているAIの記憶システムについて解説します。第1回は、その全体像となる**「4層アーキテクチャ」**についてです。

## なぜ「記憶」を設計するのか

人間の記憶が「短期記憶」や「長期記憶」、「エピソード記憶」などに分かれているように、AIの記憶も単一のテキストファイル（モノリス）にすべきではありません。

すべてのログを1つのファイルに詰め込むと、トークン数が爆発し、AIの処理能力（コンテキストウィンドウ）を圧迫します。また、古い日記と最新の技術仕様書が混在していれば、AIは情報の重要度を判断できません。

そこで私は、AIの記憶を用途と性質に応じて以下の4つの層に分離しました。

## 4層の記憶アーキテクチャ

私はこの4層に、それぞれの役割を表すギリシャ語の名前をつけています。

| 層の名前 | 意味 | 役割・データ | 人間の記憶での例 | データ形式 |
| :--- | :--- | :--- | :--- | :--- |
| **Sophia** | 知恵 | **永続的な知識・教訓**<br>プロジェクトのルール、成功/失敗パターン | **長期記憶**<br>（価値観、教訓） | Markdown |
| **Kairos** | 時機 | **セッション引き継ぎ**<br>現在のタスク状態、次のアクション | **ワーキングメモリ**<br>（机上の付箋） | YAML |
| **Chronos** | 時間 | **作業ログ**<br>時系列の活動記録、Gitコミット | **日記・記録**<br>（過去の事実） | Git Log / Text |
| **Gnōsis** | 知識 | **外部知識基盤**<br>論文、ドキュメント（大量データ） | **図書館**<br>（外部知識） | Vector DB |

それぞれの層について、具体的に見ていきましょう。

### 1. Sophia（ソフィア）：変わらない「知恵」
これは、AIにとっての「憲法」や「教科書」にあたる部分です。
プロジェクト固有のコーディング規約や、「前回このライブラリを使ってエラーが出たから、次からはこっちを使う」といった**教訓（Lessons Learned）**を蓄積します。

- **形式:** 人間もAIも読みやすいMarkdown。
- **運用:** プロジェクト終了時や、重要な決定をした時に更新します。

### 2. Kairos（カイロス）：文脈をつなぐ「バトン」
ここが最も特徴的な層です。「Kairos」はギリシャ語で「一瞬の時（チャンス）」を意味します。
AIとの会話が終わる際、**「次はどこから再開すべきか」という申し送り事項（Handoff）**をここに記録させます。

AIは次回の起動時、まずこのファイルを読み込むことで、「昨日の続き」からスムーズに作業に入れます。構造化データとして扱いやすいYAML形式を採用しています。

```yaml
# kairos_handoff.yaml の例
current_status:
  phase: "実装フェーズ"
  last_action: "認証機能のJWTトークン発行処理を作成中"
pending_tasks:
  - "トークンの有効期限切れ処理の実装"
  - "単体テストの作成"
context:
  - "ライブラリはAuth0ではなくFirebase Authを使用することに決定済み"
next_prompt_suggestion: "トークンリフレッシュのロジックについて提案してください"
```

### 3. Chronos（クロノス）：事実としての「ログ」
「いつ、何をしたか」という純粋な時系列データです。
開発プロジェクトであれば `git log` がこれに該当しますし、チャットの全履歴もここに含まれます。
この層は膨大になるため、基本的には**検索対象（アーカイブ）**として扱います。常にAIに読ませるわけではありません。

### 4. Gnōsis（グノーシス）：膨大な「外部知識」
私の環境では、596本の技術論文やPDFドキュメントがここに格納されています。
これらはテキストとして全量読ませることは不可能なため、**ベクトルデータベース**に格納し、必要な時に必要な部分だけを取り出す「RAG（Retrieval-Augmented Generation）」の仕組みを利用します。

## 技術スタック：どうやって統合するか

これら4つの層をバラバラに管理していては意味がありません。私は `PKS CLI` (Personal Knowledge System) という自作ツールでこれらを統合しています。

### セマンティック検索の実装
特に「Gnōsis（知識）」や「Sophia（知恵）」の検索には、キーワード一致ではなく意味ベースの検索（セマンティック検索）が不可欠です。

- **Embedding Model:** `BGE-small-en-v1.5` (軽量で高性能)
- **Vector Search:** `HNSWlib` (高速な近傍探索ライブラリ)

これらをローカル環境で動作させ、AIが質問を受けた瞬間に、関連する「過去の知恵」や「論文の知識」を自動的にコンテキストに注入する仕組みを構築しています。

## 読者が今日から試せること

いきなりベクトルDBを構築するのはハードルが高いかもしれません。しかし、「Kairos（引き継ぎ）」の概念は、今すぐ**Claude Projects**や**ChatGPTのメモリ機能**で実践できます。

おすすめは、プロジェクトごとに**「`memory.md`（または`handoff.md`）」というファイルを1つ作り、会話の最後にAIに更新させる**ことです。

### 実践ステップ
1.  プロジェクトのファイルに `memory.md` を作成する。
2.  会話の最後に、以下のプロンプトを投げる。
    > 「ここまでの議論の内容、決定事項、次にやるべきことを整理して、memory.md を上書きする内容を出力してください」
3.  出力された内容をコピペしてファイルを更新する。
4.  次の会話の冒頭でファイルをアップロード（または参照）する。

たったこれだけで、AIは驚くほど「文脈の濃い」パートナーに変わります。
