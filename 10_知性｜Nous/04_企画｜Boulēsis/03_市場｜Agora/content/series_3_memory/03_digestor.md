# 【記憶スタック #03】積読を知識に変える。596本の論文を自動消化するパイプライン「Digestor」

「あの新しい論文、もう読んだ？」

エンジニアや研究者にとって、この言葉ほど心拍数を上げるものはありません。
arXivには毎日数え切れないほどの論文が投稿され、X（旧Twitter）のタイムラインは「画期的な手法が登場！」という言葉で溢れかえっています。すべてに目を通したいけれど、物理的に時間が足りない。結果、ブラウザのタブと「後で読む」リストだけが無限に増殖していく——。

そんな「情報の洪水」に溺れかけた私が構築したのが、**論文自動消化パイプライン「Digestor」**です。

これは、私が寝ている間に世界中の論文を巡回し、必要なものだけを選別し、要約して私の「第二の脳」に格納してくれるシステムです。すでに596本以上の論文がこのシステムによって処理され、いつでも引き出せる知識としてストックされています。

今回は、この自動化パイプラインの裏側と、LLMを用いた構造化要約の具体的な実装について解説します。

## Digestorの全体像：情報の「胃袋」を作る

「Digestor（消化器官）」と名付けたこのシステムは、単に論文をダウンロードするだけのスクリプトではありません。人間が食事をして栄養を吸収するように、情報を**収集し、選別し、分解（要約）し、保存**します。

システムは以下の4つのステップで構成されています。

1.  **収集 (Collect):** 特定のキーワードや分野に基づいて論文メタデータを取得
2.  **選別 (Filter):** 読む価値があるか、自分の興味に合致するかを判定
3.  **要約 (Digest):** LLMを使って構造化された要約を作成
4.  **保存 (Index):** ベクトル化してデータベース（Gnōsis DB）に格納

この一連の流れを、ノーコードツールである **n8n** と Pythonスクリプト、そして **Claude (LLM)** を組み合わせて完全自動化しています。

## 1. 収集：Semantic Scholar APIの活用

情報の入り口として、私は主に **Semantic Scholar API** と **arXiv API** を併用しています。特にSemantic Scholarは、引用グラフ（どの論文がどの論文を引用しているか）の情報が豊富で、「影響力のある論文」を探すのに適しています。

n8nのCRONノードをトリガーにして、毎日深夜にAPIを叩きます。

```python
# 概念的な取得ロジック（Python）
import requests

def fetch_papers(query="Large Language Models", year="2024-2025"):
    url = "https://api.semanticscholar.org/graph/v1/paper/search"
    params = {
        "query": query,
        "year": year,
        "fields": "title,abstract,authors,citationCount,url",
        "limit": 100
    }
    response = requests.get(url, params=params)
    return response.json()['data']
```

## 2. 選別：LLMに読ませる前の「門番」

すべての論文をLLMに読ませていると、APIコストが膨大になりますし、ノイズも増えます。そこで、メタデータレベルでのフィルタリングを行います。

*   **トピック照合:** タイトルやアブストラクトに特定のキーワードが含まれているか。
*   **引用数フィルタ:** （過去の論文の場合）一定以上の引用数があるか。
*   **年度フィルタ:** 直近の研究であるか。

このフィルタを通過した論文だけが、「要約待ちリスト（Queue）」に追加されます。

## 3. 要約：LLMによる構造化（ここが重要）

ここがDigestorの心臓部です。単に「要約して」と頼むと、LLMはふわっとした文章を返してしまいがちです。後で検索・活用しやすくするために、**出力フォーマットを厳密に指定（構造化）**します。

私は、ローカルの `incoming/` フォルダにPDFが落ちてくると、スクリプトがそれを検知し、テキスト抽出して Claude API に投げる仕組みを作っています（これを `/eat` コマンドと呼んでいます）。

### 実際に使用しているプロンプト（抜粋）

```markdown
あなたは熟練したリサーチャーです。以下の論文を読み、次のJSONフォーマットで出力してください。

{
  "title_ja": "日本語タイトル",
  "core_problem": "この論文が解決しようとしている具体的な課題（1文で）",
  "methodology": "提案手法の革新的な点（技術的なキーワードを含めて）",
  "results": "定量的・定性的な主な結果",
  "impact": "この研究が分野に与える影響や、自分のプロジェクトへの応用可能性",
  "tags": ["タグ1", "タグ2"]
}

注意：
- 専門用語は無理に翻訳せず、英語のままにすること。
- 断定的な表現を使うこと。
```

このようにJSONで出力させることで、後述するデータベースへの登録がプログラム的に処理できるようになります。

## 4. 保存と対話：Gnōsis DBへの統合

生成されたJSONデータは、Markdownファイルに変換され、前回の記事で紹介した「Gnōsis DB」に格納されます。同時に、テキスト全体をEmbedding（ベクトル化）し、ベクトル検索が可能な状態にします。

ここまで準備ができると、何が起こるでしょうか？

私は自分のナレッジベースに対して、チャット形式で質問ができるようになります。これを **gnosis-dialog Skill** として実装しています。

**私:** 「最近のLLMのハルシネーション対策で、面白そうなアプローチはある？」
**Digestor (AI):** 「はい、データベース内にいくつか関連する論文があります。特に『Chain-of-Verification』という手法（2023年）では、モデル自身に検証質問を生成させることで精度を向上させています。また、最新の『Self-RAG』では……」

このように、**「自分が読んでいない論文」であっても、自分のデータベースにあれば、AIが代理で内容を把握し、文脈に合わせて教えてくれる**のです。これが「記憶スタック」の威力です。

## 実装の裏側：n8nとファイルシステム

全体のオーケストレーションには **n8n** を使用しています。

1.  **n8n (Trigger):** 毎日深夜に実行。
2.  **API Call:** 新着論文リストを取得。
3.  **Filter:** 条件に合わないものを破棄。
4.  **File Operation:** 合格した論文のPDFをダウンロードし、ローカルサーバーの `incoming/` フォルダへ保存。
5.  **Watcher Script:** フォルダを監視するPythonスクリプトが、PDFを検知して Claude API へ送信 (`/eat` 処理)。
6.  **Move:** 処理が終わったPDFは `processed/` フォルダへ移動。
7.  **Save:** 要約JSONをMarkdown化してObsidianの保管庫（Gnōsis DB）へ書き込み。

この「フォルダに置くと処理される」という非同期な仕組みにすることで、手動で拾ってきたPDFも `incoming/` に放り込むだけで同じフローに乗せることができます。

| ステップ | 使用ツール | 役割 |
| :--- | :--- | :--- |
| **収集** | Semantic Scholar API | 論文の探索 |
| **制御** | n8n | ワークフロー管理 |
| **読解** | Claude 3.5 Sonnet | テキスト抽出・要約 |
| **保管** | Obsidian / Vector DB | ナレッジベース化 |

## 読者が試せること：まずは「RSS × AI要約」から

いきなりフルスクラッチでパイプラインを組むのは大変です。まずは以下のステップで「情報の自動消化」を体験してみてください。

1.  **arXivのRSSを取得する:**
    自分の興味ある分野（例: `cs.CL` - 計算言語学）のRSSフィードURLを取得します。
2.  **ZapierやIFTTTを使う:**
    「RSSに新しいアイテムが追加されたら」→「ChatGPT（またはClaude）に送る」というフローを作ります。
3.  **プロンプトを工夫する:**
    単に送るだけでなく、「この論文の『新規性』を3行でまとめてSlackに通知して」という指示を挟みます。

これだけで、「タイトルだけ見てスルーしていた論文」の中身が、向こうから飛び込んでくるようになります。

Digestorを運用し始めてから、「読み逃し」の不安が消えました。596本の論文すべてを精読したわけではありませんが、**「必要になったら、いつでもAI経由で詳細を引き出せる」**という状態が、精神的な余裕と知的生産性を劇的に向上させてくれました。
