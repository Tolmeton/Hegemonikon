# 【環境設計 #3】AIのルールを「進化」させる技術 — 違反を資産に変えるサイクル

「さっき教えたばかりなのに、また同じ間違いをした……」
「プロンプトをどれだけ工夫しても、なぜか指示をすり抜けられる」

AIと協働して開発や執筆を行っていると、こうした徒労感に襲われることはありませんか？
完璧な指示書（システムプロンプト）を作ろうと意気込み、長大なルールを書いても、AIはふとした瞬間にそれを破ります。そして私たちは、「AIはまだ賢くないから仕方がない」と諦めるか、その場しのぎの修正をしてしまいがちです。

しかし、**「完璧なルール」など最初から存在しない**としたらどうでしょう？

環境設計シリーズ第3回のテーマは、**ルールの「進化」**です。
AIのミス（違反）を単なる失敗として処理せず、それを分析し、ルール自体をアップデートしていく「エンジニアリング」としての手法を解説します。

## ルールは「固定」するものではなく「育てる」もの

多くの人は、システムプロンプトを「一度書いたら終わりの契約書」のように扱います。しかし、環境設計における正しいアプローチは、それを**「生きている法律」**として扱うことです。

法律が時代の変化や新たな判例によって改正されるように、AIへの指示も「違反」という判例を元に修正され続けなければなりません。

私が実践している**ルールの進化サイクル**は以下の通りです。

1.  **違反発生**: AIが期待と異なる挙動をする
2.  **ログ記録**: その事実を隠さず記録する
3.  **原因分析**: 「なぜ」その指示が守られなかったかを特定する
4.  **ルール改善**: 指示の曖昧さを排除し、バージョンを上げる
5.  **再テスト**: 同じ状況で正しく動作するか検証する

このサイクルを回すために不可欠なツールが、**`violations.md`（違反ログ）**です。

## 違反を資産に変える `violations.md`

プロジェクトのルートディレクトリに、`violations.md` というファイルを一つ作ってください。ここには、AIがルールを破ったすべての事例を記録します。

これはAIを叱責するためのものではなく、**「システム（指示書）のバグ」を追跡するためのイシュー管理**です。

### 実際の記述フォーマット

```markdown
## 2023-10-25: 確信度の欠如 (BC-6違反)

- **状況**: ユーザーがライブラリの仕様について質問した際、AIが古いAPIを自信満々に提案し、エラーが発生した。
- **原因**: ルール BC-6 では「不確実な場合は明示せよ」としていたが、AIはハルシネーション（幻覚）により「確実だ」と誤認していた。
- **対策**: BC-6 を改訂。「確信度」という主観的な言葉を廃止し、情報源（SOURCE）と推測（TAINT）の明示的なタグ付けを義務化する。
- **修正コミット**: v2.1 -> v2.2
```

このように記録することで、「AIがよく間違えるパターン」が可視化されます。そして、その対策がルールに反映されているかどうかが、環境の品質を決定づけます。

## 実践事例：ルール「BC-6」の進化

具体的に、一つのルールがどのように進化していくかを見てみましょう。
ここでは、私が実際に運用している **BC-6（情報の確実性担保）** というルールの変遷を紹介します。

### v1.0：ナイーブな指示

最初は、誰もが書くようなシンプルな指示でした。

```markdown
[BC-6]
回答に自信がない場合は、その旨を正直に伝えてください。
嘘や推測を事実のように語ってはいけません。
```

**結果**: AIは「自信があります！」と言いながら嘘をつきました。LLMにとって「自信」は主観的なパラメータであり、ハルシネーション（もっともらしい嘘）が発生している時ほど、AIの「自信」は高まる傾向にあったからです。

### v2.0：構造化と追跡（TAINT/SOURCE）

`violations.md` での分析を経て、主観的な「自信」を排除し、情報の出処を問う形式へ変更しました。

```markdown
[BC-6] 情報の信頼性明示
情報の出処に基づき、以下のタグを付与すること。

- [SOURCE]: 公式ドキュメントやコードベースに明記されている事実
- [TAINT]: 学習データに基づく推測、または未検証の提案

回答には必ずどちらかの根拠を含めること。
```

**結果**: 劇的に改善しました。AIは「これは推測（TAINT）ですが……」と前置きするようになりました。しかし、まだ抜け穴がありました。「できないこと」を「できる」と言い張るケースです。

### v2.3：否定の検証基準を追加

さらに運用を続け、最終的にルールはここまで具体的になりました。

```markdown
[BC-6] 情報の信頼性と検証 (v2.3)

1. **TAINT/SOURCE システム**
   - [SOURCE]: コンテキスト内に存在するファイル、URLからの引用。
   - [TAINT]: 内部知識のみに基づく回答。これには「〜と思われる」「一般的手法では」等の表現を伴うこと。

2. **不可能断定チェック (Impossible Check)**
   - ユーザーの要望が技術的に不可能、または現在のコンテキストで実行不可能な場合、即座にそれを指摘すること。
   - 無理に回答を生成しようとして [TAINT] 情報を [SOURCE] のように振る舞わせることを固く禁ずる。
```

このように、**「抽象的な願い」から「具体的な判定ロジック」へとルールを進化させる**ことこそが、環境設計の真髄です。

## フィードバックループを「正」に回す

このプロセスを行うか否かで、AIとの関係性は大きく変わります。

| 特徴 | 負のループ（放置） | 正のループ（進化） |
| :--- | :--- | :--- |
| **違反への対応** | 「また間違えた」とAIを責める | 「指示が悪かった」とルールを直す |
| **ルールの状態** | 固定・形骸化 | 常に更新・最適化 |
| **AIの挙動** | 同じミスを繰り返す | 徐々に賢くなる |
| **ユーザーの心理** | 信頼できないため監視コスト増 | 信頼できるため裁量を拡大できる |

違反を隠蔽したり見逃したりすることは「負のループ」への入り口です。逆に、違反を直視し、それをルールの修正につなげれば、システムは雪だるま式に賢くなっていきます。これを**正のフィードバックループ**と呼びます。

## 今すぐ読者が試せること

この記事を読み終えたら、以下の3ステップを試してみてください。

1.  **`violations.md` を作成する**
    プロジェクトのルートにファイルを作り、直近でAIが犯したミスを1つ書き出してください。
2.  **「精神論」を「仕組み」に変換する**
    そのミスを防ぐために、「注意してね」という精神論ではなく、「〇〇の場合は××せよ」という条件分岐（If-Then）のルールを考えてみてください。
3.  **システムプロンプトを更新する**
    考えたルールを既存のプロンプトに追加・修正し、バージョン番号を上げてみましょう。

「AIが指示を聞かない」と嘆く前に、私たちが「指示の出し方」を進化させる番です。
ルールが洗練されればされるほど、AIはあなたの最強のパートナーへと近づいていきます。
