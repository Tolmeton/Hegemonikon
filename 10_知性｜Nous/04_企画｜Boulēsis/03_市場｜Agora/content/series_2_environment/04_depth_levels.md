# AIとの対話に「深度」を設定する — 全力と省エネの使い分け【環境設計 Vol.4】

「ただのタイポ（誤字）修正をお願いしただけなのに、AIが長考し始めて、修正の背景や言語学的意味まで解説し始めた」

あるいは逆に、「複雑なシステム設計の相談をしているのに、AIが即答してきて、浅い回答しか返ってこなかった」

皆さんは、最近の高性能なAIモデル（OpenAIのo1/o3やClaude 3.5 Sonnetなど）を使っていて、このような経験をしたことはないでしょうか？

これらはすべて、**「タスクの難易度」と「AIの思考深度」のミスマッチ**が原因です。

シリーズ「環境設計」の最終回となる今回は、AIとの対話における**「深度（Depth）」のコントロール**についてお話しします。すべての質問にフルパワーで答えさせるのは、時間もコストも非効率です。AIのギアを適切に切り替え、快適な開発環境を構築する方法を解説します。

---

## なぜ「深度」の制御が必要なのか

AI、特に最近の「推論モデル（Thinking Models）」は、回答を出力する前に内部で思考（Chain of Thought）を行います。これは素晴らしい機能ですが、**常に全力で思考することが正解とは限りません。**

車に例えてみましょう。
どんなに高性能なスポーツカーでも、コンビニの駐車場でバックするときにアクセルをベタ踏み（フルパワー）にはしません。逆に、急な坂道を登るときにトップギア（省エネ走行）では登りきれません。

*   **1速（L3 Deep）**: パワーが必要な難問。燃費（時間・トークン）は悪いが、確実に登れる。
*   **5速（L0 Bypass）**: 高速巡航や単純作業。燃費が良く、レスポンスが速い。

AIとの対話も同じです。タスクに応じて「ギア」を使い分けることで、待ち時間を減らし、回答の精度を最適化できます。

## 定義すべき「4つの深度レベル」

私の環境では、AIへの指示を以下の4つのレベル（L0〜L3）に分類し、プロンプトやカスタムコマンドで制御しています。

| レベル | 名称 | 目的 | 内部思考 (CoT) | 具体的なユースケース |
| :--- | :--- | :--- | :--- | :--- |
| **L0** | **Bypass** | 単純応答 | **なし** | `ls`コマンドの結果確認、タイポ修正、単純な構文エラーの指摘 |
| **L1** | **Quick** | 実装・軽微判断 | 短い計画→即実行 | 正規表現の作成、短い関数の単体テスト作成、ライブラリの使い方の確認 |
| **L2** | **Standard** | 設計・分析 | **構造化された思考** | リファクタリング、機能追加の設計、エラー原因の特定（デフォルト） |
| **L3** | **Deep** | 本質問題 | **全展開 + 多角検証** | アーキテクチャ選定、再現性の低いバグ調査、セキュリティ要件の策定 |

### L0: Bypass（思考停止モード）
「とにかく早く結果だけくれ」という場面です。AIが気を利かせて「このコマンドの意味は〜」と解説し始めるのを防ぎます。
*   **指示イメージ:** 「解説不要。コードのみ出力せよ。」

### L1: Quick（瞬発モード）
少しだけ考えてほしいが、長考は不要な場面です。
*   **指示イメージ:** 「簡潔に方針を立ててから実装せよ。代替案の比較は不要。」

### L2: Standard（通常モード）
デフォルトの設定です。要件を整理し、エッジケースを考慮してから回答させます。
*   **指示イメージ:** 「タスクを分析し、ステップバイステップで解決策を提示せよ。」

### L3: Deep（深海モード）
人間の手に負えない、あるいは非常に抽象度の高い問題を扱う場合です。ここでは「Thinking Budget（思考予算）」を最大まで引き上げます。
*   **指示イメージ:** 「即答禁止。まず問題の前提条件を疑い、複数の視点から検証を行え。内部思考を最大限に活用せよ。」

## 実装例：Cursor / Windsurf での制御

概念は分かっても、毎回「これはL1で…」と手打ちするのは面倒です。
そこで、開発ツールの「ルール機能」を使って、この深度制御をコマンド化します。

以下は、Cursorの `.cursorrules`（またはプロジェクト固有のプロンプト設定）に記述する例です。

```markdown
# Depth Control Rules

ユーザーが以下のコマンドプレフィックスを使用した場合は、指定された深度レベルで応答してください。

## /noe- (Level 1: Quick)
- **Thinking**: Minimal. Skip philosophical analysis.
- **Output**: Code or direct answer only.
- **Use case**: Syntax fix, simple regex, one-line changes.

## /noe (Level 2: Standard) - DEFAULT
- **Thinking**: Structured. Analyze requirements -> Plan -> Execute.
- **Output**: Explain the "Why" briefly, then show the code.
- **Use case**: Refactoring, new function implementation.

## /noe+ (Level 3: Deep)
- **Thinking**: Extended. Use full Chain of Thought.
- **Process**:
  1. Restate the problem.
  2. List edge cases and potential risks.
  3. Explore multiple architectural approaches.
  4. Select the best one and implement.
- **Use case**: Complex bug debugging, System architecture design.
```

※ `noe` は "Noetic"（知的な）や "No-Effort" など、覚えやすい略称をつけてください。

### Extended Thinking Budget の活用

OpenAIの o1/o3 などをAPIや対応エディタで使う場合、プロンプトだけでなく**パラメータ設定**も重要です。

*   **L0/L1**: `reasoning_effort` (thinking budget) を `low` またはゼロに近づける。
*   **L3**: `reasoning_effort` を `high` に設定し、数万トークンの思考を許容する。

これにより、L0では数秒で返ってくるレスポンスが、L3では数分かけて（しかし極めて精度の高い）回答として返ってくるようになります。

## シリーズ「環境設計」まとめ

全4回にわたり、「AI時代の開発環境設計」について解説してきました。最後に振り返ってみましょう。

1.  **意志より環境（Vol.1）**
    *   人間の「頑張る」という意志力はあてにならない。AIが自然とサポートしてくれる「環境」を作ることが最優先。
2.  **ルールは認知プロテーゼ（Vol.2）**
    *   プロンプトやルールファイルは、私たちの認知能力を拡張する「義足（プロテーゼ）」である。毎回ゼロから考えるのではなく、外部化したルールに頼る。
3.  **ルールの進化（Vol.3）**
    *   ルールは一度作って終わりではない。日々の開発の中で違和感を感じたら即座に更新し、チーム全体で「生き物」として育てる。
4.  **深度制御（Vol.4 / 今回）**
    *   AIのパワーを制御する。全てのタスクに全力投球させるのではなく、ギアチェンジによって効率と品質を最大化する。

このシリーズでお伝えしたかったのは、**「AIに使われるのではなく、AIが最も輝く舞台をこちらが整える」**という姿勢です。

## 読者が明日から試せること

まずは、次の質問をAIに投げかける前に、一瞬だけ止まって考えてみてください。

**「この質問は、L0（即答）案件か？ それともL3（熟考）案件か？」**

そして、もしL0なら「解説不要、コードだけ修正して」と付け加えてみてください。もしL3なら「すぐに答えを出さず、まず問題の背景を整理して」と指示してみてください。

この「深度の指定」を意識するだけで、AIとの対話の質（そしてあなたのストレスレベル）は劇的に改善されるはずです。
