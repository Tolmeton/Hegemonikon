# 【環境設計 #2】AIのルールは「縛り」ではない。思考を拡張する「認知プロテーゼ」としての設計論

「AIに指示を出しても、微妙にニュアンスが伝わらない」
「最初は良かったのに、会話が長引くと前の設定を忘れてしまう」
「毎回『コードは書き換えないで』と書くのが面倒くさい」

あなたも、AIとの対話でこんなストレスを感じたことはないでしょうか？

多くの人は、プロンプトを「命令書」だと考えています。しかし、複雑なタスクをこなすAIに必要なのは、その場しのぎの命令ではなく、**安定して能力を発揮できる「環境」**です。

環境設計において、ルールはAIを縛り付けるための「校則」や「罰則リスト」ではありません。それはAIの思考を支え、本来の能力を引き出すための**「認知プロテーゼ（認知的義肢）」**なのです。

今回は、私が実際に運用している18個のルール（Behavior Criteria）を例に、AIのパフォーマンスを最大化する設計論を解説します。

## ルールは「能力を解放する道具」である

「プロテーゼ（義肢）」と聞くと、失われた機能を補うものを想像するかもしれません。しかし、ここでの意味合いは**「拡張」**です。

例えば、眼鏡は視力を矯正する道具ですが、顕微鏡は「肉眼では見えないものを見る」ための拡張デバイスです。同様に、優れたルールセットは、AIが単独では陥りやすい「ハルシネーション（幻覚）」や「早とちり」を防ぎ、論理的で安全な思考を強制的にブーストさせる**外的な思考構造**として機能します。

**「外的な構造が、内的な能力を解放する」**。これが環境設計の基本思想です。

## 18個のルールを「環境」として実装する

私は現在、AIエージェントの挙動を制御するために、`BC-1`から`BC-18`までの18個のルールを定義しています。これらは大きく2つのカテゴリに分類されます。

### 1. Always-On（常時適用）
どんなタスクであれ、常に守るべき基本姿勢です。AIの人格や安全性の根幹を成します。

### 2. Context-Triggered（条件付き適用）
特定の状況（コーディング中、執筆中、デバッグ中など）でのみ発動するルールです。トークン節約のため、そして文脈を汚染しないために使い分けます。

| カテゴリ | ルール数 | 役割 | 例 |
| :--- | :---: | :--- | :--- |
| **Always-On** | 5 | 安全性、基本姿勢、メタ認知 | 確信度の明示、破壊的操作の禁止 |
| **Context-Triggered** | 13 | 専門技能、特定手順 | テスト駆動開発、ドキュメント形式 |

## 具体的な設計思想：3つの主要ルール

では、具体的にどのようなルールが「認知プロテーゼ」として機能するのか、代表的な3つを見てみましょう。これらはシステムプロンプトや`.cursorrules`などに記述して運用します。

### BC-1: 流し読み禁止 (No Skimming)

AIはトークンを節約しようとして、長いコンテキストを要約（圧縮）して理解しようとする傾向があります。これが「見落とし」の原因です。

```markdown
# BC-1: No Skimming (Deep Reading Protocol)
- ユーザーからの入力、添付ファイル、参照コードは、一言一句「完全」に読むこと。
- 要約や推測による補完を禁止する。
- 曖昧な点があれば、回答を生成する前に必ず質問を行うこと。
```
このルールは、AIに対して「急いで答えを出すな、まずは読め」という思考のブレーキをかけます。

### BC-5: Proposal First (破壊的操作前の確認)

AIが良かれと思ってコードを書き換え、元のロジックを破壊してしまった経験はありませんか？ これを防ぐのが「提案ファースト」です。

```markdown
# BC-5: Proposal First
- ファイルの削除、大幅な書き換え、外部APIへのPOSTなど、不可逆な変更を行う前には必ず「提案」を行うこと。
- ユーザーからの明示的な承認（"Approve" または "Yes"）があるまで実行してはならない。
```
これは単なる安全装置ではなく、**「思考のステップを一段階増やす」**効果があります。一度プランを言語化させることで、AI自身による自己修正の機会が生まれるからです。

### BC-6: 確信度明示 (Confidence Level)

AIは平気で嘘をつきます（もっともらしく話します）。これを防ぐには、AIに「メタ認知（自分の思考に対する客観視）」を強制させます。

```markdown
# BC-6: Explicit Confidence
- 回答には必ず確信度（0-100%）を付記すること。
- 確信度が80%未満の情報については、その根拠または「推測である旨」を明記すること。
- 知識がない場合は正直に「知らない」と答えること。
```
このルールがあるだけで、AIの出力は「断定」から「検討」へと質が変化します。

## Khōra (場) による使い分け：聖域と遊び場

環境設計において重要なもう一つの概念が、ギリシャ哲学用語を借用した**「Khōra（コーラ／場）」**の概念です。すべての場所で同じルールを適用する必要はありません。

### 聖域（Sanctuary）
*   **適用場所:** 本番環境のコード、マスターブランチ、重要なドキュメント。
*   **ルール:** 全ての安全装置（BC-1〜18）をフル稼働。
*   **目的:** ミスゼロ、安全性最優先。

### 遊び場（Playground）
*   **適用場所:** 新規アイデア出しのチャット、実験用スクリプト、tmpフォルダ。
*   **ルール:** 最小限（Always-Onのみ、あるいはルールなし）。
*   **目的:** 創造性、スピード、試行錯誤。

「ここは遊び場だから、自由にコードを書いていいよ」とAIに伝える（コンテキストを切り替える）ことで、AIは慎重なアシスタントから、大胆なアイデアマンへと変貌します。

## SafeToAutoRun：自律実行の線引き

最近のAIエージェント（CursorのComposerやAutoGPTなど）は、コマンドを自動実行する機能を持っています。ここで重要になるのが `SafeToAutoRun` というフラグの考え方です。

| 操作タイプ | SafeToAutoRun | 理由 |
| :--- | :---: | :--- |
| **Read** (ls, cat, grep) | `true` | 情報収集は無害であり、自動化すべき。 |
| **Write** (echo, mv, rm) | `false` | 破壊的変更は人間が確認すべき（BC-5の発動）。 |

この線引きをルールとして環境に埋め込んでおくことで、「目を離した隙に重要なファイルが消えていた」という事故を防ぎつつ、調査作業の自動化による恩恵を受けることができます。

## 読者が今すぐ試せること

AIの環境設計は、今日からすぐに始められます。以下の3ステップを試してみてください。

1.  **「Always-On」ルールを3つ決める**
    *   例：「嘘をつかない（確信度を書く）」「勝手にファイルを消さない」「日本語で答える」
    *   これをCustom Instructionsやシステムプロンプトの冒頭に固定します。

2.  **「Proposal First」を徹底する**
    *   AIに作業を依頼する際、プロンプトの最後に「実装の前に、まずは計画を箇条書きで提案してください」と付け加えてみてください。これだけで出力精度が劇的に向上します。

3.  **役割（ロール）ではなく、制約（ルール）を与える**
    *   「あなたは優秀なエンジニアです」という曖昧な役割定義よりも、「あなたは変数を定義する際、必ずTypeScriptの型注釈をつけなければなりません」という具体的な制約の方が、AIの能力を引き出します。

AIにとってのルールは、私たち人間にとっての「習慣」や「規律」と同じです。良い習慣が良い人生を作るように、良いルールが良いAIを作ります。
