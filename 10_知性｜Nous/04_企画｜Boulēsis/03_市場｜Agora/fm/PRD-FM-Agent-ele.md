## Pre-mortem (事前失敗分析)

> **「6ヶ月後、FM-Agent は失敗した。何が起きたか?」**

### 🐯 Tigers (現実的な脅威 — 対策必須)

| # | 失敗シナリオ | 確率 | インパクト | 緩和策 |
|:--|:-------------|:-----|:----------|:-------|
| T1 | VLM が FM 特有の UI 要素 (ポータル内ポータル、タブパネル) を誤認識 | 高 | 高 | ドメイン特化の few-shot プロンプト + 認識失敗時の手動フォールバック |
| T2 | Windows VM の FM Pro が不安定 (ライセンス認証, 自動更新) | 中 | 高 | スナップショット復元 + 自動更新無効化 + ライセンス監視 |
| T3 | Creator の工数不足で MVP が遅延 | 高 | 中 | スコープ縮小 (KR1 の XML 貼付のみ先行) |

### 🐯📄 Paper Tigers (見かけだけの脅威)

| # | 心配事 | なぜ Paper Tiger か |
|:--|:-------|:---------------------|
| PT1 | 「FM が AT-SPI に対応しないのは致命的」 | → Bytebot VLM パスで解決済 |
| PT2 | 「FM Pro のライセンス費用が高い」 | → 社長の会社が既に保有 |
| PT3 | 「VLM は遅すぎる」 | → 配備・検証は即時性不要。バッチ実行で十分 |

### 🐘 Elephants (誰も言わない根本リスク)

| # | 隠れたリスク | なぜ Elephant か | 対処 |
|:--|:-------------|:------------------|:-----|
| E1 | **FM Pro 自体の将来性** | Claris の方向性は不透明。FM 市場自体が衰退すれば FM-Agent は不要に | → FM-Agent の技術 (VLM GUI 自動化) は他プラットフォームに転用可能な汎用技術と位置づける |
| E2 | **社長の退任リスク** | 社長がいなくなると FM 受託案件の発注源が消失 | → creator 個人の FM 開発能力 + OSS 化で社外にも価値を提供 |
| E3 | **VLM の進化が FM-Agent を陳腐化** | Claude Computer Use 等が FM を直接操作可能になれば FM-Agent 固有の価値がなくなる | → FM ドメイン知識 (DDR 照合、ポータル構造理解) が差別化要素。汎用エージェントには再現困難 |
