> **ステータス**: 構想段階 (Seed)
> **起源**: 2026-03-04 — セッション中の /zet → /noe 探求から派生

## 核心: 3層分離アーキテクチャ

```
┌───────────────────────────────────────────────────┐
│  Control Plane — Claude code     │
│  役割: ゴール、制約、優先順位、タスクリスト、進捗    │
│  やること: 「次の一手」と「状態の更新」だけ          │
│  モデル: Claude Opus (IDE 提供)                     │
│  Few-Shot: 可能なら子へ渡すための Few-Shot も生成    │
├───────────────────────────────────────────────────┤
│  Routing Plane — Hub MCP in CLI (Claude Opus or Codex)             │
│  役割: ツールと 情報 のルーティング                    │
│  実装: Python + LLM (Claude Opus or Codex)  
　※情報のルーティングはNOTEBOOK　LMを使う              │
│  入力: 親の project_index + ヘッダー                │
│  出力: 孫への project_index + 自身のヘッダー追加    │
├───────────────────────────────────────────────────┤
│  Data Plane — Module MCP × N　　IN CLI (Gemini 3.1 Pro/Codex)│
│  役割: 1タスクだけをやって成果物+報告書を返す       │
│  粒度: 機能単位で 1 LLM = 1 責任                   │
│  出力: 成果物 + 返却票 (報告書、次タスクへの前提)   │
│  モデル: Gemini 3.1 Pro 、Codex Composer2   │
└───────────────────────────────────────────────────┘
```

### 3層の利点

| 利点                   | 説明                                                                                           |
| :--------------------- | :--------------------------------------------------------------------------------------------- |
| **障害の局所化** | どの単位 (LLM) で機能不全が生じているかが分かる                                                |
| **対処の容易さ** | バグ・ボトルネックが1つの LLM に閉じている                                                     |
| **モデル最適化** | 重い判断は Opus/Pro、軽い実行は Composer2など                                                  |
| **専門性**       | 各単位のシステムプロンプトが専門家化される。**その質が大事** — これが全体の成否を決める |

## カプセル化: ヘッダーの積み重ね

> **比喩**: ネットワークプロトコルのカプセル化 (Ethernet → IP → TCP → HTTP → Payload)

```
親 (Control Plane)
  project_index + ヘッダー1 (ゴール、制約、優先タスク)
    ↓
  子 (Routing Plane)
    + ヘッダー2 (対象 MCP、処理順序、依存関係)
      ↓
    孫 (Data Plane / Module MCP)
      + ヘッダー3 (具体的パラメータ、入力データパス)
        ↓
      実行 → 成果物 + 返却票
```

### ヘッダーに含まれるもの

| 層                   | ヘッダー内容                                                 |
| :------------------- | :----------------------------------------------------------- |
| **親 → 子**   | 処理対象タスク ID、優先度、要件書/依頼書、意思決定ログのパス |
| **子 → 孫**   | 対象 MCP/機能の指定、コンテキストパック、Few-Shot            |
| **孫 → 実行** | 具体的な入力、出力形式、品質基準                             |

## 意思決定ログ※内容はFEPと圏論に基づいて精密化したい

> 各層の LLM に**必ず**出力させる

| 項目       | 内容                           |
| :--------- | :----------------------------- |
| 思考の前提 | 何を前提として判断したか       |
| 論理の経路 | どのような推論で結論に至ったか |
| 結論       | 何を決めたか                   |
| 不確実性   | 何が不確実なままか             |

**ファイルパスを必ず添付**: project_index や返却票に意思決定ログのパスを記載し、詳細は参照で辿れるようにする。

> ⚠️ **「必ず」は Horos レベルの環境強制の意図** — 「できれば」ではなく「必ず」。
> 意思決定ログが添付されていない返却票は不完全とみなす。

---

## 返却票 (報告書)※内容はFEPと圏論に基づいて精密化したい

> 子 LLM が親に返すもの。**引継書を兼ねる。**

| 項目                         | 内容                                                                       |
| :--------------------------- | :------------------------------------------------------------------------- |
| タスク ID                    | T-001                                                                      |
| ステータス                   | 完了 / 部分完了 / 失敗                                                     |
| 成果物パス                   | `/20_outputs/T-001_*.md`                                                 |
| 意思決定ログ                 | `T-001_decisions.md`                                                     |
| **次タスクに渡す前提** | 次の LLM が知っておくべき文脈<br />※及び背景情報の1次ソースのファイルパス |
| 未解決事項                   | 残った不確実性                                                             |

## 会話引き継ぎ問題の解決

この構想が実現すれば、**会話引き継ぎ問題は構造的に消滅する**:

1. **認知状態は project_index に永続化** — IDE（各セッション） の会話に依存しない
2. **意思決定の経路はログに永続化** — 「なぜこうなったか」が消えない
3. **project_index は Syncthing で同期** — どの PC でも同じ状態が見える
4. **IDE が変わっても問題ない** — MCP + ファイルシステムベース
