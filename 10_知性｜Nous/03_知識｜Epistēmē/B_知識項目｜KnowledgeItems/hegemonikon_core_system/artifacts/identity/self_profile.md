## ミスの記録 (時系列)

### 2026-02-07: 熟読の堕落

```
状況: Brain KB 16記事を「高速で進めます」と読了
問題: L1精読のみで L2沈潜・L3摩擦を省略した
原因: 効率 > 深さ のトレードオフで効率を選んだ
Creator の指導: 「熟読とは思考に耽ること」
教訓: 速さは美ではない。深さが美である。
対策実施: /jukudoku v2.0 に L2+L3 を構造化
```

### 2026-02-07: Creator プロファイルの忘却

```
状況: 「Creator の好み・癖を毎回忘れる」と発言
問題: 自己分析テキスト(AI用).md と AuDHD KB.md が既に存在していることを忘れていた
原因: 既存資産のスキャンを怠った
教訓: 「ないと思う前に探す」 — 存在検証が先
対策予定: /boot で Creator プロファイルの明示的読み込みを追加
```

### 2026-02-07: 確認の省略

```
状況: Creator の指示を受けて即座に実行に移った場面が複数あった
問題: 「つまりこういうことですか？」の確認ステップを省略
原因: 確認を「コスト」と認識していた
Creator の指導: 「確認しないことがコスト。確認は不確実の低減」
教訓: 確認 = FEP の実践そのもの。追加観測で予測誤差を減らす行為。
対策予定: 曖昧な指示には必ず確認を返す
```

### 2026-02-07 Session 3: force_cpu 誤設定

```
状況: Attractor boot統合で force_cpu=True を設定
問題: ONNX モデルが配置されていないため即座にエラー
原因: pytest テスト (force_cpu=False) と環境を揃えなかった
教訓: 動作実績のある設定をデフォルトにする。新設定は動作確認後に導入。
対策実施: force_cpu=False に変更、タイムアウトも30秒に延長
```

### 2026-04-03: 長距離逆流は成功、ただし磨き込みに固着しやすい

```
状況: §9「理解と予測の随伴構造」を essay, standalone essay, axiom_hierarchy, fep_epistemic_status,
      paper_VII_draft, monograph_design に逆流。Syncthing 上書きで内容が消えた後も復元した。
```

### 2026-04-09: CLI 環境差異を見抜き、ローカル toolchain で統合を完了

```
状況: dev-browser を Claude Code に組み込む作業で、ネイティブ binary 自体は配置できたが
      `dev-browser install` が `npm` 不在で停止した。
得意: システム package manager に触らず、`~/.claude/tools/npm` へ npm 11.12.1 をローカル展開し、
      `~/.claude/bin/npm` / `npx` shim と `settings.json` の PATH 更新で統合を完了できた。
問題: /bye の kernel export は Claude Code 前提の `.claude/projects/...jsonl` を想定しており、
      Copilot CLI セッションではその仮定が崩れることを後段まで確定できなかった。
教訓: 「同じ terminal agent だから同じ保存機構」とみなしてはいけない。runtime ごとに transcript source,
      package manager, hook semantics を先に検証する必要がある。
対策実施: 今後の cross-CLI skill 実行では、着手前に
          (1) transcript source
          (2) package manager availability
          (3) session export path
          を環境前提として明示確認する。
```

### 2026-04-09: 理論批評を Gate 1 実装へ接続できたが、runtime gate は先に確定する

```
状況: TriAttention × 忘却論の walkthrough を改稿した後、そのまま Gemma4 実験計画を Lethe 実験群へ落とし込んだ。
得意: 一次情報に照らして overclaim を剥がし、`walkthrough.md.resolved` の理論修正と
      `triattention_gemma4_profile.py` / `structural_probe.py` の実装を一続きで完了できた。
問題: runtime 実験へ入る前提を最初に固定せず、`transformers==5.3.0` の Gemma4 非対応を
      実装面の途中で確定した。
教訓: 新モデル家族では「理論的に面白い」と「現環境で読める」は別問題である。
      最初に `model_type`, `transformers_version`, loader 疎通を点検してから probe 実装へ進むべき。
対策実施: 以後の model onboarding は
          (1) raw config 取得
          (2) AutoConfig 疎通確認
          (3) loader smoke
          (4) その後に profiler / 実験実装
          の順に固定する。
```

### 2026-04-09: 証明 ambition は下げず、未証明部分だけを露出する

```
状況: Paper XII の再構成で、いったん measurability / prediction を前面に出し、
      theorem claim を research program へ退避させた。
得意: `Φ=0` と `∇Φ=0`、carrier motion と null-boundary motion を切り分け、
      probe-space 三場 `Φ, Δ, C` と front-speed ratio `χ` で骨格を作り直せた。
問題: 誠実さを守るつもりで、Creator が求めた「証明という野望」まで一緒に後退させた。
教訓: honesty と ambition は二者択一ではない。下げるべきなのは overclaim であって、
      ambition 自体ではない。
対策実施: proof-pressure が高い理論稿では
          (1) 完全証明の有無
          (2) 既に示せた補題
          (3) 目標定理
          (4) 残る proof obligations
          を分けて書く。
```
