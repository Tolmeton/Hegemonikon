# /fit: Fit 判定（体系への馴化診断）

> **親クラス**: A2 Krisis (旧体系 — _archive/legacy/skills/ に移行)
> **目的**: 制作物が HGK 体系に密に Fit しているか（🔴 表面付着 / 🟡 吸収 / 🟢 馴化）を判定する
> **親コマンド**: /dia
>
> **v4.0 変更**: `/eat` 専用 → **全制作物共通**に汎化。θ7.2（完了禁止）と連携。

---

## 第一原理

```yaml
公理:
  - 「機能は発動可能でなければならない」← これだけが公理
  - 「新しいモジュールには新しいコマンドが必要」← これは仮定であり逃げ道

定義:
  表面付着 (Superficial):
    formula: A + B = A に B がくっついている状態
    問題: 重複あり、境界が残る、「付着」
    例: スタブだけ作って中身がない、grep でゼロ件だから「対応した」
    
  吸収 (Absorbed):
    formula: A + B → A' (構造的に統合されているが境界が見える)
    例: テストは通る、参照関係は解決されている、だが用語の境界が残る
    
  馴化 (Naturalized):
    formula: A + B → A'' (境界が消え、元からそうだったかのように一体化)
    検証: 第三者が「元から存在した」と誤認する
    追加検証: 消しても何かが壊れる（依存関係が存在する）

naturalized_state:
  定義: |
    制作物の形が消え、既存体系が「最初からそうだったかのように」強くなった状態。
    消したら何かが壊れる — つまり体系の一部になっている。
```

## 制作物タイプ別 Fit 判定基準

### コード

| 🔴 表面付着 | 🟡 吸収 | 🟢 馴化 |
|:-----------|:--------|:--------|
| ファイルが存在する | テストが通過する | 他モジュールから依存されている |
| コンパイル/構文エラーなし | 既存モジュールとの統合が動作 | 消したらテストが壊れる |
| grep で残存ゼロ | エッジケースが考慮されている | 体系の一部として自然に見える |

### WF/Skill

| 🔴 表面付着 | 🟡 吸収 | 🟢 馴化 |
|:-----------|:--------|:--------|
| ファイルが存在する | trigonon が定義されている | 他WFから参照されている |
| frontmatter がある | 射（bridge/anchor）が定義 | CCLマクロで使用されている |
| 行数 < 100（スタブ） | cognitive_algebra が定義 | 消したら他WF/マクロが壊れる |
| 構造が単なる平文 | SkillRegistry で Phase パース可能 | Phase 単位で動的抽出・実行可能 |

### 設計変更

| 🔴 表面付着 | 🟡 吸収 | 🟢 馴化 |
|:-----------|:--------|:--------|
| ドキュメントが更新された | 関連コードも修正済み | 体系全体の一貫性が維持 |
| 変更意図が書かれている | テストが通過する | version/lineage が更新 |

### 修正/バグ修復

| 🔴 表面付着 | 🟡 吸収 | 🟢 馴化 |
|:-----------|:--------|:--------|
| grep でゼロ件 | テストが通過する | 関連する全テストが通過する |
| 対象ファイルを修正した | エッジケースが処理される | 修正が他箇所に波及していない |
| 「対応した」と言える | 修正の理由が説明できる | 同種の問題の再発防止策がある |

## 核心テスト: 「消しても壊れないか」

> **これが Fit の本質。何も壊れないなら、それは体系に Fit していない。**

```
問い: この制作物を消したら、何が壊れるか？

何も壊れない     → 🔴 表面付着（体系に根を張っていない）
テストが壊れる   → 🟡 吸収（構造的依存がある）
WFが壊れる
+ 他モジュールが壊れる
+ 概念が欠落する → 🟢 馴化（体系の不可分な一部）
```

## 処理フロー (6 ステップ)

// turbo-all

### Step 0: 制作物の特定

```yaml
対象:
  制作物: [何を作った/変更したか]
  タイプ: [コード / WF / 設計 / 消化 / 修正]
  体系内の位置: [どの WF/モジュール/定理に関わるか]
  
期待状態:
  - 制作物の「形」が消えている（別物として浮いていない）
  - 体系が「最初からそうだった」かのように強化されている
  - 消したら何かが壊れる
```

---

### Step 1: 境界残存チェック

**問い**: 制作物の「形」がまだ見えるか？

```python
boundary_signals = [
    "制作物の名前がファイル名/セクション名として浮いている",
    "「〜から移行した」「〜を追加した」等の履歴コメントが目立つ",
    "制作物固有の用語が体系内で浮いている",
    "機能が重複している（同じことを2箇所で定義）",
    "スタブだけで中身がない",
]
```

チェック項目:

- [ ] 制作物の名前がそのまま浮いていない
- [ ] 体系の用語体系に統一されている
- [ ] 既存の構造に溶け込んでいる

---

### Step 2: 機能重複検出

**問い**: 同じ機能が複数箇所で定義されていないか？

```yaml
重複パターン:
  type_a: "同一機能が2箇所で定義されている"
  type_b: "70%以上の機能が重複している"
  type_c: "異なる名前で同じ概念を表している"
```

チェック項目:

- [ ] 新規コマンド/WFを不必要に作成していない
- [ ] 既存WFのモード/サブ機能として吸収されている
- [ ] 機能の「住所」が1箇所に統一されている

---

### Step 3: 強化度評価

**問い**: 体系は「より強く」なったか？

```yaml
評価軸:
  capability_expansion:
    問い: 前にできなかったことができるようになったか？
    スコア: 0-3
    
  coherence_improvement:
    問い: 体系の一貫性は上がったか？
    スコア: -1, 0, +1
    
  cognitive_load_reduction:
    問い: 覚えることは減ったか？
    スコア: -1, 0, +1

empowerment_score = capability + coherence + cognitive
# 最大 5, 最小 -2
```

チェック項目:

- [ ] 新しい能力が追加されている
- [ ] 既存の動作が破壊されていない
- [ ] 覚えるべきコマンド数が増えていない

---

### Step 4: 消しても壊れないかテスト（v4.0 追加）

**問い**: この制作物を消したら何が壊れるか？

```yaml
手順:
  1. 制作物が参照されている箇所を列挙する
  2. 制作物に依存するテストを列挙する
  3. 制作物を消した場合のインパクトを推定する

判定:
  何も壊れない:     🔴 表面付着
  テストが壊れる:   🟡 吸収
  体系が壊れる:     🟢 馴化
```

---

### Step 5: Fit 判定

3段階レベルを適用:

```
┌─────────────────────────────────────────────────────────────────┐
│ 🔴 表面付着 (Superficial)                                        │
│   • ファイルが存在する（だけ）                                     │
│   • grep でゼロ件（だけ）                                         │
│   • 参照関係が未解決                                               │
│   • 境界が明確に見える                                             │
│   • 消しても何も壊れない                                           │
│   → 未踏の領域が多い。体系に根を張っていない。                     │
├─────────────────────────────────────────────────────────────────┤
│ 🟡 吸収 (Absorbed)                                                │
│   • 参照関係は解決、テストは通過                                   │
│   • しかし境界がまだ見える                                         │
│   • 消したらテストが壊れる                                         │
│   → 構造的には統合されているが、馴化には至っていない。             │
├─────────────────────────────────────────────────────────────────┤
│ 🟢 馴化 (Naturalized)                                             │
│   • 境界が消えている                                               │
│   • 「最初からそうだった」と見える                                 │
│   • 体系が純粋に強化されている                                     │
│   • 消したら体系が壊れる                                           │
│   → 不可分な一部。                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## θ7.2 (N-7) 連携

> `/fit` は θ7.2（完了禁止）の環境的強制の実体。
> θ7.2 (N-7) が「完了という概念を禁止」し、`/fit` が「代わりに何を出力すべきか」を具体化する。

| θ7.2 (N-7) が禁止するもの | `/fit` が代わりに出力するもの |
|:--------------------|:--------------------------|
| 「修正済み」 | 📍 現在地: 🟡 吸収 |
| 「対応した」 | 🕳️ 未踏: {検証していないこと} |
| 「できた」 | → 次: {残課題} |

## Hegemonikón 哲学

> **「システムの末端、端の端まで、可能な限り統制し、一貫させる」**
>
> 「美しくない」は感情的表現ではなく技術的警告。
>
> - 命名規則の不統一

> - 責務の曖昧化
> - 表面だけなでた浅い作業
