# Anthropic Computer Use — 行動仕様分析 (A1)

## 1. Executive Summary

Anthropic Computer Use は、Web ではなく **OS 全体のデスクトップ環境** を対象とする API/Agent アーキテクチャである。純粋な **スクリーンショットベースの視覚理解** と、マウス座標ベースの汎用操作を用いており、DOM 構造に依存しないため、あらゆるネイティブアプリや未学習の UI にも対応できる「汎用性」と「自律性」を持つ。

---

## 2. 操作プリミティブ表

| カテゴリ | 操作 | ターゲット | 特徴 |
| :--- | :--- | :--- | :--- |
| Interaction | `mouse_move` | 座標 (x, y) | スクリーンショットをグリッド分割して座標を特定 |
| Interaction | `left_click` | 座標 (x, y) | 要素のピクセル中心を狙う |
| Interaction | `left_click_drag`| 座標A → 座標B | スクロールや範囲選択に使用 |
| Input | `type` | アクティブなフォーム | キーボードエミュレーション（文字列全体を送信） |
| Input | `key` | 特殊キー | Return, Tab, PageDown 等のシステムキー |
| Extraction | `screenshot` | 全画面 | 毎アクションの結果を取得する必須プリミティブ |

## 3. OODA サイクル分析

* **Observe（観察）: 1.0〜2.0秒**
  * **メカニズム:** 画面全体のスクリーンショットを撮影し、Claude 3.5 Sonnet / 3.7に送信。DOM情報は一切使わない。
* **Orient（情勢判断）: 2.0〜3.0秒**
  * **メカニズム:** 画像からターゲット要素のピクセル座標を算出し、現在のアプリケーション状態を判断。
* **Decide（意思決定）: 2.0〜4.0秒**
  * **メカニズム:** LLM が「次にどのツール（bash, computer, str_replace_editor）を使うか」を選択。
* **Act（行動）: 0.1〜0.5秒**
  * **メカニズム:** Docker 環境内の X11 サーバーに対して xdotool 経由でマウス/キーボードイベントを発火。

## 4. エラーリカバリ表

| 失敗モード | 検知方法 | 回復戦略 |
| :--- | :--- | :--- |
| 要素が見つからない | Observe (スクリーンショット内に該当要素なし) | `key('PageDown')` または `left_click_drag` で画面をスクロールして再探索 |
| クリックミス | Observe (クリック後の画面変化なし、または入力カーソルが違う場所にある) | 再度スクリーンショットを取得し、座標を再計算して `left_click` をリトライ |
| UI の変化 | Observe (想定していたモーダルやポップアップが邪魔している) | ポップアップのクローズボタンの座標を特定してクリックし、メインタスクに復帰 |

## 5. 設計判断マトリクス

| 設計軸 | トレードオフ / 課題 | Anthropic のアプローチ |
| :--- | :--- | :--- |
| **コンテキスト理解** | 汎用性(Vision) vs 確実性(DOM) | **完全スクリーンショットベース (Vision重視)**。WebだけでなくExcel等あらゆるアプリを操作可能にするため、DOMを捨てた。 |
| **パフォーマンス** | 全画面撮影と推論は遅くコストが高い。 | `wait` 時間を最小化できずレイテンシは高い。**速度より「人間と同じ画面を見る」汎用性を優先**。 |
| **UX/UI** | API提供のため、エンドユーザー向けUIはない。 | **開発者向け (API First)**。Docker 等に組み込んで利用者が独自のループやUI (Streamlit等) を構築する設計。 |
