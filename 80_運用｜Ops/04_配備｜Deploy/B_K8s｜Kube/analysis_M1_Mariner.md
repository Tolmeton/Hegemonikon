### 1. Executive Summary
Project Marinerは、**DOMとスクリーンショットのハイブリッド理解（Vision-Language Model）**を基盤とする高度な自律型ブラウザエージェントである。複雑な情報収集タスクを自律的にサブゴールへ分解・実行する一方、不可逆な操作（購入や送信）の前には必ず**Human-in-the-Loop (HITL)**による確認を挟む「安全性と自律性のトレードオフ」を完璧に制御したコパイロット型アーキテクチャを実現している。

### 3. OODA サイクル分析 (Axis 2, 6, 8: 認知・状態管理・ページ理解)
エージェントの1アクションごとの認知サイクル（OODAループ）のプロファイルです。

*   **Observe（観察）: 所要時間 1.0〜1.5秒**
    *   **メカニズム:** DOMツリー（Accessibility Tree）の解析と、スクリーンショットの視覚情報（VLM）をハイブリッドで取得。
    *   **特徴:** 画面上のインタラクティブ要素に内部的なバウンディングボックス（Set-of-Markの手法に類似）を付与し、座標とDOMノードをマッピングしている。
*   **Orient（情勢判断）: 所要時間 0.5〜1.0秒**
    *   **メカニズム:** 現在の画面状態と、短期記憶（ワーキングメモリ内の過去の行動履歴）、および上位のサブゴールを照合。
    *   **特徴:** 「ポップアップが出現した」「ロードが遅延している」などの状態変化をコンテキストとして認識。
*   **Decide（意思決定）: 所要時間 2.0〜4.0秒 [★ボトルネック]**
    *   **メカニズム:** 大規模言語モデル（Gemini等）へのプロンプト送信と推論。次に実行すべき操作プリミティブとターゲット要素（座標またはセレクタ）を決定。
    *   **特徴:** 推論レイテンシが最大だが、この間にサイドバーUIで「考え中（Thinking...）」のアニメーションを出し、ユーザーの体感待機時間を軽減している。
*   **Act（行動）: 所要時間 0.2〜0.5秒**
    *   **メカニズム:** Chrome DevTools Protocol (CDP) または拡張機能APIを介したイベントの発火。
    *   **特徴:** 人間らしいカーソル移動（ベジェ曲線）は省略し、即座にDOMイベントを発火させている（速度優先）。

### 4. タスク分解ツリー (Axis 3: 階層的目標管理)
エージェントは高レベルのプロンプトを以下のような有向非巡回グラフ（DAG）として動的に分解・管理しています。

```text
[Goal] 条件に合う商品/物件を比較し、最適なものをリストアップして購入準備をする
 ├── [Subgoal 1] 検索プラットフォームへのアクセス
 │    └── [Action] navigate_to("google.com") -> type_text(query) -> click(search)
 ├── [Subgoal 2] 候補リストの抽出 (Loop: Top 3 items)
 │    ├── [Action] open_tab(item_link)
 │    ├── [Action] scroll_to_element(details_section)
 │    ├── [Action] extract_data(price, specs) -> Save to State
 │    └── [Action] close_tab()
 ├── [Subgoal 3] データの比較と構造化
 │    └── [Action] switch_tab(workspace) -> fill_form(extracted_data_json)
 └── [Subgoal 4] 最終アクションの実行準備 (Security Boundary)
      ├── [Action] click("Add to Cart" or "Draft Email")
      └── [Action] wait_for_user(prompt="手続きを進めますか？")
```

### 5. エラーリカバリ表 (Axis 4: 堅牢性戦略)
自律稼働中の例外処理に関する仕様です。

| 失敗モード | 検知方法 (Observe/Orient) | 回復戦略 (Decide/Act) | 結果 |
| :--- | :--- | :--- | :--- |
| **要素の隠蔽** (Cookieバナー等) | ターゲット要素の可視性判定(Visible: False) または VisionとDOMの不一致 | バナーの「Accept/閉じる」ボタンを探索して `click`、その後元の操作をリトライ | 障害物の排除とタスク復帰 |
| **動的ロード遅延** | `wait_for_element` のタイムアウト (例: 5秒) | ページ状態を再取得（Re-observe）、最大3回までエクスポネンシャルバックオフで待機 | ネットワーク遅延への耐性 |
| **CAPTCHA遭遇** | VisionモデルによるCAPTCHA UIの画像認識 | 自律実行を即座に一時停止（Halt）し、ユーザーに手動解決を要求する通知をスロー | アカウント凍結の防止 |
| **ハルシネーション** | 存在しない要素への `click` 試行時、CDPがエラーを返す | エラーログをプロンプトに含めてLLMに再推論（Self-Correction）を要求 | 軌道修正（別要素の探索） |

### 6. パフォーマンスプロファイル (Axis 7: 定量指標の目安)
Marinerクラスの商用レベルエージェントに求められるパフォーマンス要件です。

*   **Total Task Time (平均タスク完了時間):** 1〜3分（タスクの複雑度に依存）
*   **Action Count (平均ステップ数):** 15〜30 アトミックアクション
*   **Average Action Latency (1操作のレイテンシ):** 3.5秒〜5.0秒（VLM推論時間を含む）
*   **Error Count (許容される内部エラー数):** 1〜2回（自己修復可能な範囲）
*   **User Intervention Count (ユーザー介入回数):** タスク開始時の1回 ＋ 最終確認の1回（計2回。完全自律の間の介入はゼロを目指す）

### 7. 設計判断マトリクス (Axis 5, 9, 10: HCI・アーキテクチャ設計)
MarinerのUI/UXおよびシステム設計におけるトレードオフと解決策です。

| 設計軸 | トレードオフ / 課題 | Mariner (DeepMind) の解決アプローチ |
| :--- | :--- | :--- |
| **自律性 vs 安全性**<br>(Axis 10) | 全自動化は誤操作（意図しない購入等）のリスクが高い。 | **Security Boundary（境界）の設定**。情報の「読み取り・抽出」は完全自律で行うが、「書き込み・決済」の直前で必ず制御をユーザーに返す（`wait_for_user`）。 |
| **ページ理解**<br>(Axis 6) | DOM解析は速いが崩れやすい。Visionのみは遅く不正確。 | **ハイブリッド・グラウンディング**。DOMからインタラクティブ要素の候補を絞り、Visionで視覚的なレイアウト文脈を理解する二段構え。 |
| **UI/UX 透明性**<br>(Axis 9) | エージェントが裏で何をしているか分からないとユーザーは不安になる。 | **サイドバーとライブハイライト**。現在実行中のサブゴールをサイドバーに進捗バーとして表示し、操作対象のDOM要素に色付きのバウンディングボックスを重畳描画（Visual Feedback）。 |
| **状態管理**<br>(Axis 8) | 複数タブを開くと、エージェントが「今何のためにこのページにいるか」を見失う。 | **外部化されたワーキングメモリ**。サイドバー（または不可視のJSONステート）に抽出したデータを逐次キャッシュし、LLMのプロンプトコンテキストに常に注入し続ける。 |

### 結論（独自エージェント実装への示唆）
独自エージェントを開発する際、単にLLMとPuppeteer/Playwrightを繋ぐだけでは実用に耐えません。本分析から明らかなように、**「OODAループにおけるObserveのハイブリッド化」**と、**「HCIの観点から設計されたサイドバーUIによる意図の透明化（Explainability）」**、そして**「不可逆操作前の明確な権限委譲プロトコル」**の3点の実装が、ユーザーの信頼を勝ち取るWeb Agentの必須要件となります。
