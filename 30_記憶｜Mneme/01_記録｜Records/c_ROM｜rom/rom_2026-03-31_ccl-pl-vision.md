# ROM: CCL-PL Vision — 圏論的処理構造の内部言語

> Date: 2026-03-31
> Context: Creator の洞察 + ディープリサーチ (コーディングと圏論の関係性) を統合

---

## Creator の核心的洞察

1. **認知動詞は CCL の「関数」のインスタンスである** — CCL が制約するのは「構造」だけ
2. **「関数」という本質の射を「定義」として扱う** — どのような処理も関数（処理の構造）として扱える
3. **処理の骨組み（構造）はどのようなドメインでも共通** — CCL-PL は普遍的な処理の構造を扱う
4. **ドメインごとの肉付けは「拡張機能」で対応** — 骨格と肉付けの分離
5. **具体と抽象の分離、モジュール化**

## 圏論的定式化

### CCL-PL = CCC + Adjunction + Fix の内部言語

| 圏論 | CCL-PL | 直感 |
|:-----|:-------|:-----|
| 対象 | 型 | データの種類 |
| 射 | 関数 | 処理の骨格 |
| 射の合成 | `_` | パイプライン |
| 恒等射 | `id` | 何もしない処理 |
| 積 | タプル/構造体 | 複数データをまとめる |
| 余積 | `match`/`if` | 場合分け |
| 指数対象 | ラムダ/`fn` | 処理を値として渡す |
| **随伴対** | `adjoint f <=> g` | 反対の操作がペアである宣言 |
| **不動点** | `~*` | 繰り返して安定する点 |
| **自然変換** | `~` | 構造を保ったまま変換 |

上の7つ = 全言語が暗黙に持つ構造。下の3つ = CCL-PL だけが言語レベルで持つ構造。

### Extension as Functor

```
Extension = 関手 F: CCL-PL 骨格圏 → ドメイン固有圏
```

- 骨格圏: CCL-PL が定義する「処理の構造」の圏
- ドメイン圏: 認知操作の圏、データ処理の圏、Web操作の圏...
- 拡張 = 関手: 骨格の各射に、ドメイン固有の「意味」を対応させる

## アーキテクチャ (3層)

```
Layer 0: CCL-PL Core (骨格)
  ├── 射の合成規則 (_, *, %, ~, ~*, >>, <<)
  ├── 随伴対宣言 (adjoint)
  ├── 関数定義 (fn, lambda)
  ├── 制御構造 (if, match, let)
  └── Prelude (IO, Math, Data)

Layer 1: Standard Extensions (標準拡張)
  ├── ccl-ext-hgk     — HGK 認知動詞 24 + 12随伴対
  ├── ccl-ext-data    — データ変換 (JSON, CSV, XML)
  ├── ccl-ext-web     — Web/DOM 操作
  └── ccl-ext-fs      — ファイルシステム操作

Layer 2: Community Extensions (将来)
  └── ccl-ext-{任意}  — 誰でも作れる
```

## VISION 3本柱

| 柱 | 内容 | 圏論的意味 |
|:---|:-----|:-----------|
| 骨格 | 射の合成・随伴・不動点・自然変換を言語レベルで提供 | CCC + Adjunction + Fix |
| 拡張 | ドメイン固有の「肉」を関手として注入 | F: Core → Domain |
| 道具 | LSP で IDE に統合し、書きながら構造が見える | 内部言語の外部表示 |

## 先行研究との差異

- Charity (1992), CPL (arXiv:2010.05167) = 圏論の数学的構造を忠実に言語化（学術的）
- CCL-PL = 処理の構造を骨格として抽出し、ドメイン拡張で肉付け（実用的）
