## 🔬 法則化

### L1: 共通関数蒸留の判定基準

```
同じ概念が2箇所以上で実装されている
 AND パス解決ロジックが異なる
 → 1箇所に蒸留すべき（/fit 🟡 → 🟢 の阻害要因）
```

### L2: SkillRegistry の拡張パターン

```
SkillParser はフェーズ構造 (## PHASE) のみ構造化する
非フェーズセクション (認知代数, Anti-Patterns, WM) は
  SkillRegistry.extract_cognitive_supplements() で生テキスト抽出する
この分離は SkillParser の責務を汚染しないための意図的設計
```

## 🎯 Value Pitch

### Before → After

**Before**: 認知代数・Anti-Patterns の抽出ロジックが `macro_executor.py` と `peras_pipeline.py` に分散していた。同じ概念の二重実装 = メンテナンスコスト倍増 + パス解決の不整合リスク。

**After**: `SkillRegistry.extract_cognitive_supplements()` という単一の「住所」に統合。WF 短縮名 (`noe`) も Skill ID (`V01`) も受け付ける統一インターフェース。消したらマクロ実行もペラスパイプラインも壊れる = 体系の不可分な一部。

### 比喩

> 二つの蛇口から同じ水を汲んでいた。
> 一つの水源に統合したことで、水質が保証され、蛇口の修理も1箇所で済む。
> — これが馴化 (Naturalization): 境界が消え、「最初からそうだった」かのように見える設計。
