- Cascade のメッセージ処理に必要な内部オブジェクト (おそらく extension_server 経由で取得すべき何か) が nil

### 3. ダミー Extension Server が受信した RPC メソッド

| メソッド | 頻度 | 備考 |
|:---------|:-----|:-----|
| `LanguageServerStarted` | 1回 | 起動通知 (body 12 bytes) |
| `LaunchBrowser` | 1回 | ブラウザ起動要求 (body 2 bytes: `\x10\x01`) |
| `GetChromeDevtoolsMcpUrl` | 40回+ | Chrome DevTools MCP URL のポーリング (body 0 bytes) |

**注目**: `GetChromeDevtoolsMcpUrl` が大量にポーリングされている。これは MCP (Model Context Protocol) 接続に必要な Chrome DevTools のURLを取得しようとしている。ダミーサーバーが空レスポンスを返しているため、永遠にリトライしている。

### 4. HTTP vs HTTPS ポート

LS は必ず2つのポートをリッスンする:

```
Language server listening on random port at XXXXX for HTTPS  (fd=9)
Language server listening on random port at YYYYY for HTTP   (fd=10)
```

- ConnectRPC (JSON/HTTP) は **HTTP ポート** (fd=10) を使う
- HTTPS ポートに HTTP リクエストを送ると TLS handshake エラー

## 推定原因と次の方針

### 仮説: SendUserCascadeMessage は Extension Server からの応答に依存

`rpcs_cascade.go:508` でクラッシュすることから、Cascade 処理は Extension Server から取得した何らかのオブジェクト (MCP URL? ツール設定? ワークスペース情報?) を前提としている可能性が高い。ダミーサーバーが空レスポンスを返しても、LS はそれを「有効な応答」として受け取り、nil フィールドにアクセスして SIGSEGV する。
