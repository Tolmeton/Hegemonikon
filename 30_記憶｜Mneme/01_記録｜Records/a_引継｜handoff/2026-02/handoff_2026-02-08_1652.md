## 5. 教訓・パターン

- **テスト前に回避策を作るな**: HTTP Request ノードをテストせずに child_process に逃げた。30秒のテストで30分のリファクタが不要だった
- **n8n Code ノードは薄く**: ビジネスロジックのみ。HTTP 送信・条件分岐は n8n ネイティブノードで
- **`$env` は Code ノード内で使えない**: HTTP Request の `={{ $env.VAR }}` または `execSync('echo $VAR')` が必要
- **`$getWorkflowStaticData('global')` は強力**: 同一 WF 内の異なる webhook 実行間で状態共有可能

---

## 法則化 (今日学んだ法則)

1. **回避策螺旋の法則**: 検証なしに回避策を採用すると、回避策が回避策を呼ぶ負のスパイラルに入る。1つの未検証仮定 (`require('https')` は動かない) が4WF×2箇所=8箇所の技術的負債を生んだ
2. **n8n ノード分離の法則**: Code ノードにロジック+IO+分岐を詰め込むと「スクリプトの GUI 化」になる。Code=ロジック、IF=分岐、HTTP Request=IO という分離が n8n の設計意図
3. **通知パラドックスの法則**: 通知システム自体がダウンしたときの通知経路を設計しなければ、最も重要な障害が最も報告されない

## KI生成 (Knowledge Item)

- **n8n HTTP Request ノード**: JS Task Runner 環境でも n8n-nodes-base.httpRequest は正常動作する。axios/1.12.0 を内部使用。`$env.VAR` は URL フィールドの `={{ }}` 式でのみ参照可能
- **n8n staticData**: `$getWorkflowStaticData('global')` は workflow 単位の永続化ストレージ。異なる webhook trigger 間でも共有。再起動しても保持される (SQLite DB に保存)
