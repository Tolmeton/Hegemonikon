### Phase 2: /dox+ — 信念の詳細化 (Belief Documentation)

#### 信念 1: 「OAuth Client ID の選択が権限を決定する」

APIアクセスは token のスコープだけでなく、**どの OAuth Client ID で発行されたか** によって暗黙の権限が異なる。gemini-cli の Client ID (`681255809395-...`) は Gemini Code Assist 用のベースライン権限を持つ。gcloud の Client ID (`32555940559.apps.googleusercontent.com`) は汎用で、同じスコープでも到達できるサービスが異なる。

**確信度**: [確信: 95%] (SOURCE: 同一ユーザー、同一スコープで、Client ID の違いのみで結果が分かれた)

#### 信念 2: 「x-goog-user-project はクォータ割当用であり、つけると新しい権限チェックが発生する」

ヘッダーなし → Google が内部的にプロジェクトを決定 (loadCodeAssist の応答に基づく)
ヘッダーあり → 指定プロジェクトの `serviceUsageConsumer` 権限チェック + API 有効化チェック

**つけると壁が増える。外すと壁が減る。**

**確信度**: [確信: 90%] (SOURCE: ヘッダーの有無で 403/200 が切り替わる実験結果)

#### 信念 3: 「loadCodeAssist が返すプロジェクト ID こそが真のエントリポイント」

GCP Console に表示されるプロジェクトは管理上のもの。Cortex API が使う「真のプロジェクト」は `loadCodeAssist` が動的に返す。これは Google One サブスクリプションと紐づく **隠しプロジェクト**。

**確信度**: [確信: 95%] (SOURCE: `driven-circlet-rgkmt` は GCP Console に表示されないが、generateContent で使用可能)
