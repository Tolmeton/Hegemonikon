# Handoff: Standalone LS Claude 到達の証明と課題

## 🎯 成果 (Value Pitch)

**不可能と思われていた「IDE非依存でのClaude直接アクセス」への道を開拓しました。**

REST API では Gemini にフォールバックされてしまう問題に対し、**Standalone LS と 自作の Extension Server (Port 50001)** を組み合わせる構成を立案・検証。結果、Claude Opus (MODEL_PLACEHOLDER_M26) への ConnectRPC リクエストが受理され、LSが実際にLLMバックエンドへプロンプトを送信するログを観測しました。

あとは応答テキストを受け取る非同期ストリーム（GetAllCascadeTrajectories / GetCascadeTrajectorySteps）のブロックを解除するだけです。IDE の制約から解放された真のマルチモデル環境が目前に見えています。

## 🧠 信念 (Doxa)

- **REST API の限界**: クライアントから提供される `tier_id` や特定フィールドは無視され、REST `generateChat` では Claude の thinking を引き出すことはできない。(100% 確信)
- **Standalone LS の鍵**: `--extension_server_port` で OAuth の CSRF とトークンを借用できる。IDE 以外のプロセスが Extension Server を名乗る（Creator の50001番ポート）構成は非常に有効。
- **ConnectRPC の挙動**: `SendUserCascadeMessage` は非同期であり、即座に `{}` を返す。結果は polling で `GetCascadeTrajectorySteps` を叩いて取得する必要があるが、現状ポーリング時にポートがブロックされる課題がある。
