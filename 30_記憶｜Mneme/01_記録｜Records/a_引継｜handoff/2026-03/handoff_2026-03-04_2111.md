# 🤝 Handoff: Mneme ベクトル検索の統一と「Kalon」の達成

## 💎 Value Pitch

**「美しさは、未来の自分への最大の防御である」**

Mneme で発生していた `1024d` と `3072d` の次元不一致エラー。これを単に「場当たり的に直す」のではなく、3回の G∘F イテレーションを経てアーキテクチャを極限まで蒸留しました（◎ Kalon 判定）。

エンベディング生成は `embedder_factory` への一元化（たった2行に蒸留）。ストレージは名実ともに `VectorStore` へと改名し、エンベディングの責務から解放。これにより、現状の pkl ベースの素朴な検索が限界を迎えた時（たとえば 10,000件を超えてメモリを圧迫した時）、システム全体を壊すことなく、ストレージ部分だけを Faiss や LanceDB に安全に差し替える**「未来の選択肢」**が生まれました。

技術的負債を取り除き、成長の余白（Function）を確保したセッションです。

---

## 🎯 意図と結果 (Intent & Outcome)

- **Situation (状況)**:
  - BGE-M3 (1024d) と VertexEmbedder (3072d) が混在し、次元不一致エラーが発生していた。
- **Background (背景)**:
  - 検索機能とインジェスト機能の各所に `EmbeddingAdapter` 呼び出しが散在し、モデル制御の Single Source of Truth が不在だった。
- **Assessment (評価)**:
  - 場当たり的な修正ではなく、構造的純化 (G) と未来への拡張性 (F) の両立が必要。
- **Recommendation (推奨)**:
  - G∘F 第1〜3ラウンドによる完全なリファクタリングを実施し、責務を分離した。

| 達成したこと | 内容 |
|:-------------|:-----|
| エンベダー統一 | `embedder_factory.py` への管理移行（3072d 完全統一） |
| コード蒸留 | `EmbedderMixin.embed()` の ndarray 対応、`reingest_vertex.py` 削除 |
| クラス名実一致 | `EmbeddingAdapter` → `VectorStore` へ改名し、後方互換エイリアスで保護 |

## 📌 Context

- **現在の状況**: 全コアファイル (`mneme_server`, 各種 `ingest`, `search`) で書換え完了。E2E 検索テスト通過。エラー撲滅。
- **重要な前提**:
