## 📋 状況 (Situation)

DE (GNOME Desktop) が OOM killer でクラッシュする問題が発生。`journalctl` と `dmesg` から、`ls-daemon` プロセスのメモリ消費と大量のログ生成が原因と特定。

## 🔍 背景 (Background)

- `ls-daemon` が `SubscribeToUnifiedStateSyncTopic` RPC を **~47回/秒** で呼び出し
- 各コールで **~562KB** のデータ転送 → メモリ/ログ爆発 → OOM
- 根本原因: `ext_server.py` の `SubscribeToUnifiedStateSyncTopic` が **Request-Response** で実装されていた (正しくは **ServerStreaming**)
- LS クライアントが EOF を受信 → 即座に再接続 → 無限ループ

## ✅ 実施内容 (Assessment & Actions)

### 1. 根本修正: ServerStreaming 実装
- `ext_server.py` の `do_POST` を修正: 接続を維持し、クライアント切断まで EOF を返さない
- `socket.MSG_PEEK` + `socket.timeout` でクライアント切断を検知

### 2. Elenchos 5粗修正 (`/ele+` で検出)

| ID | 重要度 | 修正内容 |
|---|---|---|
| C-1 | Critical | `_max_hold_seconds` (3600秒) でスレッド枯渇防止 |
| C-2 | Critical | `_shutdown_event` パターンで shutdown デッドロック防止 |
| M-1 | Medium | varint デコーダで 128バイト超トピック名対応 |
| M-3 | Medium | 未知トピックに空 initial_state 返却 (KeyError 防止) |
| L-2 | Low | 接続維持ログを INFO → DEBUG |

### 3. テスト追加 (M-2)
- `test_ext_server.py` に6件追加 → **11 passed in 3.05s** (既存5 + 新規6)
- 接続維持、graceful shutdown、未知トピック、varint (短/長/空) をカバー

### 4. インフラ反映
- Docker LS イメージ rebuild (`docker compose up -d --build`)
- ホスト側 `ls-daemon.service` を `systemctl --user restart` で再起動
- Gradle 拡張 (`vscjava.vscode-gradle`) を無効化 → ~450MB RAM 削減

## ⚡ 設計上の決定

| 決定 | 理由 | 却下案 |
|---|---|---|
| `MSG_PEEK` でクライアント切断検知 | OS レベルで確実 | `select()` — Windows 非対応の懸念 |
| `_max_hold_seconds = 3600` | LS の1セッション寿命に十分 | タイマーなし — スレッド枯渇リスク |
| `_shutdown_event` パターン | `server_close()` だけでは接続維持ループを脱出できない | `daemon=True` — 不正終了リスク |
