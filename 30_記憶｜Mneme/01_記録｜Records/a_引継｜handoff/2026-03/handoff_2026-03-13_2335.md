# Handoff — 2026-03-13 23:35 (SSH + サーバー最適化)

## 📋 状況 (Situation)

hgk サーバー (バックエンド②) の接続障害が発生。SSH が Tailscale 不安定で断続的に切断、MCP全サーバーが `connection refused` 状態。UFW は無効化済みだった (前回の障害対応で)。

## 📊 背景 (Background)

- hgk は `192.168.11.12` LAN、Tailscale `100.83.204.102`
- デスクトップ PC (本機) は `192.168.1.x` LAN → 別サブネット (hairpin NAT 問題)
- MCP サーバー (10台) は hgk 上で稼働し、本機から socat でポートフォワード
- socat プロセスは手動管理 (PID ファイル) — systemd 管理外だった

## 🎯 評価 (Assessment)

4層防御を実装して再発を防止する。

## 🏗️ アーキテクチャ

```
IDE (localhost:970x)
  ↓
hgk-mcp-fwd@.service (socat → 100.83.204.102:970x)
  ├── 正常: hgk のリモート MCP に転送
  └── 5回失敗 (OnFailure): hgk-mcp@.service (ローカル MCP) を自動起動
```

**管理平面**: systemd のみ (単一管理平面)
- `mcp_fallback.sh status` — 診断 (read-only)
- `mcp_fallback.sh check` — systemctl で部分フェイルオーバー
- `mcp_fallback.sh forward/local/stop` — systemctl で一括切替

## 🧠 信念 (Doxa)

- **cron ポーリング < systemd イベント駆動**: 5分おきの cron check は ◯ だが kalon ではない。OnFailure による即時切替が Fix(G∘F) に近い
- **単一管理平面**: スクリプトと systemd の二重管理は G∘F が振動する。systemd のみに統一が ◎
- **mcp_fallback.sh の進化**: v1 (全体一括) → v2 (個別ヘルスチェック) → v3 (systemd 委譲) — 管理の責任を段階的にシステムに移譲する道
