## 📊 評価 (Assessment)

### Θ(B) 自動計算の実現可能性

**結論**: [推定 75%] conv セッションファイルから自動的に Θ(B) を計算可能。

**根拠**:
1. **H(s)**: MCP サーバー分布のエントロピー — conv ファイル内の `hermeneus`, `ochema`, `mneme` 等のサーバー名出現頻度から計算可能
2. **H(a)**: ツール使用頻度のエントロピー — `hermeneus_run`, `ochema_ask` 等の具体的ツール名の出現頻度から計算可能
3. **R(s,a)**: サーバー × ツールの相互情報量 — サーバー名とツール名のペアから計算可能
4. **S(B)**: MCP 応答成功率 — セッションファイル内のエラー/成功パターンから推定可能だが精度はやや低い

**期待される n**: 472 セッション中、MCP ツール使用がある (≒ HGK+ らしい) セッションは [推定] 200-300 件。
フィルタリング後 n=50-100 は現実的。

### 課題とリスク

1. **操作的定義の一致**: v2.py の `hgk_internal_results()` はセッションログから手動計算。conv からの自動抽出が同じ操作的定義に従えるか要確認
2. **テキスト抽出の精度**: conv ファイルは Markdown テキスト。ツール呼び出しの正確な抽出にはパーサー開発が必要
3. **S(B) の近似**: 応答成功率はセッションテキストから直接計算困難。代替指標の検討必要
4. **MCPToolBench++ との比較可能性**: 操作的定義が異なりすぎると比較の意味が薄れる

### 代替アプローチ: tape データ

tape データ (511エントリ) は構造化済み JSONL だが、**WF 単位** (hermeneus_run, ochema 等) であり、
個別の MCP ツール呼び出し粒度ではない。WF 実行パターンから H(s)/H(a) を計算する場合、
操作的定義を v2 と異なるレベルで再定義する必要がある。

## 🎯 推薦 (Recommendation)

### 次セッションで実施すべきこと

1. **conv パーサー PoC** (最優先): 5セッション分で MCP ツール名を自動抽出し、H(s)/H(a)/R(s,a) を計算するスクリプトを作成
2. **操作的定義の明確化**: v2 の手動計測と conv 自動抽出の対応表を作成し、定義の一致を確認
3. **S(B) 代替指標**: セッション内の「エラー」「失敗」パターンから S(B) を近似する方法を検討
4. **n 目標の設定**: n≥30 を最低目標、n≥50 を理想目標として設計

### 参考: v6.1 の三層分離との統合

n が増えた場合:
- **n≥10**: Bootstrap CI が意味を持つ → Layer 3 (存在証明) から Layer 2 (統計的検証) に昇格可能
- **n≥30**: 正規分布近似が適用可能 → パラメトリック検定も使える
- **n≥50**: Layer 2 にフルに統合でき、MCPToolBench++ との直接比較が統計的に有意になる
