## 🧠 設計判断

| 判断 | 選択 | 理由 |
|---|---|---|
| Auto-Routing の 2段フォールバック | インメモリ → API | ochema 内は同一プロセスでインメモリ短絡が有効。プロセス再起動後は API で復旧 |
| opt-out メカニズム | `cached_content="none"` | 空文字は「未指定」、`"none"` は「明示的拒否」の意味分離 |
| ROM→CAG のトリガー条件 | テキスト > 500文字 | 短すぎるテキストのキャッシュは API コスト対効果が低い |
| `cortex_cache.py` の変更なし | 既存 API で十分 | `cache_health()` + `list_caches()` + `create_cache()` で要件を満たす |

## 💡 Value Pitch

**Before**: `ask_cortex` を呼ぶたびに Agent が `cached_content` を手動指定する必要があった。省略すると CAG の恩恵を受けられず、API コストとレイテンシが増大。ROM 蒸留後もキャッシュは古い boot 時のまま放置。

**After**: `ask_cortex()` を引数なしで呼ぶだけで、最新の Boot/ROM キャッシュが自動適用される。Agent の意志依存を排除し、環境強制で CAG を有効化。ROM 焼付け後はキャッシュが自動更新され、常に最新文脈で推論。

**比喩**: 以前は毎回カーナビに住所を手入力していた。今は車に乗ればカーナビが自動的に最新の目的地を表示する。
